## 2. 전면 vpos 앵커 구현 시도 (3개 위치, 모두 노트 미이동)

`page_vpos_base` 기준 `current_height = (para.first_vpos − base)` 동기화를 다음에 삽입·검증:

| 위치 | 결과 | 진단 |
|------|------|------|
| 메인 루프 표-플러시 fit 직전(line 271) | 노트 9쪽 유지 | `ANCHOR_REACH` 디버그 **미발화** → pi=124~128 이 경로 미도달 |

- 모든 시도 후 **소스 되돌림(커밋 없음)** — 엔진 클린 상태.

## 3. 차단 사유 (정직)

페이지네이션 엔진은 **항목별 다중 경로**(메인 루프 표-플러시 / `paginate_text_lines` / partial-split / picture·shape / multicolumn / forced-break)와 분기를 가진다. 노트(단일 항목, fit)의 break 결정 지점을 3회 시도로 특정하지 못했다(추가된 디버그가 해당 pi에 발화하지 않음). 즉 **전면 vpos 앵커는 모든 fit-판정 지점에 일관 적용해야 하는 다중 경로 리팩터**이며, 빌드-시행착오식 증분 편집(빌드 1.5분/회)으로 안전히 완료할 수 없다.

## 4. 권고 (방향 결정 필요)

전면 vpos 앵커(A)는 다음을 요하는 **전용 리팩터**:
1. 엔진 per-item 디스패치 전수 매핑(어느 경로가 어느 항목을 처리하는지) — 정적 분석.
2. 단일 공통 앵커 지점 도입 또는 모든 fit-판정 지점에 vpos 오프셋 일관 적용.
3. bypass(TAC 수식/그림/Shape, vpos-reset, multicolumn) 보존.
4. 광범위 골든 재판정 + 페이지수/LAYOUT_OVERFLOW 회귀 검증.

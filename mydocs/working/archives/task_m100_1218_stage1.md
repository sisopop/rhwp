## 확정된 근본 원인

`typeset.rs typeset_table_paragraph` 에서 **`pre_table_end_line=0`** — 즉 엔진이 **표 옆을 흐르는 줄이 0개**라고 판단.
- 그 결과 `is_wrap_around_table && pre_height>0` 분기를 못 타고(else) **표 높이 전부 누적** + POST 에서 **전체 5줄을 표 아래 텍스트로 누적**.
- 실제로는 파일 LINE_SEG 상 본문이 표 옆으로 흐름(어울림) → 표와 같은 y 영역 공유해야 함.
- 정상이라면 `max(옆텍스트, 표) + 아래텍스트 ≈ 90px` 인데 163.7px 로 계산.

## 미해결 / 주의

- **모순**: 163.7px 누적이면 다음 단락(pi=259 ①)이 아래로 밀려 **간격**이 생겨야 하나, 실제 렌더는 **겹침**(overlay 상 pi=259 가 pi=258 시작 +73.5px). → pi=259(①)가 `current_height` 가 아니라 **wrap-around 존 기준으로 재배치**되는 경로가 따로 있음(typeset.rs ~1214~1391 흡수 로직). 표 높이 누적과 ① 위치 계산이 **이중으로 얽힘**.
- LINE_SEG `sw` 패턴(ls0~3=26788, ls4=19220)도 "옆/아래" 판정과 정합 확인 필요.

## 판단

근본 원인 위치(`pre_table_end_line=0`)는 확정. 그러나 수정 대상이 **pagination 코어 루프**(메모리 task1046/1049/1184 가 명시한 고위험 영역)이고, ① 재배치 경로와 얽혀 **단순·안전한 최소 수정이 아직 불명확**.
→ Stage 2 진입 전, ① 위치 계산 경로까지 계측으로 확정한 뒤 설계 필요. 가설 기반 코어 수정은 광범위 회귀(#1204 가 5건 회귀 낸 전례) 위험.

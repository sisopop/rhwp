- 단계: Stage 1 — 증상 A(콘텐츠 다음 쪽 밀림) 6px+ 출처 특정

## 1. 현상 재확인 (페이지 8, "추진일정은" 노트)

- 한컴 2022 PDF: 노트 "※ 추진일정은…"(21.3px)이 **8쪽 하단**(추진일정 표·보고회 직후).
- rhwp: 노트가 **9쪽 맨 위**(pi=127)로 밀림.
- 페이지네이터: 8쪽 `used=925.8px`(본문 941.1px, 여유 15.3px) → 21.3px 노트 거부.

## 2. 렌더러 vs 페이지네이터 실측 (43.6px 과측정)

- 렌더러(`--debug-overlay`): 추진일정 표(pi=126) 바닥을 **y=988.0**에 그림 → 본문 바닥(1046.9)까지 **58.9px 여유** → 노트 들어감.
- 페이지네이터 추정 콘텐츠 바닥 = body_y(105.8) + used(925.8) = **1031.6**.
- **페이지네이터가 렌더러보다 43.6px 과측정** → 노트 거부.
- 렌더러 항목 top y는 HWP **vpos** 와 일치(VPOS_CORR). 즉 렌더러가 한컴 정답에 부합, 페이지네이터가 틀림.

## 3. 국소화 (`RHWP_TYPESET_DRIFT=1`)

페이지 8 모든 단락에서 `fmt_total`(페이지네이터) > `vpos_h`(HWP 소스):

| pi | lines | fmt_total | vpos_h | diff | 비고 |
|----|-------|-----------|--------|------|------|
| 120 | 3 | 102.9 | 83.3 | +19.6 | sb4 + lh_sum52 + **ls_sum46.9** |
| 121 | 3 | 102.9 | 83.3 | +19.6 | |
| 122 | 1 | 44.7 | 20.0 | +24.7 | sb6.7 + lh20 + **ls18** + trail_ls18 |
| 123 | 1 | 37.0 | 17.3 | +19.6 | |
| 125 | 1 | 44.7 | 20.0 | +24.7 | |

- `fmt_total = sb + lh_sum + ls_sum`. **`ls_sum`(line_spacing 합)·`trail_ls`가 vpos 대비 과다 계상**.
- 1줄 단락도 fmt_total 44.7 vs vpos_h 20.0 (+24.7) — line_spacing 1회분(18px)+trailing 이 통째로 더해짐.

## 4. 원인 결론

**페이지네이터 단락 높이 측정(`fmt_total`)이 line_spacing(`ls_sum`/`trail_ls`)을 실제 레이아웃(vpos)보다 과다 계상**한다. 렌더러는 VPOS_CORR로 vpos에 스냅하여 실제(=한컴) 높이로 그리므로, 페이지네이터만 부풀어 누적 잔차(8쪽 43.6px)가 노트를 다음 쪽으로 밀어낸다.

- #1022(trailing_ls over-correction 제거), #537/#479/#452 와 동일한 line-spacing 계상 계열의 **페이지네이터 측** 문제.
- 코드 위치: `src/renderer/typeset.rs` 의 `fmt_total` 산정(`TYPESET_DRIFT_PI` ~1494 주변), `src/renderer/height_measurer.rs` 단락 높이.

## 5. 리스크 (Stage 3 대비)

- `fmt_total` line_spacing 보정은 **전 문서 페이지네이션에 영향**(모든 단락 +20px대 영향) → 광범위 회귀 위험 큼. 단계별 LAYOUT_OVERFLOW·페이지 수·골든 검증 필수.
- VPOS_CORR(렌더러)와 페이지네이터를 **단일 측정 공간**으로 정합하는 방향이 본질.

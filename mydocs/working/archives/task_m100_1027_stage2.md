## 1. 시각 대조 (한컴 2022 PDF vs rhwp, 페이지 11)

- `pdftoppm`로 PDF 11쪽, `rsvg-convert`로 rhwp 11쪽을 동일 폭(760px) 렌더 후 대조.
- **PDF**: 제목 박스 아래 ❶ 항목까지 여백(breathing room) 있음.
- **rhwp**: ❶ 항목이 박스에 더 바짝 붙음 — 여백 부족.

## 2. 정량 측정 (`--debug-overlay`, 페이지 11 항목별 렌더 top y)

| 전이 | 간격 |
|------|------|
| pi=167(박스) → pi=168(❶) | **16.0px** |
| pi=168(❶) → pi=169(❷) | 33.6px |
| pi=172(박스) → pi=173(❶) | **16.0px** |
| pi=186(박스) → pi=187(❶) | **16.0px** |
| pi=190(박스) → pi=191(❶) | **16.0px** |
| 일반 항목 간 | 33.6px (일정) |

- **모든 박스에서 박스→첫 항목 = 16.0px = 일반 항목 간격(33.6px)의 절반.** 약 17px 부족.
- #990/#1001 가드와 무관(Stage 0 scratch 검증: 가드 제거해도 렌더 불변). 별도 원인.

## 3. 판정: 증상 B 실재 (경미)

rhwp가 **박스(TAC Shape) 단락 아래 여백을 일반 간격의 절반(16px)으로 좁게** 그린다. PDF/한컴은 더 넉넉. 사용자 보고와 일치하는 실재 결함이나 크기는 ~17px/박스로 경미.

## 4. A·B 공통 뿌리 가설

- 박스 단락(pi=167 = "   " + TAC Shape)의 **line_spacing**:
  - **A(페이지네이터)**: `fmt_total`에 line_spacing(`ls_sum`/`trail_ls`)을 과다 계상 → 단락당 +20px 과측정.
  - **B(렌더러)**: 박스 아래에 line_spacing을 충분히 진행하지 않음 → ❶ 16px로 붙음.
- 즉 line_spacing의 **렌더러↔페이지네이터 불일치**의 양면. Stage 3에서 **단일 측정 공간** 정합 시 A·B 동시 해소 가능성.
- 단, B의 정확한 목표 간격(한컴 기준)은 작업지시자(Windows 편집기) 시각 판정이 최종 권위 — Stage 3 수정 후 재판정.

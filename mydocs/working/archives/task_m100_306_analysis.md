## 초기 가설 (모두 기각)

### 가설 1: 표 `wrap=TopAndBottom + tac=true` 처리 오류
페이지 4에 `단 1` 없고 해당 쪽에 이런 표가 있어 의심.

**기각**: 페이지 4 생성 원인 추적 결과 표 처리 문제 아님. 직접 원인은 `ColumnBreak::Column` (pi=71).

### 가설 2: 문단 높이 과다 계산
**기각**: 개별 문단 높이는 적절함. 문제는 "HWP가 의도한 열 배치" 와 "우리 엔진이 계산한 열 배치" 가 근본적으로 다름.

### 가설 3: 빈 문단 처리 과다
**기각**: 빈 문단 수 많으나 총 공간 소비는 전체 차이(~800px)보다 훨씬 작음.

### 가설 4: `ColumnBreak::Column` (raw break 0x08) 조건부 처리
21_언어 문서에 19곳의 0x08 플래그 존재. 각각이 강제 열/쪽 전환 유발.

**부분 기각**: 조건부 처리(`문단이 남은 공간에 들어가면 브레이크 무시`) 실험 결과:

| 샘플 | PDF | 기존 | 조건부 실험 |
|------|-----|------|-----------|
| 21_언어 | 15 | 19 | **14** (−1) |
| exam_math | 20 | 20 ✓ | **10** (심각) |
| exam_kor | ? | 25 | 25 |
| exam_eng | ? | 11 | 10 |

exam_math에서 심각한 회귀 — Column break가 exam_math에선 유효한 정보. 단일 휴리스틱으로 해결 불가.

## 진짜 원인 (가설 E) — LINE_SEG `vpos` 데이터 활용 부족

`dump -s 0 -p 50` 결과 결정적 단서:

```
ls[0]: vpos=75116    ← 첫 열 line 0
ls[1]: vpos=76932
...
ls[8]: vpos=89644    ← 첫 열 line 8 (9줄까지)
ls[9]: vpos=0        ← **새 열/쪽 시작 (vpos 리셋)**
ls[10]: vpos=1816    ← 둘째 열 line 0
ls[11]: vpos=3632    ← 둘째 열 line 1
```

**HWP 파일 자체에 각 줄(`LINE_SEG`)의 세로 위치가 `vpos` 로 저장되어 있고, `vpos=0` 리셋이 새 열/쪽의 시작을 표시한다.** 한컴 오피스(PDF 생성)는 이 정보를 그대로 사용하여 배치하므로 원본과 정확히 일치.

### 우리 엔진이 다른 점

우리 `pagination/engine.rs` 는 파라그래프 단위 높이를 자체 계산하여 순차 적재 방식으로 배치. 이 때 LINE_SEG의 `vpos=0` 정보를 열 경계 신호로 사용하지 않음. 따라서 HWP가 "이 줄에서 열을 나눈다" 라고 지시한 지점을 존중하지 못함.

### 결과

21_언어 페이지 3의 예:
- **PDF 단 0**: pi=44~70 포함 (4번·5번 문제까지) — HWP LINE_SEG vpos가 그렇게 지시
- **SVG 단 0**: pi=44~50(9줄)만 포함 — 우리 엔진이 pi=50을 단 0에서 일찍 끊음

이 "일찍 끊김" 이 뒷 페이지로 누적 전파되어 총 4쪽 초과 발생.

### 왜 exam_math 는 문제 없는가

exam_math LINE_SEG vpos 분포가 우리 엔진의 자체 계산 결과와 우연히 거의 일치 (혹은 오차가 열 경계에서 상쇄). 21_언어는 오차가 누적되는 패턴.

## 시도한 실험들 (모두 롤백)

1. **Column break 완전 비활성화**: 21_언어 14쪽, exam_math 10쪽 (후자 심각 회귀)
2. **조건부 Column break (피트 판정)**: 동일 결과
3. 이유: 단순 휴리스틱은 두 샘플의 서로 다른 특성을 동시에 만족시킬 수 없음

모두 `src/renderer/pagination/engine.rs` 에서 원복 완료.

## 향후 작업 방향

### 권장 — Epic 수준의 구조적 재설계

페이지네이션 엔진을 **LINE_SEG vpos 우선 모드** 로 전환:

1. 파라그래프 순회 대신 LINE_SEG 순회로 변경 (또는 병행)
2. `vpos=0` 리셋을 열/쪽 경계 신호로 해석
3. 현재 열의 가용 공간 계산을 LINE_SEG vpos 연속성 기준으로
4. 파라그래프 높이는 fallback (LINE_SEG가 없거나 비신뢰 시)

이는 기존 페이지네이션의 상당 부분 재설계로 **별도 마일스톤 또는 Epic 이슈** 로 분리 권장. 회귀 위험 관리 위해 단계적 접근 필요.

### 대안 — 샘플 우선순위 낮춤

21_언어 같은 특정 샘플의 쪽수 일치는 PDF 렌더링 완성도의 한 지표일 뿐. 실제 사용상 내용은 모두 들어감(+4쪽 과잉 생성이라도 잘림은 없음). 다른 우선순위 높은 이슈 처리 후 여유가 될 때 Epic 수준으로 진행.

## 학습

- HWP 파일은 LINE_SEG vpos로 **사전 계산된 레이아웃 정보** 를 제공한다. 페이지네이션 엔진은 이를 respect 해야 원본과 일치한다.

- 단순 휴리스틱(Column break 비활성화/조건부 등)은 한 샘플 최적화가 다른 샘플 회귀로 이어짐.

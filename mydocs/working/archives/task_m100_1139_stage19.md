# Task M100 #1139 Stage 19

## 목적

Stage 18 커밋 이후 남은 12쪽 하단과 13쪽 시작부의 한컴 정답지 차이를 추가로 좁힌다.

## 시작 기준

- Stage 18에서 글자처럼 취급 그림 문단의 중복 예약 줄 보정을 적용했다.
- `3-09월_교육_통합_2022.hwp` 전체 페이지 수는 한컴 기준과 동일한 23쪽이다.
- 12쪽 그래프 이후 수식 흐름은 크게 개선되었지만, 하단에서 다음 페이지로 넘어가는 위치가 한컴과 아직 미세하게 다르다.

## 작업지시자 기준

- 한컴 정답지와 비교해 12쪽 왼쪽 하단 및 다음 쪽 시작 위치를 맞춘다.
- PNG 산출물을 만들면 SVG보다 PNG 경로를 우선 보고한다.

## 분석 계획

1. Stage 18 PNG와 한컴 정답지를 기준으로 12쪽 하단의 마지막 표시 줄과 13쪽 시작 줄을 비교한다.
2. `dump-pages`와 `target/debug/rhwp info ... -p`를 함께 사용해 해당 문단의 줄별 split 위치를 확인한다.
3. 보정이 글자처럼 취급 그림 문단에만 적용되어야 하는지, 미주 문단 split 일반 규칙에 반영되어야 하는지 분리한다.

## 검증 대기

- `cargo fmt`
- `cargo build`
- 12쪽 PNG 시각 비교
- 필요 시 `wasm-pack build --target web --out-dir pkg`

## 분석 결과

- 12쪽 하단의 `문15)` 흐름에서 새 미주 시작 문단이 원본 `LINE_SEG` vpos 기준으로 약 47px 아래로 보정되면서, 뒤따르는 수식 줄들이 한컴보다 이르게 다음 쪽으로 밀렸다.
- 페이지네이터 쪽에서는 새 미주 사이 간격을 실제 미주 모양의 `미주 사이` 값으로 cap했지만, 렌더러의 `HeightCursor`는 같은 cap을 모르고 절대 vpos 보정을 다시 적용해 12쪽 하단 overflow가 재발했다.
- `pi=669/671/673`은 텍스트가 비어 보이지만 실제로는 수식 컨트롤을 가진 문단이므로 빈 문단으로 제거하면 안 된다.

## 수정 내용

- `typeset.rs`에서 compact endnote flow의 새 미주 시작 간격을 미주 모양의 `미주 사이` 값 기준으로 cap한다.
- `height_cursor.rs`에서 compact endnote flow의 새 numbered endnote 시작 점프를 억제할 때, 억제한 만큼 vpos base도 함께 이동시켜 다음 줄들이 다시 절대 vpos gap을 복원하지 않게 했다.

## 1. 시도한 수정 (구현계획서 v2 §1)

`height_measurer.rs`에 3건 적용:

1. 경로 B `total_content_height` 보정: per-cell `max` → `line_sum + nested_h` 합산.
2. 경로 A `content_height`(중첩 표 분기): `last_seg_end.max(text_height)` → `(text_height + nested_total).max(last_seg_end)`.
3. `remaining_content_for_row`: 중첩 표 셀은 줄 단위 스냅 대신 비례 경로 사용.

## 3. 심화 진단 — 근본 원인 재확정

계측(`RHWP_DEBUG_*`)으로 렌더러 실제 그림과 대조한 결과, 구현계획서 v2 §1의 "per-cell `max`" 가설은 **불완전**했다.

### 핵심 사실

pi=308 행 1 셀(문단 51개 + 중첩 표 11개)에서:

- 중첩 표 호스트 문단의 **저장된 `LINE_SEG.line_height`가 한컴 원본 레이아웃 기준 중첩 표 높이를 이미 포함**한다(예: 2x3 표 호스트 p[3] `lh=6076` HU ≈ 81px). 렌더러 코드도 이 전제로 동작(`table_partial.rs:1092` "LINE_SEG.lh에 중첩 표 높이가 이미 포함"). `height_measurer.rs:712` 주석("미포함")은 부정확.
- 그러나 우리 렌더러는 중첩 표를 **재레이아웃**한다(`calc_nested_table_height` = `resolve_row_heights` 기반). 재레이아웃 결과가 한컴 저장 레이아웃보다 **계통적으로 ~9~10% 크다**(중첩 표·텍스트 모두). 예: 1x5 표 측정 17.1px vs 렌더러 `calc_nested_table_height` 22.8px, 1x1 표 145.8 vs 153.3.
- 결과적으로 렌더러의 실제 그림(`para_y` 누적 ≈2660px)은 **한컴이 저장한 `line_segs`(vpos·lh) 어느 조합으로도 예측 불가**하다. `line_segs`는 한컴 원본 레이아웃 좌표이고, 렌더러는 자체 폰트 메트릭으로 다시 잰다.

### 의미

페이지 143 오버플로의 근본 원인은 *중첩 표 측정의 단순 버그*가 아니라, **`height_measurer`(한컴 저장 레이아웃 신뢰)와 렌더러(자체 재레이아웃) 사이의 계통적 높이 불일치**다. 이는 문서 전반의 페이지 수 드리프트(SVG 184 vs PDF 179)와 **동일한 폰트 메트릭 기인 결함**으로, 수행계획서 §4가 "별도 사안"으로 제외한 항목과 뿌리가 같다.

저장 레이아웃을 신뢰하는 어떤 측정 보정으로도 렌더러의 재레이아웃 결과를 정확히 예측할 수 없다. 측정을 키우면(시도처럼) 다른 문서(aift.hwp)에서 과대 측정 회귀가 난다.

## 4. 해결 방향 (작업지시자 결정 요청)

페이지네이터가 렌더러의 재레이아웃 결과를 정확히 알 수 있는 길은 둘뿐이다.

- **안 A — 페이지네이터가 렌더러 높이 함수를 직접 사용.** 렌더러에는 `LayoutEngine::calc_cell_remaining_content_height`(셀의 content_offset 이후 잔여 높이를 per-paragraph로 정확히 계산, `calc_nested_table_height` 사용)가 *이미 구현*되어 있으나 현재 호출되지 않는 사문이다. 문서 주석: "pagination 엔진이 MeasuredCell 기준으로 content_offset을 산출하므로 동일 기준이어야 함". 분할 표 연속분 측정을 이 함수로 대체하면 페이지네이터 = 렌더러가 정의상 일치 → 오버플로 0. 다만 `LayoutEngine`(렌더러)을 페이지네이터에서 접근 가능하게 해야 하며, 측정 경로 교체로 페이지네이션이 전반적으로 바뀌어 골든 회귀 점검 범위가 넓다. → 구현계획서 v3 개정 필요.

- **안 B — 방어 가드.** 측정 정밀도는 손대지 않고, 렌더러가 본문 영역 하단을 넘기지 않도록 분할 표 연속분에 안전 여유(예: 측정값 ×1.15) 또는 렌더 후 경계 초과 시 강제 재분할 가드를 둔다. 작고 국소적이나 휴리스틱이며, 페이지 수를 보수적으로 더 늘린다.

- **안 C — 범위 축소.** 페이지 143(pi=308)은 *문단 51개 + 중첩 표 11개*의 병리적 분할 표로, 폰트 메트릭 드리프트가 집약된 케이스다. task992는 페이지 171(데코레이션 오분류, 명확·국소)만 해결하고 페이지 143은 폰트 메트릭/페이지 드리프트 사안으로 인계.

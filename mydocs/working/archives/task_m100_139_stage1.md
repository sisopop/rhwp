## 완료 내용

### 1. 명령어 대응표 (3장)

한컴 수식 명령어와 LaTeX 대응 명령어를 11개 분류로 매핑했다:
- 구조(8종), 큰 연산자(10종), 행렬/배열(8종), 괄호(4종), 그리스 문자(3종)
- 연산자/관계 기호(8종), 글자 장식(6종), 글꼴 스타일(6종), 공백/제어(5종)
- 한컴 전용 명령어(4종), LaTeX 전용 기능(10종)

**핵심 발견**: 한컴 명령어의 ~80%는 LaTeX에서 `\` 접두어만 추가하면 대응. 구조적 차이가 큰 것은 분수(중위 vs 전위), 행렬(`\begin{env}` 환경 블록) 등.

### 2. GUI 수식 입력 방식 비교 (2장)

한컴, MS Word, MathType, LyX, rhwp를 3가지 축으로 비교했다:
- **편집 위치**: 인라인(MS Word) vs 별도 대화상자(한컴, MathType, rhwp)
- **렌더링 방식**: WYSIWYG(MS Word, MathType) vs 스크립트+미리보기(한컴, rhwp)
- **구조 삽입**: 시각적 갤러리(MS Word) vs 아이콘+텍스트 삽입(한컴, rhwp) vs 탭 도구바(MathType)

**핵심 통찰**: MS Word가 가장 진보된 GUI 수식 편집 경험. rhwp 단기 개선은 듀얼 입력 모드(한컴/LaTeX) + 자동완성이 가장 효과적.

### 3. 문법 구조 비교 및 장단점 분석 (4~5장)

- 설계 철학 비교: 명령어 접두어, 중위/전위, 환경 블록, 대소문자 등
- 동일 수식 4개 예시 (근의 공식, 정적분, 행렬, 연립방정식)
- 한컴/LaTeX 각각의 장단점 및 전략적 관점 분석

### 4. 파서 아키텍처 분석 (7장)

- 현재 파이프라인: tokenizer → parser → AST(EqNode) → layout → render
- **핵심 발견**: AST 이하 레이어(layout, render)가 완전히 문법 독립적 → 토크나이저와 파서만 확장하면 LaTeX 지원 가능

### 5. 확장 전략 비교 및 개선 방향 (8~9장)

3가지 전략을 비교하여 **전략 A(듀얼 토크나이저)**를 권장 전략으로 선정했다:

| 기준 | A: 듀얼 (권장) | B: 전처리 | C: 통합 |
|------|--------------|----------|---------|
| 기존 코드 보호 | ★★★★★ | ★★★★ | ★★ |
| 표현 범위 | ★★★★★ | ★★★ | ★★★★★ |
| 구현 난이도 | 중간 | 낮음 | 높음 |

후속 구현 타스크 분리 계획과 예상 규모도 포함했다.

# Task #1253 Stage3

## 목적

Stage1/Stage2에서 rhwp-studio UI와 HWPX 미주 구분선 굵기 파싱은 보정했지만, `3-09월_교육_통합_2022.hwp` 9쪽 미주 영역 렌더가 한컴오피스 2024와 여전히 다르다. 이번 단계에서는 UI/파서가 아니라 실제 미주 조판 공통 로직을 다시 추적한다.

## 문제 현상

- 작업지시자가 2026-06-03에 `3-09월_교육_통합_2022.hwp` 9쪽 비교 스크린샷을 제공했다.
- rhwp-studio 9쪽 미주 시작/문항 간격/테이블 하단 이후 흐름이 한컴오피스 2024와 다르게 보인다.
- Stage2에서 “공통 로직”이라고 확인한 부분은 미주 모양 설정값 매핑과 HWPX 선 굵기 파싱 중심이었고, 실제 페이지 내 `LINE_SEG`/VPOS 소비와 미주 사이 간격 적용이 한컴과 동일하다는 뜻은 아니었다.

## 확인할 항목

1. `3-09월_교육_통합_2022.hwp` 9쪽의 미주 문단 시작 y, 구분선 y, 문1/문2/문3 제목 y를 덤프로 수집한다.
2. 기존 회귀 테스트가 페이지 포함 여부만 확인하고 내부 y 위치 차이를 놓치고 있는지 확인한다.
3. `endnote_between_notes_pagination_margin()`, `HeightCursor.endnote_between_notes_hu`, `endnote_line_vpos_base`가 같은 미주/다른 미주/페이지 첫 미주에서 각각 어떻게 적용되는지 추적한다.
4. 한컴 기준 기본 `미주 사이 7mm`, `구분선 아래 2mm`가 원본 `LINE_SEG`에 이미 반영된 구간과 rhwp가 별도로 추가해야 하는 구간을 공통 규칙으로 분리할 수 있는지 확인한다.

## 진행 기록

- 2026-06-03: Stage2/PR 준비 커밋 이후, 작업지시자가 한컴오피스와 rhwp-studio 9쪽 렌더 차이를 보고했다. 새 스테이지 문서를 만들고 원인 분석을 시작했다.
- 2026-06-03: `cargo run --bin rhwp -- dump-pages samples/3-09월_교육_통합_2022.hwp -p 8`로 9쪽 덤프를 확인했다. 9쪽은 페이지 수/문단 포함 여부는 맞지만, 2단 쪽에서 `hwp_used`가 실제 `used`보다 크게 튀고 `pi=522`가 `vpos=124748..94604`처럼 되감긴 범위를 가진다.
- 2026-06-03: `export-svg`로 9쪽 좌표를 확인했다. rhwp 기준 정답표 하단은 `y≈325.17px`, 녹색 미주 구분선은 `y≈351.09px`, `문1)` 시작은 `y≈370.97px`이다. 기존 회귀 테스트는 `문1)`이 정답표 아래인지와 구분선 존재 여부만 확인하므로, 한컴오피스와의 내부 y 위치 차이를 잡지 못한다.
- 2026-06-03: 현재 미주 조판은 `endnote_between_notes_pagination_margin()`, `new_endnote_between_notes_px`, `default_between_notes_gap`, `allow_default_late_question_tail` 등 여러 조건 분기가 섞여 있다. 설정값 매핑은 공통화됐지만 실제 VPOS 소비는 아직 케이스별 보정에 의존한다.

## 1차 판단

Stage2에서 처리한 공통 로직은 UI/API/HWPX 파서의 설정값 흐름을 맞춘 것이다. 지금 보고된 9쪽 차이는 설정값이 아니라 페이지 안에서 미주 `LINE_SEG.vertical_pos`, 마지막 줄 `line_spacing`, 새 미주 시작 간격을 어떻게 소비하는지의 문제다.

따라서 Stage3의 수정 방향은 다음과 같다.

1. `3-09월_교육_통합_2022.hwp` 9쪽 정답표 하단, 구분선, `문1)`, `문2)`, `문3)`의 y 좌표를 회귀 테스트로 명시한다.
2. 한컴 기준과 비교해 구분선/첫 미주가 위로 붙는지 또는 아래로 밀리는지 확정한다.
3. `미주 사이 7mm` 기본값을 “원본 VPOS에 이미 반영된 gap”과 “렌더러가 추가해야 하는 최소 gap”으로 나눠 공통 규칙을 다시 정리한다.

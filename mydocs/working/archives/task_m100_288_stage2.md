# Task #288 단계 2 보고서 — PDF 실측 결과 "증상 없음" 확정

## 타스크

[#288](https://github.com/edwardkim/rhwp/issues/288) — 수식 SVG 레이아웃: 빈 runs 줄의 TAC 수식 x 정렬 (#287 Phase 2)

- 단계 2 목적: 한컴 PDF 의 수식 x 를 정밀 측정해 #287 수정 결과와 대조, 특수 정렬 규칙 역추정

## 측정 방법

PDF content stream 을 Node.js 로 직접 파싱해 text matrix (Tm) 을 추출.

```js
const re = /stream\r?\n([\s\S]*?)\r?\nendstream/g;
// zlib.inflateSync → BT/Tm/Tj 명령 추출
```

### PDF CTM 및 Tm

`samples/exam_math_8.pdf` 본문 stream 내 CTM:

```
0.119935 0 0 -0.119869 0 841 cm
```

즉 PDF 페이지 좌표계 = 원본 좌표 × 0.119935.

본문 요소별 Tm.e (x 원본 좌표):

| 요소 | PDF Tm.e | SVG x (96dpi px) | 비율 (SVG/PDF) |
|------|---------|------------------|---------------|
| "(" | 481 | 102.07 | 0.2122 |
| "모든" 이후 텍스트 런 | 789 | ≈ 135.42 ("모") | 0.1716 (텍스트 내부 offset, 보정 필요) |
| 작은 수식 `n` | 1016 | 217.07 | 0.2136 |
| "에" | 1062 | 226.89 | 0.2137 |
| "대" | 1160 | 247.21 | 0.2131 |
| **큰 수식 첫 glyph** | **627** (y=965) | **133.27** | **0.2125** |

**x 비율이 모든 요소에서 약 0.2125 로 일치**. 즉 SVG ↔ PDF 좌표 변환 비율은 고정되어 있고, 큰 수식 x=133.27 은 PDF 의 627 에 정확히 대응한다.

### 상대 오프셋 검증

- PDF: 수식이 "(" 보다 `627 - 481 = 146` 단위 오른쪽
- SVG: 수식이 "(" 보다 `133.27 - 102.07 = 31.2` px 오른쪽
- 환산: `146 × 0.2125 = 31.03` px
- SVG 실측 31.20 px ↔ 환산값 31.03 px **≈ 0.17 px 오차 (반올림 수준)**

### y 축 (참고)

- PDF: 수식 y=965, "(" y=755 → 차이 210 단위 → `210 × 0.119869 × 96/72 ≈ 33.6` px
- SVG: 188.29 - 160.43 = 27.86 px
- 오차 약 5.7 px — baseline 계산 방식 차이 수준 (수식의 top-left 기준 vs 글꼴 baseline 기준)

## 결론

**#287 의 수정만으로 이미 한컴 PDF 와 x 축이 사실상 일치한다. #288 은 실제 증상이 없는 이슈.**

이슈 #288 본문의 전제였던:
- "PDF x≈162-182" → **측정 오류** (PDF content stream 실측 결과 해당 x는 존재하지 않음)
- "HWP composed text 에 `\t` 삽입" → **단계 1 에서 이미 부인됨**

둘 다 근거 없음. 원래의 "눈대중" 비교에서 나온 허상이었다.

## 교훈

1. **눈대중 대신 콘텐츠 스트림 실측**: PDF 의 Tm 을 직접 추출해 좌표를 수치로 대조하는 방식이 훨씬 정확. 시각 이미지 비교는 스케일·여백 때문에 10-50 px 수준의 오차가 쉽게 생긴다.
2. **가설 유지 비용**: 단계 1 에서 "#288 의 탭 가설이 틀렸다" 를 확인했음에도 "그럼 다른 규칙이 있을 것" 이라는 가정을 유지했는데, 실제로는 **원래 규칙도 맞고 수정 결과도 맞았다**. 부정 확인 후 "증상 자체가 있는지" 를 한 번 더 의심해야 함.
3. **범위 확대의 유혹**: 단계 1 에서 "규칙 역추정 후 추가 수정" 으로 진행했으면 존재하지 않는 문제를 위한 코드를 작성할 뻔했다.

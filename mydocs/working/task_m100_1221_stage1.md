# Stage 1 조사보고서 — Task M100 #1221 (z-표 행 압축)

**브랜치**: `local/task1221` (← devel)

## 근본 원인 (확정)

z-표 값들은 **인라인 수식(equation, tac=true)**:
- z-열 셀[2]: 4개 수식 / 3문단. p[0]=2줄(1.0 @ `lh=825`, 1.1 @ `lh=900`), p[1]=1.2, p[2]=1.3.
- P-열 셀[3]: 4개 수식 / 4문단, 각 `lh=900`.

`composer.rs:659` 의 `corrected_lh` 는 **TAC 표가 lh에 포함된 경우만 아래로 보정**(Task #19). 인라인 수식이 `line_height` 보다 **클 때 위로 보정하지 않음**. 따라서:
- z-열 p[0] line0 의 수식(1.0)이 `lh=825`(<수식 높이 900)로 렌더 → line1(1.1)과 겹침("1.01").
- z-열 줄높이(825/900/900/900) ≠ P-열(900×4) → 행이 누적 어긋남 + valign=Center 중앙정렬 오프셋 차이.

## 수정 방향 (광범위 영향)

줄 높이 = `max(line_seg.line_height, 인라인 객체(수식/도형) 높이)` 로 클램프.
- **영향 범위**: composer/line-height 는 **표 셀뿐 아니라 인라인 수식 포함 모든 문단**에 적용 → 전 문서 영향. 본 세션의 contained 수정들과 위험도 차원이 다름.
- 안전 적용에는 인라인 수식 높이 산출 + svg_snapshot/표·수식 다수 샘플 시각 회귀의 광범위 검증 필요.

## 판단

근본 원인은 확정. 그러나 수정 대상이 **인라인 객체 줄높이(전 문서 영향)** 라, 마라톤 세션 말미에 무리하게 변경하기보다 **전용 사이클에서 신중히 구현·검증**하는 것이 안전. (#1184 류 고위험 측정/줄높이 영역.)

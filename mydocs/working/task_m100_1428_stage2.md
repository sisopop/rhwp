# Task 1428 Stage 2

## 목표

- 누름틀 고치기 대화상자에서 `고치기` 완료 후 키보드 입력이 다시 동작하도록 한다.
- 대화상자 완료 후 hidden textarea 포커스와 캐럿 표시가 현재 커서 위치로 복귀하도록 한다.
- 마우스로 빈 누름틀 안내문을 클릭했을 때 누름틀 내부 편집 상태로 들어가지 못하는 문제를 함께 보정한다.

## 문제 인식

- 누름틀 고치기 완료 뒤 모달은 닫히지만 문서 입력 대상이 다시 포커스를 얻지 못해 키보드 입력이 먹지 않는다.
- 완료 후 커서 위치 기준으로 캐럿이 다시 보이지 않아 사용자가 현재 입력 위치를 확인하기 어렵다.
- 키보드로 누름틀 경계 바깥 상태가 남은 뒤 마우스로 안내문을 클릭하면, 동일 charOffset의 경계 이탈 상태가 우선되어 누름틀 활성 필드로 전환되지 않을 수 있다.
- 앞 누름틀 값과 뒤 빈 누름틀이 붙어 있는 경우, 뒤 안내문 클릭 hit-test가 빈 필드 start가 아니라 안내문 표시 폭 안쪽 charOffset으로 떨어져 두 번째 누름틀 진입이 실패할 수 있다.

## 구현 방침

- 누름틀 속성 갱신 성공 후 문서 입력 핸들러의 focus/caret 복귀 경로를 호출한다.
- 커서 위치 자체를 임의 이동하지 않고, 현재 cursor position 기준 캐럿을 다시 표시한다.
- 기존 누름틀 속성 저장 및 필드 선택 상태는 유지한다.
- 마우스 일반 클릭으로 ClickHere 필드 위치에 진입하면 경계 이탈 키를 해제하고, 빈 안내문이면 field start로 정규화한 뒤 활성 필드로 설정한다.
- 직접 필드로 판정되지 않는 안내문 표시 폭 클릭은 같은 문단의 가장 가까운 앞쪽 빈 ClickHere 필드 start로 보정한다.

## 검증 계획

- 누름틀 고치기 후 키보드 입력이 현재 커서 위치에서 다시 동작하는지 확인한다.
- `123입력하세요`처럼 일반 텍스트 뒤 빈 누름틀이 있을 때 안내문을 마우스로 클릭해 누름틀 내부에 진입할 수 있는지 확인한다.
- `1234입력하세요`처럼 앞 누름틀 값과 뒤 빈 누름틀 안내문이 붙어 있는 상황에서 두 번째 누름틀 안내문을 마우스로 클릭해 진입할 수 있는지 확인한다.
- 관련 TypeScript 단위 검증 또는 타입 검사를 수행한다.

## 구현 결과

- `FieldEditDialog`에 `onClose` 콜백을 추가했다.
- `field:edit` 명령에서 누름틀 고치기 대화상자가 닫힌 뒤 `updateCaret()`와 `focus()`를 호출해 현재 커서 위치 기준 캐럿/입력 포커스를 복구하도록 했다.
- 누름틀 속성 변경 시 `document-mutated`와 `document-changed`에 `field-edit` 사유를 함께 전달하도록 정리했다.
- 마우스 일반 클릭 경로에서 ClickHere 필드 진입 시 `fieldStartExitKey`/`fieldEndExitKey`를 해제하고 빈 guide 클릭은 field start로 정규화하도록 했다.
- hit-test가 빈 guide start를 직접 반환하지 않는 인접 누름틀 케이스는 `getFieldList()`의 빈 ClickHere 범위와 guide 길이로 가장 가까운 guide start를 찾아 보정하도록 했다.

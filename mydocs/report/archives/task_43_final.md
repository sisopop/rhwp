#### 2. 호환성 Gap 분석

| 등급 | 수 | 비율 | 의미 |
|------|-----|------|------|
| A (직접 매핑) | 5 | 1% | 래퍼만 작성 |
| B (변환 매핑) | ~115 | 26% | 파라미터 변환 래퍼 |
| C (신규 구현) | ~290 | 65% | WASM 코어 구현 필요 |
| D (아키텍처 차이) | 2 | 0.4% | 구조적 재설계 |
| X (스텁 처리) | ~37 | 8% | 빈 함수 |

**아키텍처 차이**: 한컴은 커서 기반 API, rhwp는 좌표 기반 API. 호환 레이어에서 JS 커서 상태 관리 시스템 필수.

#### 3. 마이그레이션 시나리오별 커버리지

| 시나리오 | 필요 API | 현재 커버리지 | 핵심 블로커 |
|----------|---------|-------------|-----------|
| 문서 뷰어 | 5개 | **100%** | 없음 |
| 기안문 자동 생성 | 12개 | **25%** | 필드 시스템 |
| 양식 편집 | 25개 | **32%** | 커서/필드/Action |
| 완전 편집기 | 449개 | **27%** | 전체 |

#### 4. 전략적 인사이트

- 가장 빈번한 시나리오(기안문 자동 생성)의 블로커가 **필드 시스템 단일 영역**
- Phase 1에서 필드 시스템만 구현하면 가장 많은 공공기관 코드의 마이그레이션이 가능
- rhwp 고유 강점: 서버 비의존, 오픈소스, AI 통합 최적화, 다중 렌더링

#### 5. 구현 로드맵

| Phase | 목표 | 전체 커버리지 |
|-------|------|-------------|
| Phase 1 | 기안문 자동 생성 100% | ~30% |
| Phase 2 | 기본 편집기 60% | ~50% |
| Phase 3 | 고급 편집기 90% | ~75% |
| Phase 4 | 완전 호환 95% | ~95% |

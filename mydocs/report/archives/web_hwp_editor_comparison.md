# 웹 HWP 편집기 아키텍처 비교 보고서

## 1. 개요

본 보고서는 공공기관에서 사용되는 상용 웹한글 기안기와 본 프로젝트(rhwp)의 아키텍처를 비교 분석한다. 두 제품은 동일한 목표(웹 브라우저에서 HWP 문서 편집)를 추구하지만, 근본적으로 다른 기술 접근 방식을 채택하고 있다.

## 2. 웹한글 기안기 (상용 제품)

### 2.1 제품 개요

웹한글 기안기는 한글과컴퓨터에서 개발한 웹 기반 HWP 편집 솔루션으로, 공공기관 전자결재 시스템에서 HWP 문서를 작성·편집하기 위해 사용된다. ActiveX 기반의 기존 데스크톱 연동 방식을 대체하여, 브라우저에서 직접 문서를 편집할 수 있도록 설계되었다.

### 2.2 아키텍처: 2-Tier 서버 구조

```
┌─────────────┐     HTTP      ┌─────────────────┐    내부통신    ┌──────────────────┐
│  웹 브라우저  │ ←──────────→ │    웹 서버        │ ←──────────→ │   필터 서버        │
│  (클라이언트) │              │  (Tomcat 7+)     │              │  (RHEL 7+)        │
│              │              │  Java 기반        │              │  한컴오피스 2018+  │
└─────────────┘              └─────────────────┘              └──────────────────┘
    ↑ HTML/JS                     ↑ API 서버                     ↑ HWP 변환 엔진
    표시/입력                     요청 중계                      파싱/렌더링/변환
```

### 2.3 핵심 구성 요소

| 구성 요소 | 역할 | 요구 환경 |
|-----------|------|-----------|
| 웹 서버 | 클라이언트 요청 수신, 필터 서버 중계, 세션 관리 | Tomcat 7+, Java 8+, CentOS/RHEL |
| 필터 서버 | HWP 파싱, 렌더링, 포맷 변환 (핵심 엔진) | RHEL 7+, 한컴오피스 2018+ 설치 필수 |
| 클라이언트 | HTML/JS 기반 편집 UI, 서버와 API 통신 | 모던 브라우저 |

### 2.4 동작 원리

1. 사용자가 브라우저에서 문서 편집 요청
2. 웹 서버가 필터 서버에 HWP 파일 전달
3. 필터 서버가 한컴오피스 데스크톱 엔진으로 HWP 파싱/렌더링
4. 변환된 결과(HTML 또는 이미지 형태)를 웹 서버 경유 클라이언트에 전달
5. 클라이언트에서 편집 → 서버에 변경사항 전송 → 필터 서버에서 HWP 재생성

### 2.5 특징 및 제약

**장점**:
- 한컴오피스 기본 편집 엔진 공유 → 높은 호환성
- 144개 단축키 기능 지원
- 기존 전자결재 시스템과 연동 API 제공

**제약**:
- 필터 서버에 한컴오피스 데스크톱 제품 설치 필수 (라이선스 비용)
- 서버 의존적 → 네트워크 단절 시 편집 불가
- 서버 부하 집중 (동시 사용자 증가 시 필터 서버 스케일링 필요)
- 서버 방식 전환에 따른 일부 API 기능 제한 발생
- Linux 서버 환경 한정 (RHEL 7+ 필수)

## 3. rhwp (본 프로젝트)

### 3.1 아키텍처: 순수 클라이언트 사이드

```
┌──────────────────────────────────────────┐
│               웹 브라우저                  │
│                                          │
│  ┌──────────┐    ┌──────────────────┐    │
│  │ Canvas   │ ←─ │  WASM 모듈       │    │
│  │ 렌더링    │    │  (Rust 컴파일)    │    │
│  └──────────┘    │                  │    │
│       ↑          │  - HWP 파서      │    │
│  ┌──────────┐    │  - 레이아웃 엔진  │    │
│  │ JS 편집기 │ ←─ │  - 렌더 트리     │    │
│  │ (editor)  │    │  - 직렬화       │    │
│  └──────────┘    └──────────────────┘    │
│                                          │
│  서버 불필요 — 모든 처리가 브라우저 내부     │
└──────────────────────────────────────────┘
```

### 3.2 핵심 구성 요소

| 구성 요소 | 역할 | 기술 |
|-----------|------|------|
| HWP 파서 | HWP 5.0 CFB 파일 → Document IR 변환 | Rust |
| 레이아웃 엔진 | Document IR → PageRenderTree 생성 | Rust |
| 렌더러 | RenderTree → Canvas 2D 드로잉 | JS + Canvas API |
| 편집기 | 텍스트 입력/삭제, 캐럿, 선택, 상태 머신 | JS |
| 직렬화 | Document IR → HWP 바이너리 변환 | Rust |
| WASM API | Rust ↔ JS 브릿지 | wasm-bindgen |

### 3.3 동작 원리

1. 사용자가 HWP 파일을 브라우저에 로드 (File API)
2. WASM 모듈이 클라이언트에서 직접 HWP 파싱 → Document IR 생성
3. 레이아웃 엔진이 페이지별 RenderTree 생성
4. Canvas API로 렌더링 (텍스트, 표, 이미지, 도형)
5. JS 편집기가 사용자 입력 처리 → WASM을 통해 Document IR 수정
6. 저장 시 WASM 모듈이 Document IR → HWP 바이너리 직렬화 → 다운로드

### 3.4 특징

**장점**:
- 서버 불필요 → 인프라 비용 없음
- 오프라인 동작 가능
- 네트워크 지연 없는 즉각 응답
- 사용자 수 증가에 따른 서버 스케일링 불필요
- 플랫폼 독립적 (브라우저만 있으면 동작)
- 한컴오피스 라이선스 불필요

**제약**:
- HWP 포맷을 자체 구현해야 하므로 호환성 확보에 개발 노력 필요
- 클라이언트 디바이스 성능에 의존
- 복잡한 문서의 경우 초기 로딩 시간 발생 가능

## 4. 아키텍처 비교

| 비교 항목 | 웹한글 기안기 | rhwp |
|-----------|-------------|------|
| **HWP 처리 위치** | 서버 (필터 서버) | 클라이언트 (WASM) |
| **HWP 엔진** | 한컴오피스 데스크톱 엔진 재사용 | Rust 자체 구현 |
| **렌더링 방식** | 서버 변환 → HTML/이미지 전달 | 클라이언트 Canvas 직접 렌더링 |
| **서버 요구** | 웹 서버 + 필터 서버 (2대 이상) | 정적 파일 서빙만 (또는 불필요) |
| **외부 의존성** | 한컴오피스 2018+ 설치 필수 | 없음 (순수 Rust + WASM) |
| **네트워크 의존** | 필수 (서버 통신) | 불필요 (오프라인 가능) |
| **동시 사용자 확장** | 필터 서버 스케일링 필요 | 클라이언트에서 분산 처리 |
| **호환성** | 한컴 엔진 기반 → 높은 호환성 | 자체 구현 → 점진적 호환성 확보 |
| **라이선스** | 한컴오피스 서버 라이선스 필요 | 독립적 |
| **OS 요구** | 서버: RHEL 7+ 한정 | 플랫폼 무관 |

## 5. 기술적 시사점

### 5.1 패러다임 차이

웹한글 기안기는 **서버 중심 패러다임**으로, 기존 데스크톱 엔진의 신뢰성을 서버에서 활용하는 전략이다. 이는 호환성을 빠르게 확보할 수 있으나, 서버 인프라 운영 비용과 네트워크 의존성이라는 구조적 제약을 수반한다.

rhwp는 **클라이언트 중심 패러다임**으로, WebAssembly 기술을 활용하여 네이티브급 성능을 브라우저에서 직접 구현하는 전략이다. HWP 포맷의 자체 구현이라는 높은 기술적 장벽이 있으나, 서버 의존성 제거와 오프라인 동작이라는 구조적 이점을 확보한다.

### 5.2 WASM 접근의 가능성

WebAssembly를 통한 클라이언트 사이드 문서 처리는 다음과 같은 이점을 제공한다:

- **제로 서버 비용**: 정적 호스팅만으로 서비스 가능
- **프라이버시**: 문서가 서버를 거치지 않음
- **확장성**: 사용자 증가가 서버 부하로 이어지지 않음
- **응답성**: 네트워크 왕복 없이 즉각 처리

이는 특히 보안이 중요한 공공기관 환경에서 문서가 외부 서버를 경유하지 않는다는 점에서 의미가 있다.

## 6. 결론

두 접근 방식은 각각의 장단점이 명확하다. 웹한글 기안기는 한컴오피스 엔진을 재사용하여 높은 호환성을 단기간에 확보했으나 서버 의존적이다. rhwp는 순수 클라이언트 사이드 처리로 서버 의존성을 완전히 제거했으나, HWP 포맷의 자체 구현에 지속적인 개발 노력이 필요하다.

WebAssembly 기술의 성숙과 함께 클라이언트 사이드 문서 처리 방식은 서버 비용 절감, 오프라인 지원, 보안 강화 측면에서 경쟁력을 가질 수 있다.

# 수행 계획서 — Task M100-1414: Document IR 구조 감사 및 정리

- 이슈: https://github.com/edwardkim/rhwp/issues/1414

- 작성일: 2026-06-15
- 브랜치: `local/task_m100_1414`

## 1. 문제 요약

rhwp는 HWP5/HWPX/HWP3 parser frontend가 공통 `Document` IR로 lowering한 뒤,
document core / renderer / serializer backend가 이를 소비하는 컴파일러형 문서 처리
아키텍처를 지향한다.

그동안 렌더링, 직렬화, HWPX-HWP 변환, 진단 도구, 편집 기능이 누적되면서
`src/model` 중심의 IR 필드에 다음 성격의 코드와 데이터가 함께 축적되었다.

- semantic document model 필드
- HWP5/HWPX/HWP3 parser frontend 보정값
- renderer/layout 편의 필드
- serializer raw roundtrip 보존 필드
- document_core 편집/변환용 중간 상태
- 과거 task에서 도입된 임시 휴리스틱 또는 호환성 가드

현재 시점에서 Document IR contract를 재검토하고, 불필요하거나 중복된 필드와
책임 경계가 흐려진 지점을 식별할 필요가 있다.

## 2. 목표

1. `src/model/*` 중심의 Document IR 필드 인벤토리를 작성한다.
2. 각 필드의 생산자(parser/frontend)와 소비자(document_core/renderer/serializer)를
   추적한다.
3. 포맷 전용 보정값이 공통 IR에 누설된 후보를 식별한다.
4. renderer/serializer 임시 필드가 IR에 고착된 후보를 식별한다.
5. raw preservation 필드와 semantic IR 필드의 경계를 정리한다.
6. 중복 필드, 죽은 필드, 동일 의미 병렬 표현을 제거/통합/보존/문서화 후보로
   분류한다.
7. 실제 코드 변경이 필요한 항목은 별도 후속 이슈로 분리할 수 있게 판단 근거를
   남긴다.

## 4. 비목표

- 이번 1차 작업에서 대규모 리팩터링을 바로 수행하지 않는다.
- serializer raw preservation 필드를 근거 없이 제거하지 않는다.
- HWP3 전용 보정값을 공통 renderer에서 처리하는 방식으로 되돌리지 않는다.
- 시각 회귀를 유발할 수 있는 layout/renderer 동작 변경은 이번 감사 범위에 포함하지
  않는다.
- Markdown/PDF/SVG 등 개별 backend의 품질 개선은 후속 이슈로 분리한다.

## 5. 진행 단계

### Stage 1 — IR 필드 인벤토리

`src/model`의 주요 구조체와 필드를 표로 정리한다.

- 필드명
- 타입
- 선언 위치
- 주석상 의도
- raw preservation 여부
- format-specific 여부
- 기본값과 optionality

산출물:

- `mydocs/tech/document_ir_audit_1414.md`

### Stage 2 — 생산자/소비자 매트릭스

각 필드가 어디에서 생성/수정되고 어디에서 소비되는지 `rg` 기반으로 추적한다.

- parser/frontend 생산자
- document_core transform/query 소비자
- renderer/layout 소비자
- serializer/backend 소비자
- diagnostics/test 소비자

산출물:

- `document_ir_audit_1414.md` 내 matrix 섹션

### Stage 3 — 문제 후보 분류

필드별로 다음 분류를 부여한다.

- `keep`: 현재 IR contract로 보존
- `document`: 의미는 유지하되 문서화 필요
- `split`: semantic IR과 backend/raw 필드 분리 필요
- `merge`: 동일 의미 병렬 필드 통합 후보
- `remove_candidate`: 소비처가 없거나 과거 임시 필드로 보이는 후보
- `followup_issue`: 구현 변경은 별도 이슈로 분리

### Stage 4 — 최종 감사 보고서

감사 결과를 최종 보고서로 정리하고, 위험도별 후속 이슈 후보를 제안한다.

산출물:

- `mydocs/report/task_m100_1414_report.md`

## 6. 검증 방법

이번 작업은 코드 변경 없는 감사 작업을 기본으로 하므로 자동 테스트보다 추적 근거의
재현성을 검증한다.

- `rg` 검색 명령과 주요 결과 위치 기록
- 필드별 생산자/소비자 근거 파일 링크 기록
- 제거/통합 후보에 대해 최소 2개 이상의 근거 축 확인
  - 선언 주석
  - 생산자 존재 여부
  - 소비자 존재 여부
  - serializer raw roundtrip 관련성
  - regression/task 문서 연결

코드 변경이 발생하는 후속 이슈에서는 별도 구현 계획서와 회귀 테스트를 작성한다.

## 7. 리스크 및 대응

| 리스크 | 대응 |
|---|---|
| raw preservation 필드를 불필요한 필드로 오판 | serializer 소비처와 roundtrip 문서 확인 전 제거 후보 확정 금지 |
| renderer 편의 필드를 IR 누설로 오판 | 실제 한컴 조판 contract인지 renderer workaround인지 근거 분리 |
| HWP3/HWP5/HWPX 포맷 차이를 과도하게 일반화 | parser/frontend 책임과 공통 IR contract를 분리해 기록 |
| 감사 범위가 과도하게 커짐 | 1차는 `src/model` 중심, 실제 코드 정리는 후속 이슈로 분리 |

## 8. 승인 요청

위 범위로 Document IR 구조 감사를 진행한다. 1차 작업은 코드 변경 없이
`mydocs/tech/document_ir_audit_1414.md`와 최종 보고서를 작성하는 방식으로 진행한다.

# Task M100-1320 수행계획서 — 편집 액션 라우터와 Undo/Redo 트랜잭션 아키텍처 정비

## 배경

#1319에서 문단/글자 서식 Undo/Redo를 보강하면서 현재 rhwp-studio의 편집 작업 라우팅이 완전히 통합되어
있지 않음을 확인했다.

현재 구조는 다음 골격을 이미 갖고 있다.

- `CommandHistory`
- `EditCommand`
- `OperationDescriptor`
- `InputHandler.executeOperation()`
- `SnapshotCommand`

하지만 일부 UI 경로는 여전히 `wasm.*` mutation API를 직접 호출하거나, `recordWithoutExecute()`에 의존하거나,
snapshot으로 우회한다. 이 상태에서는 Undo/Redo 사용자 경험이 기능별로 달라질 수 있고, 복원 payload에 UI
단위와 core raw 단위가 섞일 위험이 있다.

## 목표

rhwp-studio의 편집 액션 라우팅과 Undo/Redo 복원 계약을 체계화한다.

사용자 관점 목표:

- 텍스트, 글자 서식, 문단 서식, 표, 이미지/도형 조작의 Undo/Redo 기대치가 서로 크게 어긋나지 않는다.
- 직접 조작, 메뉴/툴바 조작, 단축키 조작이 같은 history 정책을 따른다.
- 실패하거나 아직 미지원인 문맥은 조용히 문서를 변경하지 않고 명확히 no-op 또는 fallback 처리한다.

기술 목표:

- history stack은 하나로 유지한다.
- UI handler는 가능한 한 `executeOperation()` 또는 후속 router를 통해 mutation을 수행한다.
- Undo/Redo payload는 UI 조회 JSON이 아니라 도메인별 before/after ID, delta, snapshot 중 적절한 계약으로
  저장한다.
- dirty scope와 renderer invalidation 정책을 command 결과와 연결할 수 있는 기반을 만든다.

## 비목표

이번 이슈에서 모든 편집 기능을 완전히 transaction/delta 모델로 이전하지 않는다.

특히 다음은 1차 범위에서 제외하거나 조사/문서화 대상으로 둔다.

- 모든 WASM API가 before/after delta를 반환하도록 core 전체 개편
- 모든 snapshot command 제거
- 표 구조 변경 전체의 정밀 delta 구현
- 이미지/도형 복합 조작의 완전한 도메인 transaction화
- Chrome/Edge 확장 앱 저장/보안 정책 변경

## 현재 관찰

초기 조사 기준:

- `CommandHistory.redo()`는 redo 대상 command의 `execute()`를 다시 호출한다.
- `recordWithoutExecute()`는 IME, 객체 이동 등에서 사용된다.
- `SnapshotCommand`는 붙여넣기, 객체 삭제, 표 삭제 등 복합 조작에 사용된다.
- `InputHandler.executeOperation()`은 command/snapshot/record를 받는 공통 입구다.
- 문단/글자 서식은 #1319에서 before/after shape ID 복원 방식으로 보강되었다.
- `input-handler.ts`, `input-handler-picture.ts`, `input-handler-mouse.ts`, `command/commands/*`에 직접
  `wasm.*` mutation 호출이 남아 있다.

## 추진 전략

하이퍼-워터폴 방식으로 단계별 승인과 판정을 받는다.

1. Stage 1 — 현황 인벤토리
   - 직접 mutation 호출을 읽기/쓰기/보기 설정/복합 편집으로 분류
   - `command`, `snapshot`, `record`, 직접 호출 경로를 표로 정리
   - Undo/Redo 사용자 영향도가 큰 경로를 우선순위화

2. Stage 2 — 복원 계약 표준안
   - command 유형별 복원 payload 원칙 정리
   - `execute()` 재호출 기반 redo 계약 명문화
   - `recordWithoutExecute()` 사용 조건 정의
   - snapshot 허용 조건 정의

3. Stage 3 — Router 1차 개선
   - 이름과 타입을 명확히 한 router/helper를 도입하거나 기존 `executeOperation()`를 보강
   - 직접 mutation 경로 중 작고 안전한 대상을 먼저 router로 이동
   - command별 refresh 정책을 문서화하고 필요한 최소 코드 보강

4. Stage 4 — 회귀 검증
   - 텍스트 입력/삭제 Undo/Redo
   - 글자 서식 Undo/Redo
   - 문단 서식 Undo/Redo
   - 표 기본 조작 Undo/Redo
   - 이미지/도형 이동·크기 조절 Undo/Redo
   - 붙여넣기/snapshot 경로 회귀

## 성공 기준

- 주요 편집 작업의 라우팅/Undo/Redo 복원 계약이 문서화된다.
- 우선순위가 높은 직접 mutation 경로 중 안전한 일부가 공통 라우터로 이동된다.
- #1319에서 보강한 문단/글자 서식 Undo/Redo가 회귀하지 않는다.
- 기존 텍스트, 표, 이미지/도형 Undo/Redo 기본 동작이 회귀하지 않는다.
- 작업지시자가 동작 테스트 또는 시각/UX 판정으로 통과를 확인한다.

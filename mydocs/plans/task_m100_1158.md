# 수행 계획서 — Task M100-1158: rhwp-studio 다크테마 지원

- 이슈: https://github.com/edwardkim/rhwp/issues/1158
- 제목: rhwp-studio 다크테마 지원 및 디자인 토큰 정비
- 작성일: 2026-06-16
- 브랜치: `local/task_m100_1158`

## 1. 문제 요약

rhwp-studio는 #22에서 CSS 디자인 토큰과 반응형 기반을 갖췄지만, 현재 색상 토큰은 light UI
중심이다. 메뉴바, 도구 상자, 서식 도구 모음, 상태바, 편집 작업영역, 공통 다이얼로그와
command palette에 하드코딩 색상과 기존 `--color-*` 토큰이 섞여 있어 단순한 dark palette 추가만으로는
부분 적용 상태가 될 가능성이 높다.

## 2. 목표

1. `system | light | dark` 테마 설정을 지원한다.
2. 사용자 선택을 `rhwp-settings` localStorage에 저장하고 새로고침 후 유지한다.
3. `prefers-color-scheme` 변경에 반응한다.
4. 다크테마는 앱 UI chrome에만 적용한다.
5. 문서 내용, 흰 편집 용지, 인쇄, SVG/export 결과는 테마에 의해 변형하지 않는다.
6. 기존 light 테마의 기본 시각을 크게 흔들지 않는다.

## 4. 구현 범위

1차 적용 범위:

- 아이콘 도구 상자
- 서식 도구 모음
- 상태바

## 5. 비목표

- 문서 페이지 또는 canvas 렌더 결과의 색상을 반전하지 않는다.
- HWP/HWPX 파서, renderer, export SVG/print 경로는 변경하지 않는다.
- 모든 개별 다이얼로그 CSS의 하드코딩 색상을 한 번에 완전 제거하지 않는다.
- 아이콘 스프라이트 자체의 다크 모드용 재채색은 1차 범위에서 제외한다.

## 6. 진행 단계

### Stage 1 — 테마 설정 기반

- `AppSettings`에 `theme.mode`를 추가한다.
- `system | light | dark` 타입과 저장/조회/갱신 API를 만든다.
- 앱 시작 시 `document.documentElement`에 `data-theme-mode`, `data-theme-effective`를 설정한다.
- `prefers-color-scheme` 변경을 감지해 system 모드에서 effective theme을 갱신한다.

### Stage 2 — semantic token과 UI chrome 적용

- `base.css`에 semantic token을 추가한다.
- light/dark effective theme별 색상을 정의한다.
- 메뉴바, 도구 상자, 서식 도구 모음, 상태바, 편집 작업영역, 공통 다이얼로그, command palette의
  주요 하드코딩 색상을 token 기반으로 전환한다.
- `--doc-paper`는 dark에서도 `#ffffff`를 유지한다.

### Stage 3 — 보기 메뉴와 검증

- `보기 > 테마 > 시스템 설정 / 밝게 / 어둡게` 항목을 추가한다.
- 현재 선택 테마가 메뉴에서 표시되도록 active 상태를 갱신한다.
- light/dark 기본 화면 e2e 스모크 테스트를 추가한다.
- TypeScript build와 관련 e2e를 실행한다.

- 다크테마에서도 편집 용지가 흰색으로 유지되는지 확인

## 8. 리스크 및 대응

| 리스크 | 대응 |
|---|---|

| 하드코딩 색상 일부 잔존 | 1차 범위 CSS를 명시하고 e2e/시각 확인으로 chrome 중심 검증 |

| light 테마 회귀 | 기존 `--color-*` 토큰은 가능한 호환 alias로 유지 |

# Task #630 수행계획서

`aift.hwp` 4페이지 목차의 `(페이지 표기)` 우측 정렬이 `·` (U+00B7) 포함 라인에서만 8.67px 좌측으로 이탈하는 결함 정정.

## 1. 문제 정의

### 1-1. 증상

| 라인 | `(` 시작 x | `·` 포함 |
|-----|---------|---------|
| 1-1 "개발 대상 기술·제품의 개요" | **592.36** | ✓ |
| 1-2 "과제의 배경 및 필요성" | 601.03 | ✗ |
| 3-1 "상용화 개발 추진방법·전략" | **592.36** | ✓ |
| 4-1 "국내·외 기술과 시장 현황" | **592.36** | ✓ |
| 7-2 "연구시설·장비 보유현황" | **592.36** | ✓ |
| 8-1 "사업비 지원·부담계획" | **592.36** | ✓ |

`·` 포함 라인만 정확히 8.67px(반각 1자) 좌측 이탈. PDF(한컴 출력) 정답지에서는 모든 라인이 동일한 우측에 정렬됨.

### 1-2. 본질 결함 (2가지 동시 작용)

**원인 A — 측정 불일치 (text_measurement.rs:863)**

`is_halfwidth_punct` 가 `U+00B7` 을 강제 반각(8.67px) 으로 처리. 한컴 저장 시점 측정값은 전각(17.34px) 으로 추정.

수치 검증:
- 두 라인의 `x_at_tab` 차이 (rhwp 측정): 43.33px
- SVG 결과 `(` 위치 차이로 역산한 `tab_width_px` 차이: 52.0px
- 차이의 차이: **8.67px = 한 개의 `·` 전각/반각 차이**

**원인 B — tab_type 해석 fallback (text_measurement.rs:361, line 247)**

native `compute_char_positions` / `estimate_text_width` 가 `tab_type = ext[2]` 을 raw u16 (RIGHT 인코딩 = `(2 << 8) | 3 = 515`) 으로 읽어 매치 실패 → wildcard LEFT fallback.

```rust
let tab_type = ext[2];  // 515 (raw u16)
match tab_type {
    1 => { /* RIGHT (right-align after-tab segment) */ }
    2 => { /* CENTER */ }
    _ => { x = tab_target.max(x); }  // ← 515 이 여기로 떨어짐
}
```

WASM 경로(line 732) 는 `inline_tab_type(ext)` (high-byte 추출) 으로 RIGHT 정확 매치. 즉 native/WASM 동일 코드 경로에서 동작 불일치.

코드 코멘트 (line 240–243) 가 "기존 golden SVG (issue-147, issue-267) 가 이 '우연한 LEFT 폴백' 동작에 의존" 명시 — 단순 수정 시 회귀 위험.

### 1-3. 두 원인의 조합 동작

LEFT fallback 분기에서:
```rust
let tab_target = x + tab_width_px;  // x = our x_at_tab, tab_width_px = HWP 저장값
x = tab_target.max(x);              // 결과: x = our_x_at_tab + tab_width_px
```

HWP 저장 시 한컴이 측정한 `HWP_x_at_tab` 기준으로 `tab_width = target_position - HWP_x_at_tab` 계산. 한컴이 `·` 를 전각으로 측정했다면 `HWP_x_at_tab = our_x_at_tab + 8.67`. 따라서:

```
final_x = our_x_at_tab + (target - HWP_x_at_tab)
        = our_x_at_tab + target - (our_x_at_tab + 8.67)
        = target - 8.67    ← 8.67 좌측 이탈
```

원인 A 만 정정해도, 원인 B 만 정정해도 결과 정합 가능. 단 양쪽 모두 정정해야 다른 케이스 (HWPX 인라인 탭, 표 우측 정렬 등) 회귀 차단.

## 2. 해결 방향

### 2-1. 권장안 (작업지시자 승인됨, 1+2 조합)

**수정 1**: `is_halfwidth_punct` 에서 `U+00B7` 제외 — 측정폭 한컴 정합

`src/renderer/layout/text_measurement.rs:859-862`
```rust
let is_halfwidth_punct = matches!(c,
    '\u{2018}'..='\u{2027}' |
    '\u{00B7}'                 // ← 이 라인 제거
);
```

영향:
- `·` 측정폭: 8.67px → 17.34px (전각, font 메트릭 그대로)
- SVG 시각: Task #257 의 `<circle>` 렌더는 `cluster_advance` 기반이라 자동으로 전각 advance 의 중앙에 그려짐 → 시각 자연스러움
- 한컴 PDF 와 `·` 좌우 시각 간격 정합 가능성 증가

**수정 2**: native `compute_char_positions` / `estimate_text_width` 의 `tab_type = ext[2]` 를 `inline_tab_type(ext)` 로 통일

`src/renderer/layout/text_measurement.rs:247` (estimate_text_width)
```rust
let tab_type = inline_tab_type(ext);  // raw u16 → high-byte
```

`src/renderer/layout/text_measurement.rs:361` (EmbeddedTextMeasurer::compute_char_positions)
```rust
let tab_type = inline_tab_type(ext);
```

영향:
- 이전 LEFT fallback 으로 처리되던 RIGHT/CENTER 인라인 탭이 정확히 정렬됨
- 코드 코멘트 (line 240–243) 의 "golden SVG 회귀 위험" 영역 — 광범위 검증 필수

### 2-2. 대안 검토 (반려)

- **수정 1 단독**: tab_type fallback 은 다른 케이스 (block table 셀 우측 정렬 등) 에 잠재적 영향 남음.
- **수정 2 단독**: `·` 측정폭이 한컴과 다르면 RIGHT 분기에서 `seg_w` 빼는 식이 흡수하지만, 다른 곳 (자연 폭 비교, justify slack 분배) 에서 잔여 오차 가능.
- 양쪽 정정으로 본질 일관화 + 회귀 위험 분산.

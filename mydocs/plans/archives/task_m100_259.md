# 수행계획서

## 1. 배경

작업지시자가 `samples/text-align.hwp` 4번 문단 (HY중고딕) 을 웹 에디터에서 열었을 때 **글자가 겹쳐 보임** 을 발견. 디버깅 결과, 한컴 한국어 폰트명 → 메트릭 DB 영문명 매핑이 대규모로 누락되어 글자 폭 계산이 기본값 (fallback) 으로 떨어지는 것이 원인.

작업지시자 지시로 본한글/본한글vf (Source Han Sans 계열) 도 함께 검토. 조사 결과 본한글 메트릭 자체가 DB 에 없어 **Pretendard 근사 매핑** 전략 채택.

## 2. 문제 본질

### 2.1 HY 계열 — "한글 정규명 → 영문 DB 이름" 매핑 부재

```
HWP 파일: "HY중고딕" (또는 별칭)
  ↓
style_resolver.rs:
  한국어 별칭 → 한국어 정규명 ("한양중고딕" → "HY중고딕")  ✅ 구현됨
  ↓
find_metric("HY중고딕", ...):
  ↓
resolve_metric_alias("HY중고딕")
  한국어 정규명 → 영문 DB 이름                             ❌ 매핑 없음
  ↓
FONT_METRICS 에서 "HY중고딕" 검색
  ↓ (DB 는 "HYGothic-Medium" 으로 저장)
None 반환 → 기본 폭 사용 → 글자 겹침
```

### 2.2 본한글 계열 — 메트릭 DB 엔트리 부재

- 본한글 (Source Han Sans) / 본명조 (Source Han Serif) / 본한글vf (variable font) 는 FONT_METRICS (396 엔트리) 에 **전무**
- `generic_fallback` 체인에 CSS fallback 으로만 간접 처리되나, rhwp 내부 폭 계산은 기본값 사용

### 2.3 차이

| 영역 | 원인 | 해결 경로 |
|---|---|---|
| HY 계열 (7종) | 매핑 누락 (DB 에 있음) | `resolve_metric_alias` 에 7건 추가 |
| 본한글 계열 (2종) | DB 부재 | **Pretendard 근사 매핑** (작업지시자 승인) |
| 본명조 계열 (1종) | DB 부재 | **Noto Serif KR 근사 매핑** (같은 방침) |

## 3. 목표

## 5. 근사 매핑의 정당성

본한글 → Pretendard 는 **정확값이 아닌 근사** 임을 명시.

**근거**:
1. **한글 원천 동일**: Pretendard 는 Inter + Source Han Sans KR 한글을 합성. 한글 글리프는 본한글과 같은 원천.
2. **이미 번들됨**: `web/fonts/Pretendard-{Regular,Bold}.woff2` 가 확장 dist 에 포함 → 크기 영향 0.
3. **라이선스 호환**: 둘 다 SIL OFL 1.1.
4. **실무 수용성**: 글자 겹침 없앰 + 시각 근사 달성 — 실사용에서 만족.

**한계**:
1. **Latin 폭 차이**: Pretendard 의 Latin 은 Inter 기반, 본한글 Latin 과 미세 차이.
2. **Weight 축 근사**: Pretendard 메트릭은 Regular + Bold 2단계. 본한글 ExtraLight/Light/Medium/Heavy 나 본한글vf 의 임의 wght 는 Regular/Bold 중 가까운 쪽으로 근사. CJK 폰트는 weight 별 한글 폭 차이가 작으므로 실무 허용.
3. **본명조 → Noto Serif KR 동일 원리**: 본명조는 serif 이므로 Pretendard 부적합, Noto Serif KR 이 같은 한글 원천.

**향후 고도화**: 본한글 TTF 를 `extract_metrics` 파이프라인으로 정식 DB 엔트리화. 별도 이슈에서 다룸.

## 1. 배경

`samples/bitmap.hwp`, `samples/한셀OLE.hwp` 를 rhwp-studio 에서 열면 파일은 정상 로드되나 **본문이 빈 페이지**로 렌더된다. 네이티브 `rhwp export-svg` 는 정상 동작 (BMP 이미지 포함 46KB SVG 생성).

세션 내 재현: `rhwp-studio/e2e/debug-load-bug.test.mjs` (puppeteer) + 스크린샷 `e2e/screenshots/debug-_bitmap.png`, `debug-_hancell_ole.png`.

## 2. 근본 원인

`src/renderer/layout/shape_layout.rs:983-1094` `ShapeObject::Ole` 처리 경로:
- OOXML 차트 → `RenderNodeType::RawSvg`
- CFB 컨테이너 → OOXML 차트 / EMF 프리뷰 / 네이티브 BMP·PNG·JPEG → `RawSvg`
- 모두 실패 → `Placeholder`

두 렌더러 match arm 비교:

| 노드 타입 | `src/renderer/svg.rs` | `src/renderer/web_canvas.rs` |
|-----------|----|----|
| `RawSvg` | 처리 O (조각 그대로 삽입) | **arm 부재 → `_ =>` 로 빠짐** |
| `Placeholder` | 처리 O (rect + 라벨) | **arm 부재 → `_ =>` 로 빠짐** |

`render_node` 의 `_ =>` 는 자식만 재귀하고, 두 노드 모두 자식이 없어 **아무것도 drawn 되지 않음**.

## 3. 범위

### 3.1 In-scope

1. **`web_canvas.rs` 에 `Placeholder` arm 추가** — 배경 rect + 중앙 라벨 텍스트 (svg.rs 동등 출력)
2. **`web_canvas.rs` 에 `RawSvg` arm 추가** — SVG 조각 내 `<image xlink:href="data:...">` 를 파싱하여 canvas `drawImage` 로 렌더
   - 대상: OLE CFB 의 BMP/PNG/JPEG 네이티브 이미지 (shape_layout.rs:1059-1062 형식)
   - 대상: EMF 폴백 SVG 조각 (shape_layout.rs:1028, `crate::emf::convert_to_svg` 출력) — 실제 SVG 요소 집합
   - 대상: OOXML 차트 SVG (shape_layout.rs:990, `OoxmlChart::render_svg`)

### 3.2 구현 전략

RawSvg 조각은 **텍스트 SVG**이므로 canvas 에서 직접 그릴 수 없다. 두 방식 중 선택:

**A안) SVG blob → `HtmlImageElement` → `drawImage`**
- `<svg>...</svg>` 로 래핑 → Blob URL → Image().onload → drawImage
- 장점: 모든 SVG 형식 (image·path·text·gradient) 통일 지원
- 단점: 비동기 → 현재 동기 렌더 구조 (`render_node(&self, node)`) 에 맞지 않음. 이미지 캐시 패턴 (`web_canvas.rs:24-28` `IMAGE_CACHE`) 재사용 가능하나 SVG 는 콘텐츠 해시가 커서 캐시 효율 낮음

**B안) SVG 텍스트 파싱 → canvas 명령 호출**
- 현재 OLE 경로에서 생성되는 SVG 조각은 대부분 단순 `<image>` 단일 요소 (BMP/PNG/JPEG 폴백)
- EMF 변환기(`src/emf/`) 가 만드는 SVG 요소는 `path/line/rect` 등 다양
- 장점: 동기, 캐시 친화
- 단점: 전체 SVG parser 를 web_canvas 에 넣는 것은 과함

**권장**: **하이브리드**
- `<image>` 단독 조각 (최다수 케이스) → href 파싱 → 기존 `draw_image` 재사용 (동기 유지, IMAGE_CACHE 그대로)
- 그 외 (EMF → path 계열 SVG 다중 요소 등) → A안 SVG-to-Image 폴백 (`HtmlImageElement` async 로드 + 후속 프레임 재렌더 트리거)

구현 단계에서 실제 조각 구조를 살펴보고 확정한다.

### 3.3 Out-of-scope

- `svg.rs` 측 렌더러 변경 (현재 정상 동작)
- `RawSvgNode` 의 데이터 모델 변경 (조각 대신 구조화된 노드 트리 만들기) — M101+ 후보

## 6. 위험 / 트레이드오프

- **비동기 렌더**: A안 SVG-Image 는 async 로드로 첫 프레임이 빈 채로 그려지고 Image onload 에서 캔버스에 덮어그려야 함. 기존 IMAGE_CACHE 로드 실패 시 placeholder 를 그리는 패턴 (`web_canvas.rs:1560`) 을 참고해 동일 구조로 맞춘다.
- **EMF 조각 형식**: `crate::emf::convert_to_svg` 의 실제 출력이 단일 `<g>` 인지 다중 요소인지 구현 단계에서 확인 필요.
- **정밀도 손실**: SVG→Image 로 해상도 고정됨. 줌 변경 시 재렌더 트리거가 필요할 수 있음 (기존 이미지와 동일한 DPR 처리 흐름에 편승).

# Task #359 수행계획서 — 페이지네이션 fit 판정 vs Layout y 진행 정합

## 이슈

[#359](https://github.com/edwardkim/rhwp/issues/359) — `samples/k-water-rfp.hwp` 페이지 1 의 LAYOUT_OVERFLOW (260~288px). pagination 단계의 fit 판정 (used=728.1px, col_height=915.5px 이내) 과 layout 단계의 실제 y 진행 (마지막 항목 y=1316.8) 사이 큰 드리프트.

## 배경

v0.7.6 사이클에서 외부 기여 PR 다수 머지 (Task #321~#332 통합 #343, Task #347 #351 등) 후 페이지 분할 정합성 회귀 발생. 본 task 는 **현재 코드 기준 정공법 분석** — 원복이나 히스토리 추적 없이 정확한 원인 식별 + 수정.

### 정공법 원칙

- 이전 commit 으로 reset 하지 않음
- 어느 PR 이 회귀를 도입했는지 추적하지 않음
- **현재 코드의 pagination fit 산정 로직 + layout y 진행 로직 두 경로의 정합성** 직접 분석

### 검증 방식 (작업지시자 결정)

- **PDF 비교 없음** — `samples/k-water-rfp.pdf` 등 한컴 PDF 와 직접 비교 불가
- **작업지시자 시각 판정** — Stage 4 의 WASM 빌드 후 작업지시자가 직접 시각으로 페이지 분할의 적정성 판정
- LAYOUT_OVERFLOW 경고 0 건 + 작업지시자 시각 통과를 정합 기준으로 사용

## 증상 정량

```
LAYOUT_OVERFLOW: page=0, col=0, para=34, type=FullParagraph, y=1288.0, bottom=1028.9, overflow=259.1px
LAYOUT_OVERFLOW_DRAW: section=1 pi=35 line=0 y=1304.0 col_bottom=1028.9 overflow=275.1px
LAYOUT_OVERFLOW: page=0, col=0, para=35, type=PartialParagraph, y=1316.8, bottom=1028.9, overflow=287.9px
```

| 단계 | 값 |
|------|----|
| body_area.y | 113.3 |
| body_area.h | 915.5 |
| col_bottom (절대) | 1028.9 |
| pagination 의 used | 728.1 |
| layout 마지막 y (pi=35) | 1316.8 |
| **드리프트** | layout 1316.8 − body 113.3 − used 728.1 = **475.4px** 정도 부정합 |

페이지 1 items=33 (pi=0~31, Shape pi=31 ci=0). pi=34, pi=35 가 layout 에서 페이지 1 에 잔존하여 col_bottom 초과. pagination 은 33 items 만 페이지 1 에 배정했지만 layout 의 y 가 그보다 더 빠르게 진행되어 충돌.

## 목표

1. **드리프트 origin 정량화** — 어느 항목 (또는 어느 패턴) 에서 fit 판정과 layout y 진행이 어긋나는지 식별
2. **정합 수정** — pagination 의 height 산정과 layout 의 y 진행을 동일 기준으로 통일
3. **회귀 검증** — 7 핵심 샘플 + form-002 + k-water-rfp 페이지 수 + LAYOUT_OVERFLOW 0건

## 범위

### 수정 대상 (예상)

- `src/renderer/typeset.rs` — pagination 의 fit 산정 (`format_paragraph::total_height`, `current_height` 누적)
- `src/renderer/layout.rs` — layout 의 y 진행 (`y_offset` 갱신, vpos 보정 등)
- `src/renderer/layout/paragraph_layout.rs` — paragraph 의 line_height/spacing 사용
- 기타 fit/y 산정 관여 위치

### 제외

- 외부 API 변경 없음
- 골든 SVG 갱신은 수정 결과의 부수효과 — 의도된 변경만 UPDATE_GOLDEN

## 접근 원칙

> v0.7.6 누적 작업의 결과로 fit/y 두 경로가 다른 기준을 사용하게 됐을 가능성. 두 경로를 같은 데이터 (LINE_SEG.vpos 또는 fmt.line_advances) 로 통일하면 드리프트 0 으로 수렴 기대.

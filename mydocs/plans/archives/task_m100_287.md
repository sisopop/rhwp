## 배경

`samples/exam_math_8.hwp` 를 `rhwp export-svg` 로 내보내면, 박스 안의 **큰 cases 수식**이 박스 좌상단(body-clip 원점, (71.8, 147.38) px)에 찍혀 박스 테두리와 앞 텍스트 줄에 겹친다. 같은 문단 내 작은 인라인 수식(`n`, `m`, `3`, `|a_m|=|a_{m+2}|`)은 정상 배치된다.

네 수식 모두 `tac=true`(인라인)이지만, 큰 수식은 HWP 의 `LINE_SEG` 에서 자기 몫의 전용 줄(ls[1], lh=4095 HU, vpos=1898)을 갖는다. 렌더러가 이 전용 줄의 vpos/lh 정보를 반영하지 못해 문제가 발생한다.

## 목적

1. `exam_math_8.hwp` 큰 수식이 한컴 PDF 와 시각적으로 동일한 위치·정렬로 렌더되도록 한다.
2. 동일 원인을 갖는 다른 샘플(큰 display 수식이 단독 줄을 차지하는 경우)도 함께 개선한다.
3. 기존 수식 샘플(`exam_math.hwp`, `equation-lim.hwp`)에 회귀가 없다.

## 조사로 확정된 사실

- 문단 0.0 `LINE_SEG`: ls[0] vpos=0/lh=1150, **ls[1] vpos=1898/lh=4095**, ls[2] vpos=6741/lh=1150
- 문제 수식은 ls[1] 전용이며 `size=24569×4095 HU`, `tac=true`
- 현재 SVG 출력: `<g transform="translate(71.80, 147.38) scale(0.9736,1)">` (col_area 원점 그대로)
- 경로: `src/renderer/layout/paragraph_layout.rs:1574-1620` 의 TAC 인라인 수식 분기
- `paragraph_layout.rs:716-729` 의 `has_tac_shape && raw_lh > max_fs * 1.5` 보정 로직이 `Control::Shape` 만 검사 — **TAC Equation 미포함**
- `paragraph_layout.rs:744-751` 의 `text_y = y + (raw_lh - line_height)` 시프트도 Shape 전용
- `paragraph_layout.rs:1586`: `eq_y = (y + baseline - layout_box.baseline).max(y)` — layout_box.baseline > text baseline 일 때 `.max(y)` clamp 가 음수 보정을 막아 줄 상단에 고정

## 접근 방침

**원칙**: ls[1] 의 `vpos`/`line_height` 가 실제로 렌더 파이프라인에서 어디서 소실되는지 먼저 확인하고, 그 결과에 따라 다음 둘 중 하나로 수정한다.

- (A) compose/layout 흐름에서 ls 의 vpos·lh 가 정상 전파되도록 고침 — 근본 해결
- (B) TAC Equation 에 대해 `has_tac_shape` 동등 보정을 추가(조건 확장 + text_y 시프트 + eq_y 재계산) — 우회 해결

(A) 가능 시 (A) 우선. (A) 가 광범위 파급을 유발하면 (B) 로 좁힌다.

수식 x 위치 역시 현재 `col_area.x` 로만 고정되어 있으나 — HWP 정렬(수식 줄의 정렬 속성, 대개 중앙) 확인 후 (A)/(B) 어느 방향이든 x 계산에 반영한다.

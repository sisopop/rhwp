- **타스크**: [#257](https://github.com/edwardkim/rhwp/issues/257)
- **마일스톤**: M100
- **브랜치**: `local/task257`
- **작성일**: 2026-04-23
- **샘플**: `samples/text-align-2.hwp` (작업지시자 제공), `samples/text-align-2.pdf` (한컴 오피스 PDF 출력, 비교 기준)
- **선행 타스크**: [#146](https://github.com/edwardkim/rhwp/issues/146) v1~v4 (Geometric Shapes / TAC 표 선행 공백 / heavy face bold — `text-align.hwp` 수렴 완료)
- **배경 조사 문서**: `mydocs/tech/text_align_2_svg_pdf_compare.md`

## 1. 배경

Task #146 종결 이후 회귀 검증 목적으로 `samples/text-align-2.hwp` 를 `rhwp export-svg` 로 내보낸 뒤 한컴 PDF 와 150dpi 나란히 비교한 결과, **#146 이 고친 3대 버그(Geometric Shapes / TAC / heavy face bold)는 재현 없이 정상 렌더링**되나, **새 버그 후보 1건**이 발견되었다.

### 증상

narrow glyph — 실제 글리프 폭이 반각(`em/2`) 보다 훨씬 좁은 문자 — 뒤에 오는 글자가 한컴 PDF 대비 **3~5 px 우측으로 밀림**.

| 위치 | 한컴 PDF | rhwp SVG |
|------|---------|---------|
| 표 셀 숫자 | `1,000항목` | `1, 000항목` (콤마 뒤 과도 간격) |
| 표 셀 숫자 | `30,000항목` | `30, 000항목` |
| 표 헤더 중점 | `어휘·표현` | `어휘· 표현` (중점 뒤 과도 간격) |

(정량: `output/svg/text-align-2/text-align-2.svg` ↔ `output/compare/text-align-2/pdf-1.png` 참조)

### 원인 분석 (사전 조사 완료)

위치: `src/renderer/layout/text_measurement.rs:260-290` `compute_char_positions`.

```rust
let base_w = if let Some(w) = measure_char_width_embedded(...) {
    w
} else if cluster_len[i] > 1 || is_cjk_char(c) || is_fullwidth_symbol(c) { font_size } else { font_size * 0.5 };
let mut w = base_w * ratio + style.letter_spacing + style.extra_char_spacing;
...
if style.letter_spacing + style.extra_char_spacing < 0.0 {
    let min_w = base_w * ratio * 0.5;
    w = w.max(min_w);
}
```

**두 단계 휴리스틱 모두 narrow glyph 에 불리하게 작용**:

1. **`base_w = font_size * 0.5` (반각 휴리스틱)**
   - 콤마(`,` U+002C)·마침표(`.`)·중점(`·` U+00B7) 등은 실제 글리프 advance 가 `font_size * 0.15 ~ 0.30` 수준
   - `is_fullwidth_symbol` 는 U+00B7(중점)을 포함하지 않음 (확인 완료: `text_measurement.rs:845-859`)
   - `measure_char_width_embedded` 에 HY헤드라인M·휴먼명조 미등록 → 폴백 진입
2. **`min_w = base_w * ratio * 0.5` 클램프**
   - 음수 자간(-8% 등) 시에도 `w` 가 반각의 절반(=`font_size * 0.25`) 아래로 못 내려감
   - 실제 PDF 가 렌더하는 advance(`font_size * 0.15 ~ 0.20`)를 재현하지 못함

**결론**: narrow glyph 에서 rhwp 는 실제 폰트 메트릭보다 약 2~3 px 과도한 advance 를 부여한다. 뒤 글자가 그만큼 우측으로 밀린다.

### 영향 범위

- 표 셀 내 숫자 (`1,000`, `30,000` 등 콤마 포함 숫자 전반)
- 일반 본문 내 콤마·마침표·중점·세미콜론·콜론이 다수 쓰이는 문서
- 콤마·중점 빈도가 높은 시험지·보고서·사양서 등
- `text-align.hwp` (#146 샘플) 에는 해당 문자가 없어 #146 회귀 검증에서 드러나지 않았음

## 2. 목표

narrow glyph 뒤 글자 x 좌표가 한컴 PDF 와 **육안상 동등**(≤ 1 px)해지도록 `compute_char_positions` 의 fallback 폭 휴리스틱을 보정한다.

**범위 (IN)**

- **N-1**: narrow glyph 감지 헬퍼 추가 (`is_narrow_punctuation(c) -> bool`) — Unicode General_Category Punctuation 중 좁게 렌더되는 집합 (`,` `.` `·` `:` `;` `'` `"` 등)
- **N-2**: `compute_char_positions` 의 `base_w` 폴백을 narrow glyph 용 별도 값(`font_size × 0.3` 수준)으로 분기
- **N-3**: 음수 자간 시 `min_w` 클램프가 narrow glyph 에 적용되지 않도록 우회
- **N-4**: 단위 테스트 — narrow glyph 목록별 advance 검증 + `text-align-2.hwp` 표 셀 `1,000` / `어휘·표현` 좌표 골든 회귀
- **N-5**: 재현 검증 — `text-align-2.hwp` SVG 를 Chrome 150dpi 로 렌더 후 PDF 와 좌우 비교해 해당 글자 x 좌표 수렴 확인

**범위 (OUT)**

- 실제 HY헤드라인M/휴먼명조 폰트 메트릭 DB 편입 (폰트 라이선스 이슈, `font_fallback_strategy.md` 범위)
- `measure_char_width_embedded` 전반 개편 (본 타스크는 fallback 경로 한정)
- WASM/Canvas 렌더 경로 동일 적용 (`web_canvas.rs`) — SVG 경로와 로직 공유하면 자동 반영, 아니면 후속 조치
- 다른 샘플(`biz_plan.hwp` 등) narrow glyph 재측정 (본 타스크 후 회귀 테스트로 검증)

## 3. 구현 단계 (초안 — 구현 계획서로 상세화)

1. **단계 1 — 현상 재현·베이스라인 고정**
   - `samples/text-align-2.hwp` 의 표 셀 `1,000항목` / `어휘·표현` 의 현재 SVG char position 값 기록
   - 단위 재현 케이스: 자간 -8%, 음수 아닌 경우, ratio 1.0 vs 0.9 조합으로 `compute_char_positions` 직접 호출 테스트 작성
   - PDF 150dpi 기준 좌표값 문서화 (비교 대상 수치 확정)
2. **단계 2 — `is_narrow_punctuation` 헬퍼 추가 + `base_w` 분기**
   - `text_measurement.rs` 하단에 헬퍼 추가 (`,` `.` `·` `:` `;` `'` `"` `\`` 등)
   - `compute_char_positions` fallback 분기에 narrow glyph 감지 → `font_size × 0.3` 반환
   - 동일 로직을 `measure_segment_from` 등 공유 경로에도 점검 (별도 char_width 클로저 존재 시)
3. **단계 3 — `min_w` 클램프 우회**
   - 음수 자간 시 `if !is_narrow_punctuation(c) { w = w.max(min_w); }`
   - 단위 테스트: 자간 -8% + 콤마 → w 가 `font_size × 0.3 + letter_spacing` 그대로 유지 (클램프 미작동)
4. **단계 4 — 회귀 테스트 + 재현 검증**
   - `tests/svg_snapshot.rs` 영향 조사 (`biz_plan.hwp` 등 narrow glyph 다수 문서) — diff 발생 시 PDF 기준 개선인지 확인 후 UPDATE_GOLDEN
   - `samples/text-align-2.hwp` Chrome 150dpi PNG ↔ PDF 150dpi PNG 좌우 비교 (`output/compare/text-align-2/` 갱신)
   - `cargo test --lib` / `cargo test --test svg_snapshot` / `cargo clippy --lib -- -D warnings` 통과

(세부 진입점·커밋 단위는 `task_m100_257_impl.md` 로 분리 작성)

## 4. 검증 방법

- `cargo run --bin rhwp -- export-svg samples/text-align-2.hwp -o output/svg/text-align-2/` 재생성 후 표 셀 `1,000항목` / `어휘·표현` 의 각 글자 x 좌표가 PDF 환산값과 **≤ 1 px** 일치
- Chrome headless 150dpi PNG (`output/compare/text-align-2/svg-chrome150.png`) ↔ `pdf-1.png` 시각 비교로 콤마·중점 뒤 간격 수렴 확인
- `cargo test --lib` 전체 통과 + 신규 테스트 2건 추가 통과 (narrow glyph advance · 음수 자간 clamp 우회)
- `cargo test --test svg_snapshot` 통과. 영향 golden 은 diff 사유 기록 후 재생성
- `cargo clippy --lib -- -D warnings` 통과

## 5. 리스크·가정

| 리스크 | 대응 |
|--------|-----|
| narrow glyph 목록 과다 확장 → 기존 샘플에서 공백 축소 regression | 화이트리스트로 시작 (8~10자 이내). Unicode 범주 기반 자동 확장은 별도 이슈 |
| `font_size × 0.3` 값이 HY헤드라인M 과 휴먼명조에서 서로 다른 최적값을 가짐 | 두 폰트 모두 narrow glyph 는 글리프 품질이 매우 유사 → 한 값으로 근사 가능. PDF 정밀 좌표 확인 단계에서 조정 |
| `svg_snapshot` 골든 대량 diff | 예상 범위 먼저 조사 후 golden 재생성 계획을 단계4 내에서 승인 요청 |
| `measure_char_width_embedded` 에 이미 콤마 등이 부분 등록되어 있을 가능성 | 단계1 에서 embedded 경로 hit 여부 로깅으로 확인 (hit 시 해당 경로 수정으로 범위 이동) |

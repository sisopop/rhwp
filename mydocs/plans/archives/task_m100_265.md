## 2. 원인 분석

파일 처음 16바이트:
```
4857 5020 446f 6375 6d65 6e74 2046 696c   = "HWP Document Fil"
6520 5633 2e30 3020 1a01 0203 0405 0000   = "e V3.00ᚱ....."
```

**HWP 3.0 바이너리 포맷** (한글 3.0, 1996~1998년 경 버전). 현재 rhwp 는 HWP 5.0 (CFB) 과 HWPX (ZIP) 만 지원 ([src/parser/mod.rs:49-62](src/parser/mod.rs#L49-L62)).

에러 메시지의 매직 넘버 `[48, 57, 50, 20, 44, 6f, 63, 75]` 는 ASCII 로 `HWP Docu` — HWP 3.0 시그니처의 처음 8바이트. 저수준 CFB 에러가 사용자에게 노출되어 파일 특성(HWP 3.0)을 알 수 없는 혼란스러운 UX.

## 4. 아키텍처 계획

### 4.1 `src/parser/mod.rs` — `FileFormat` 확장

```rust
pub enum FileFormat {
    Hwp,      // HWP 5.0 (CFB)
    Hwpx,     // HWPX (ZIP+XML)
    Hwp3,    // HWP 3.0 바이너리 (미지원, 감지만)
    Unknown,
}

pub fn detect_format(data: &[u8]) -> FileFormat {
    if data.len() >= 8 {
        // CFB/OLE (HWP 5.0)
        if data[0..4] == [0xD0, 0xCF, 0x11, 0xE0] {
            return FileFormat::Hwp;
        }
        // ZIP (HWPX)
        if data[0..4] == [0x50, 0x4B, 0x03, 0x04] {
            return FileFormat::Hwpx;
        }
        // HWP 3.0: "HWP Document File V3.00"
        if data.len() >= 23 && &data[0..23] == b"HWP Document File V3.00" {
            return FileFormat::Hwp3;
        }
    }
    FileFormat::Unknown
}
```

### 4.2 `ParseError` — 신규 variant

```rust
pub enum ParseError {
    ...
    UnsupportedFormat { format: &'static str, hint: &'static str },
}

impl Display for ParseError {
    ...
    ParseError::UnsupportedFormat { format, hint } =>
        write!(f, "지원하지 않는 포맷: {}. {}", format, hint),
}
```

### 4.3 `parse()` 진입점 — Hwp3 감지 시 UnsupportedFormat 반환

```rust
match detect_format(data) {
    FileFormat::Hwp   => ... (기존)
    FileFormat::Hwpx  => ... (기존)
    FileFormat::Hwp3 => Err(ParseError::UnsupportedFormat {
        format: "HWP 3.0",
        hint: "현재 rhwp 는 HWP 5.0 과 HWPX 만 지원합니다. 한컴오피스 또는 LibreOffice 에서 HWP 5.0 포맷으로 다시 저장 후 시도해주세요.",
    }),
    FileFormat::Unknown => ... (기존, 에러)
}
```

### 4.4 프론트엔드 연쇄 영향

`ParseError` 는 `Display` 통해 WASM 경계로 문자열로 전달되므로, HOP · Chrome ext · github.io 3곳 모두 별도 수정 없이 새 에러 메시지가 자연 표시됨. 각 프론트엔드의 에러 표시 로직이 `Error::message` 를 그대로 노출한다면 수정 불필요. 확인 필요.

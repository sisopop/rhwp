# Task #312 수행계획서: column 가용 공간 계산 정확도 조사

상위 Epic: #309
선행: #310 (분석 도구), #311 (vpos-reset 가설 부정)
브랜치: `task312` (task311에서 분기)

## 목적

우리 페이지네이션 엔진의 column 가용 공간 계산이 HWP의 실제 사용 공간보다 관대한 원인을 식별하고 보정하여, **21_언어 SVG 19쪽 → 15쪽 (PDF 일치)** 달성.

## 배경 / 데이터 근거

#311 검증 결과: 페이지 7 21_언어 사례
- HWP: `pi=117 line 0` 까지만 단에 배치 (단 사용 ≈ vpos=89700 HWPUNIT = 1196px)
- 우리: `pi=117 전체` (line 0+1, ~29.4px) 단에 채움 → 단 가용 1226.4px
- 차이 ≈ 16~30px (정밀 측정 필요)

vpos-reset 강제 분리만으로는 효과 없음 (#311). 단 가용 공간 자체를 줄여야 후속 paragraphs가 자연 흐름으로 다음 단으로 이동.

## 작업 범위

### 1. 정밀 측정 도구 (1단계)

`dump-pages` 또는 신규 `diag-column-fit` 명령에서:
- 각 단의 누적 used height (`current_height` 진행)
- HWP가 의도한 단 사용 공간 (vpos-reset 직전 줄의 vpos + line_height)
- 차이 (px, HWPUNIT, 기여 요인별 breakdown)

도구 출력 예:
```
페이지 7 단 0: avail=1226.4px, our_used=1208.7px, hwp_intended=1196.0px, diff=+12.7px
  contributors: trailing_ls=+8.0, sp_after_last=+4.7, ...
```

### 2. 차이 origin 식별 (2단계)

후보 (`engine.rs:602~620` 등):

a. **Trailing line_spacing 처리**
```rust
let trailing_ls = para.line_segs.last()
    .map(|seg| crate::renderer::hwpunit_to_px(seg.line_spacing, self.dpi))
    .unwrap_or(0.0);
let effective_trailing = if st.current_height + para_height > available_now { ... };
st.current_height + (para_height - effective_trailing) <= available_now + 0.5
```
trailing 제외 정책이 HWP 실제 동작과 다를 가능성

b. **Spacing-after 마지막 문단 처리**
페이지 마지막 문단의 sp_after를 단 사용 공간에 포함시키는지 여부

c. **Line height 계산 vs LINE_SEG line_height**
`MeasuredParagraph.line_heights`는 우리 폰트 메트릭으로 재계산. HWP의 LINE_SEG `line_height` 직접값과 차이 가능

d. **부동소수점 오차 0.5px 마진**

e. **컨트롤(표/Shape) 후 문단 처리** - inline control 뒤의 줄 간격

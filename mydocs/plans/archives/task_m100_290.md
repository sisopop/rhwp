### 2.3 근본 원인 위치

`src/renderer/layout/paragraph_layout.rs:1213-1226`

```rust
if has_tabs && run.text.ends_with('\t') {
    if let Some(last_tab_pos) = run.text.rfind('\t') {
        let text_before_tab = &run.text[..last_tab_pos];
        let w_before = estimate_text_width(text_before_tab, &text_style);
        let abs_before = text_style.line_x_offset + w_before;
        let tw = if tab_width > 0.0 { tab_width } else { 48.0 };
        let (tp, tt, _) = find_next_tab_stop(                  // ← inline_tabs 무시
            abs_before, &tab_stops, tw, auto_tab_right, available_width,
        );
        if tt == 1 || tt == 2 {
            pending_right_tab_render = Some((tp, tt));         // ← 잘못된 설정
        }
    }
}
```

마지막 `\t` 의 종류를 판정할 때 `find_next_tab_stop` (TabDef 전용) 만 사용. 본문 `tab_extended` (inline_tabs) 는 참조하지 않음. 따라서 inline_tabs 가 LEFT 으로 명시한 `\t` 도 TabDef stop 이 모두 abs_before 보다 작으면 `auto_tab_right` 폴스루로 무조건 RIGHT 으로 오판되어 다음 run 이 우측 끝으로 역산 배치됨.

    // run 시작 시점의 inline_tabs 인덱스 = composed.tab_extended 의 누적 \t 카운트
    let last_inline_idx = inline_tab_cursor + total_tabs_in_run - 1;

### 4.2 inline_tab_cursor 추적

cross-run 판정은 "이 run 의 마지막 `\t` 가 composed.tab_extended 의 어느 인덱스인가" 를 알아야 함. 현재 paragraph_layout 에서 run 진행 중 inline_tabs 인덱스를 추적하는 변수가 있는지 단계 1 에서 확인. 없으면 도입.

### 4.3 est_x 측 (`paragraph_layout.rs:807-820`)

`pending_right_tab_est` 도 동일하게 `find_next_tab_stop` 만 사용한다면 동일 보정 필요. 단계 1 에서 코드 확인 후 결정.

### 4.4 ext[2] 의 정확한 의미 — 단계 1 에서 검증

현재 트레이스에서 확인된 값: `ext[2] = 256` (모든 LEFT 탭). HWP 사양상 `ext[2]` 가 type 인지, 아니면 다른 필드인지 (256=0x0100 는 valid tab type 아님) — 다음 중 하나로 결정:

- (가) `ext[2]` 의 하위 byte 가 type — 256 은 0x00 = LEFT
- (나) `ext[2]` 자체가 type — 0/1/2/3 외 값은 모두 LEFT 로 폴백 (현재 코드 동작과 일관)
- (다) ext 의 다른 인덱스가 type

샘플 케이스 외에 RIGHT/CENTER 탭이 포함된 HWP 샘플을 찾아 ext 값을 확인하면 확정 가능.

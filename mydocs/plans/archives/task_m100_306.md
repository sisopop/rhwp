# Task #306: 21_언어 페이지네이션 과잉 (19쪽 → 15쪽) — 수행계획서

## 목표

`samples/21_언어_기출_편집가능본.hwp` 의 SVG 출력을 PDF 원본 15쪽과 동일한 쪽수로 맞춘다. Task #304 완료 후 남은 2차 이슈.

## 현황

| | 쪽 수 |
|--|------|
| PDF 원본 | 15 |
| SVG (현재) | 19 (+4) |

각 쪽 구조(바탕쪽·2단 조판·문항 번호)는 PDF와 일치하나, 한 쪽당 본문량이 적어 콘텐츠가 뒷쪽으로 밀림.

## 초기 조사 (dump-pages)

### 쪽별 배치 요약

전체 19쪽에서 `단 1`(우측 단) 존재 여부:

- 모든 쪽: 단 0 존재
- **페이지 4**: 단 0만 (items=9), **단 1 없음** ← 이상 징후
- 기타 쪽: 단 0 + 단 1 모두 존재

### 페이지 4 상세 (아이템 높이 합계 ≈ 532px, body 1226.4px, 43% 채움)

```
body_area: x=117.2 y=209.8 w=888.2 h=1226.4
단 0 (items=9)
  FullParagraph  pi=71  h=39.6 "6.-윗글에 따를 때..."
  Table          pi=72 ci=0  3x3  410.8x243.4px  wrap=TopAndBottom tac=true
  FullParagraph  pi=73~77  h=44.0 각 (5문항 선택지 ①~⑤)
  FullParagraph  pi=78, 79 h=14.7 "(빈)"
```

페이지 4의 `단 0`이 절반만 차고 `단 1`로 흐르지 않고 새 쪽 시작. PDF는 여기에 더 많은 내용을 담음. `Table wrap=TopAndBottom tac=true` 가 의심 요소.

## 원인 가설

### 가설 1: 다단 조판에서 `wrap=TopAndBottom` 표 이후 단 전환 로직 오류
표 바로 아래 문단들이 표와 함께 `단 0`에 묶이고, 표 높이 때문에 `단 1`로 넘어가지 않고 쪽이 끝나버림. 본래 표 아래 선택지들은 `단 1`로 흐르거나 다음 쪽 `단 0`으로 가야 함.

### 가설 2: 2단 문서의 단 균형(balancing) 과소
`column balancing` 이 단 0을 충분히 비우지 않고 단 1로 넘겨서, 단 0·단 1 둘 다 덜 채운 뒤 페이지 종료.

### 가설 3: 문단 높이(ls/sb/sa) 과다 계산
각 문단이 PDF보다 높게 계산되어 누적 콘텐츠가 쪽을 더 빨리 채움. 누적 오차가 19쪽까지 쌓임.

### 가설 4: 빈 문단/줄 처리 과다
`(빈)` 문단이 PDF에선 압축되거나 최소 높이로 처리되는데 rhwp가 전체 줄 높이(14.7px)로 계산. 페이지 하단 잉여 공간 잠식.

## 조사 단계 (Stage 1)

### Stage 1-1. 페이지 4 집중 분석 (가설 1 검증)
- PDF 페이지 4의 실제 콘텐츠 확인 → 어느 문단까지 담겨 있는지 식별
- `pi=71~79` 이후 `pi=80, 81, ...` 가 PDF 4쪽에 있다면, 우리 렌더러가 잘못 쪽을 끊은 것
- `dump -s 0 -p 71,72,73,...` 로 ParaShape·LineSeg 상세
- `pagination/engine.rs` 에서 `wrap=TopAndBottom` + `tac=true` 표 처리 경로 추적

### Stage 1-2. 문단 높이 계산 검증 (가설 3 검증)
- 몇몇 문단의 `lines` 높이를 PDF의 동일 문단과 비교
  - 예: pi=73 "①-대상에 따라..." h=44.0 → PDF에서는 몇 줄인지 확인
- line_height·line_spacing·before/after 누적을 수동 재계산
- 차이 있으면 `compute_line_height` / `paragraph_height` 함수 점검

### Stage 1-3. 빈 문단 처리 확인 (가설 4 검증)
- 본 샘플에서 `(빈)` 문단이 얼마나 자주 나오는지 집계 (grep)
- PDF에서 해당 빈 문단이 실제로 rendering에 공간을 차지하는지 관찰
- 필요 시 ParaShape의 `hide_empty_line` 플래그 적용 여부 확인

### Stage 1-4. 컬럼 밸런싱 정책 확인 (가설 2 검증)
- `pagination/engine.rs` 에서 단 0 → 단 1 전환 트리거 조건
- 쪽별 items 분포(단 0 items=7 / 단 1 items=21 처럼 심한 불균형)를 세어 균형 붕괴 횟수 파악

### Stage 1-5. 원인 특정 → 구현 계획서 작성
- Stage 1-1~1-4 결과 종합
- `task_m100_306_impl.md` 작성 (수정 파일·함수 식별, 3~6단계 구현 계획)
- 승인 요청

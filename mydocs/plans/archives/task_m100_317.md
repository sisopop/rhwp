# Task #317 수행계획서: HWPX 어댑터 페이지 수 +1쪽 잔존 origin 조사

상위: 잔존 사안 (Epic #309 외부)
선행: #313 (TypesetEngine main 전환), #314 (어댑터 normalize 부분 완료)
브랜치: `task317` (task314에서 분기)

## 목적

#314 의 normalize 적용 후에도 잔존하는 hwpx-h-02 의 페이지 수 +1쪽 차이의 진짜 origin 식별 + 보강. 격리된 3개 테스트 재활성화.

## 배경

### 현재 상태 (#314 이후)

```
[#178 Stage 4] hwpx-h-02: orig=9, after_adapter=10
```

#314 normalize 로 paragraph 필드 차이 86건 해소했으나 페이지 수 차이는 그대로. 이는 **이미 해소된 차이가 paginate 결과에 직접 영향이 아니었음**을 의미.

### 잔존 paragraph 필드 차이 (#314 stage1 측정)

| 필드 | 잔존 건수 | 비고 |
|------|----------|------|
| `char_count` | 3 | paragraph 0.0, 0.34, 1.0 |
| `char_count_msb` | 2 | 마지막 paragraph 표시 |
| `raw_break_type` | 4 | column_type=Page 동일이나 raw 0x00 vs 0x04 |
| `raw_header_extra` | 130 | 전체 paragraph (header 끝 padding) |
| `controls_len` | 2 | 어댑터 SectionDef 의도 |

가장 의심: `char_count` (특히 0.34 — +1쪽 발생 페이지 3 의 paragraph)

### 페이지 3 단 0 의 차이

#314 stage1 측정:
- direct used=1268.5px, items=20 (Table pi=51 포함)
- reloaded used=595.5px, items=18 (Table pi=51 빠짐, 다음 페이지로)

같은 paragraph 가 어디서 누적 차이가 발생하는지 line-by-line 추적이 필요.

## 작업 범위

### 1. 잔존 차이 정밀 측정

- `char_count` 차이 paragraph 의 실제 값 비교 (text 길이 vs char_count)
- `raw_break_type` 영향 trace (typeset이 어디서 사용?)
- 직렬화/파싱 라운드트립 손실 확인 (paragraph 외 데이터: style, doc_info, 표 cell paragraph 의 다른 필드)

### 2. 페이지 3 단 0 paragraph-by-paragraph 추적

진단 도구 추가/활용:
- direct vs reloaded 의 paragraph 추가 시점에 current_height 누적 비교
- 어디서 누적 차이가 시작되는지 식별
- TypesetEngine 의 fits 판정 input 비교 (paragraph height, available, controls 처리)

### 3. 식별된 origin 보강

후보:
- a) HWPX 파서가 채우지 못하는 paragraph 필드를 normalize 에 추가
- b) HWP 직렬화기가 잃는 정보를 보존하도록 수정
- c) HWP 파서의 default 채움이 실제 spec 과 다른 경우 수정
- d) TypesetEngine 의 paragraph 처리 차이 (가능성 낮음)

### 4. 격리 테스트 재활성화

- `tests/hwpx_to_hwp_adapter.rs` 의 `#[ignore]` 3건 제거
- 사유 메시지 제거
- 통과 확인

### 5. hwpx-h-01, h-03 동시 검증 (회귀 방지)

## 비범위

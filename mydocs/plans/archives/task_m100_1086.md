## 4. 재검토 기준

이번 재검토에서는 기존 폐기 진행분의 root cause 단언을 재사용하지 않는다. 대신 다음 기준으로 다시 확인한다.

1. 한컴 정답 27 pages 와 rhwp 29 pages 의 page-by-page 매칭
2. `pi=52`, `pi=180`, `pi=255` 세 표 각각의 추가 페이지 기여량 계량
3. 기존 분할 엔진의 단일 권위 함수 확인
   - `src/renderer/typeset.rs::typeset_block_table`
   - `src/renderer/layout/table_layout.rs::advance_row_cut`
   - `src/renderer/layout/table_layout.rs::advance_row_block_cut`
   - `src/renderer/layout/table_layout.rs::row_cut_content_height`
   - `src/renderer/layout/table_partial.rs::layout_partial_table`
4. 구현 전 트러블슈팅 재확인
   - `page_margin_area_calculation`: k-water-rfp 29→27 과거 해결 이력. 이번 원인과 혼동 금지.
   - `typeset_page_num_newumber_application`: PartialTable 직후 안전마진 회귀.
   - `cell_split_nested_table_visibility`: 잔량 추적 대신 누적 위치 추적.

## 5. 가설

### H1 — `pi=52` 거대 셀 내부 paragraph 분할

`s1:pi=52` 의 `cell[14]` 는 24 paragraphs 를 포함하고, 셀 내부 vpos 가 중간에 0으로 재시작한다. 현재는 RowBreak 표의 행/셀 분할이 이미 존재하지만, 한컴 정답에서 이 셀이 더 자연스럽게 쪼개지는지 확인이 필요하다.

### H2 — `pi=180` 마지막 1행 orphan fragment

`s1:pi=180` 은 마지막 `rows=31..32` 한 행만 page 17 에 떨어진다. 목표 27 pages 를 위해서는 이 orphan fragment 가 실제 추가 페이지인지, 한컴은 page 16 에 수용하는지 확인해야 한다.

### H3 — `pi=255` 기존 `start_cut/end_cut` 경로의 잔여 영향

`s1:pi=255` 는 이미 셀 내부 cut 경로를 사용한다. 이것이 올바른 2-page split 인지, 또는 추가 페이지 연쇄에 기여하는지 확인한다.

## 1. 배경

PR #553 (Task #511 HWP3 Square wrap 보완) 작업 중 발견. 동일 문서를 HWP3 원본과 한컴이 HWP5/HWPX로 변환한 파일을 rhwp가 파싱했을 때 페이지 수가 다름.

| 파일 | 파서 | 페이지 수 (rhwp) | 한컴 뷰어 (정답) |
|------|------|------------------|------------------|
| hwp3-sample.hwp | HWP3 | **16** ✓ | 16 |
| hwp3-sample-hwp5.hwp | HWP5 | 17 ✗ | 16 |
| hwp3-sample-hwpx.hwpx | HWPX | 17 ✗ | 16 |
| hwp3-sample5.hwp | HWP3 | **64** ✓ | 64 |
| hwp3-sample5-hwp5.hwp | HWP5 | 68 ✗ | 64 |
| hwp3-sample5-hwpx.hwpx | HWPX | 68 ✗ | 64 |

작업지시자 확인: **HWP3 원본 페이지 수가 정답** (한컴 한글 뷰어 결과 일치). 한컴이 변환 시 자체 변경하는 것이 아닌, **rhwp의 HWP5/HWPX 파서/페이지네이션이 한컴 한글97의 "마지막 줄 tolerance" 동작을 미반영**한 것이 본질.

## 2. 진단 결과 (사전 read-only 분석)

### 2.1 직접 원인

`src/parser/hwp3/mod.rs:1831-1833` (Task #460 commit `ab819fa6`):

```rust
// HWP3 last-line tolerance: 한글97은 마지막 줄이 본문 영역을 약간 넘어도 해당 페이지에 배치한다.
// 1600 HWPUNIT(= 한 빈 줄 높이)만큼 하단 여백을 줄여 이 동작을 근사한다.
margin_bottom: ((doc_info.bottom_margin as u32) * 4).saturating_sub(1600),
```

### 2.2 Binary 검증

HWP3 hex 분석 결과 `bottom_margin = 1063 (HWP3 unit) × 4 = 4252 HU = 15.0mm` (HWPX XML과 일치).
- HWP3 파서: `4252 - 1600 = 2652 HU ≈ 9.4mm` (페이지 수 16, 한컴 정답)
- HWP5/HWPX 파서: `4252 HU = 15.0mm` (페이지 수 17, +1 회귀)

### 2.3 본질

한글97의 "마지막 줄 tolerance"는 **한글97 페이지네이션 동작**이지 포맷 특성이 아님. HWP5/HWPX 파일도 한컴 한글에서 보면 동일 동작을 해야 정합. 현재는 **HWP3 파서에만 적용된 비대칭 보정**이 문제.

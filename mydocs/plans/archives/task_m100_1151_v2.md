# Task #1151 v2 수행계획서 — 셀 영역 floating picture → 글자처럼 취급(inline) 토글 시 LINE_SEG.line_height 갱신

v1 수행계획서: [task_m100_1151.md](task_m100_1151.md) · v1 구현계획서: [task_m100_1151_impl.md](task_m100_1151_impl.md) · v1 PR: [edwardkim/rhwp#1160 (CLOSED)](https://github.com/edwardkim/rhwp/pull/1160) · 한컴 동작 검증: [hancom_picture_tac_toggle.md](../tech/hancom_picture_tac_toggle.md) · 검증 protocol: [hancom_picture_tac_toggle_verification.md](../tech/hancom_picture_tac_toggle_verification.md)

## 1. 배경

Task #1151 v1 (PR #1160) 은 표 셀에 picture 삽입 시 한컴 패턴 **floating sibling (tac=false, wrap=Square)** 으로 배치하는 작업이었다. 사용자 후속 검증에서 다음 시나리오가 한컴과 어긋남이 확인되었다:

> 셀 위 floating picture → 그림 속성 대화상자에서 "글자처럼 취급" 토글 시
> - 한컴: parent paragraph 의 inline 으로 전환되며 그 paragraph 의 LINE_SEG.line_height 가 picture height 로 정확히 갱신됨. picture 가 표 위/아래로 picture 만큼 공간 차지하며 시각적으로 표와 겹치지 않음.
> - rhwp v1: `treat_as_char` 비트만 토글되고 LINE_SEG.line_height 미갱신 → picture 가 표와 같은 위치에 inline 으로 그려져 시각적으로 겹침 (작은 picture). 큰 picture 는 paragraph 의 다른 layout 효과로 표가 밀려나면서 한컴과 시각이 비슷하지만 정합 미달.

### 한컴 동작 확정 (H1, Scenario A~D dump 분석 2026-05-29)

사용자가 [hancom_picture_tac_toggle_verification.md](../tech/hancom_picture_tac_toggle_verification.md) protocol Scenario A~D 를 한컴 2022 에서 직접 수행, `samples/tac-verify/` 에 8개 hwp 공유. rhwp dump 비교 결과 네 시나리오 동일 패턴:

| 시나리오 | picture height | parent ls[0].lh before→after | text_len | paragraph count |
|---------|----------------|---------------------------------|----------|------------------|
| A (1×1, 작은) | 5331 | 1000 → 5331 | 0 (불변) | 2 (불변) |
| B (1×1, 큰) | 16038 | 1000 → 16038 | 0 (불변) | 2 (불변) |
| C (3×3, 중앙 셀) | 4847 | 1000 → 4847 | 0 (불변) | 2 (불변) |
| D (본문 floating) | 19019 | 1000 → 19019 | 0 (불변) | 1 (불변) |

확정된 fix 정책 — 4가지 필드 단순 갱신:

```rust
// 1. picture 자체 속성
pic.common.treat_as_char = true;
pic.common.horz_rel_to = HorzRelTo::Para;
pic.common.vert_rel_to = VertRelTo::Para;
pic.common.horizontal_offset = 0;
pic.common.vertical_offset = 0;
// 2. parent paragraph 의 LINE_SEG[0]
seg.line_height = seg.line_height.max(picture_height_hu);
seg.text_height = seg.text_height.max(picture_height_hu);
seg.baseline = (seg.line_height as f64 * 0.85) as i32;  // 한컴 비율
// 3. paragraph.text / char_offsets / char_shapes — 변경 없음
// 4. picture control 의 paragraph 위치 — 변경 없음
```

`baseline ≈ lh × 0.85` 는 한컴 산출물 네 시나리오 모두에서 관찰된 비율 (4531/5331, 13632/16038, 4120/4847, 16166/19019 모두 0.850).

## 2. 목표

`set_picture_properties_native` 가 picture 의 `treat_as_char` 를 false→true 로 변경할 때 한컴 정합 4가지 필드 갱신을 수행한다.

### 성공 기준

- 셀 영역 floating picture → 그림 속성 → "글자처럼 취급" 체크 → picture 위치 그대로, treat_as_char=true, h/v_rel_to=Para, h/v_offset=0, parent paragraph 의 line_segs[0].line_height = picture height.
- 시각적으로 picture 가 parent paragraph 의 inline 글리프로 그려져 표와 겹치지 않음 (Scenario A 등가 케이스).
- 본문 floating picture (셀 외부) 의 tac 토글도 동일 path 가 동작 (Scenario D 등가).
- 본문 inline picture 신규 삽입 (`insert_picture_native` 본문 분기) 회귀 없음. v1 의 셀 안 picture 삽입 path 도 회귀 없음.
- `cargo test --lib` 전수 통과 + `cargo clippy --lib -- -D warnings` + `cargo fmt --all -- --check`.
- 한컴 산출물 (`samples/tac-verify/scenario-*-after.hwp`) 과 dump diff 일치.

## 6. 위험 요소

- **Risk (낮음)**: B 시나리오의 LINE_SEG.cs/sw 자동 재계산 가정. paginate 후 확인. 어긋나면 helper 에 명시 갱신 추가.
- **Risk (낮음)**: baseline 비율 0.85 가 한컴의 모든 화면 / 폰트 환경에서 동일한지 미확정. 본 분석의 네 시나리오는 동일 비율. 다르면 단위 테스트의 단언 완화 (>= 0.84 등).
- **Risk (낮음)**: `apply_picture_props_inner` 의 다른 속성 갱신이 migration 직전에 picture 의 상태를 의도치 않게 바꿀 가능성. snapshot 기반 was_tac 비교로 차단.

## 2. 배경

미지원 파일을 열 때 현재 오류 메시지가 저수준 파서 실패로 노출되면 사용자는
파일이 깨졌는지, 아직 지원하지 않는 포맷인지 구분하기 어렵다.

이슈의 대표 사례는 HWPML 2.1 등 legacy HWPML 계열이다. rhwp의 정식 지원
입력은 현재 HWP 5.0 CFB, HWPX ZIP/XML, HWP 3.0 일부 파서 경로이며,
legacy raw XML HWPML은 별도 감지와 명확한 거부 경로가 필요하다.

## 3. 현황 조사

| 파일 | 확인 내용 |
|------|-----------|
| `src/parser/mod.rs` | `FileFormat::{Hwp,Hwpx,Hwp3,Unknown}`와 `detect_format()` 존재 |
| `src/parser/mod.rs` | `ParseError::UnsupportedFormat { format, hint }`는 있으나 `Unknown` 및 legacy HWPML 전용 분기 없음 |
| `src/parser/mod.rs` | `parse_document()`는 `Hwpx`/`Hwp3` 외 나머지를 HWP 파서로 보내므로 `Unknown`도 CFB 오류로 흐를 수 있음 |
| `src/error.rs` | `ParseError`는 `Display` 문자열로 `HwpError::InvalidFile`에 전달됨 |
| `src/wasm_api.rs` | WASM 생성자 실패는 `JsValue::from_str(err.to_string())`로 JS에 전달 |
| `rhwp-studio/src/main.ts` | `showLoadError()`가 WASM 오류 문자열을 토스트와 상태 표시줄에 그대로 표시 |
| `rhwp-studio/src/core/wasm-bridge.ts` | `loadDocument()`가 기존 문서를 `free()`한 뒤 새 문서 생성 실패 시 내부 `doc` 참조를 명시적으로 초기화하지 않음 |
| `mydocs/plans/task_m100_265.md` | HWP 3.0 감지와 친절한 오류 메시지 선행 작업. 단 현재 HWP3는 파서 경로가 추가되어 #1053과 직접 동일하지 않음 |

## 4. 문제 정의

현재 구조의 문제는 두 가지다.

1. legacy HWPML 계열을 포맷 감지 단계에서 구분하지 못한다.
2. 알 수 없는 파일이 HWP 파서로 진입하면 "지원하지 않는 포맷"이 아니라 CFB
   magic 오류처럼 보일 수 있다.
3. 미지원 문서 로드 실패 후 내부 WASM 문서 참조가 정리되지 않으면 다음 정상 문서
   로드가 깨진 상태에서 시작될 수 있다.

따라서 #1053은 파싱 본체 구현이 아니라, 입력 포맷 식별과 사용자/호출자에게
전달되는 오류 의미를 안정화하는 작업이다.

## 5. 목표

- HWPML 2.1 등 legacy raw HWPML 파일을 지원 대상이 아닌 포맷으로 명확히 감지한다.
- 감지된 미지원 포맷은 안정적인 오류코드와 한국어 설명을 반환한다.
- 알 수 없는 포맷도 저수준 CFB 오류 대신 일반 미지원 포맷 오류로 반환한다.
- 기존 HWP 5.0, HWPX, HWP 3.0 지원 경로는 변경하지 않는다.
- rhwp-studio에서는 기존 오류 표시 흐름을 유지하되, 사용자가 원인을 이해할 수 있는 메시지가 보이게 한다.
- 미지원 문서 로딩 실패 후에도 다음 정상 문서가 문제 없이 열리도록 `WasmBridge`
  문서 상태를 초기화한다.

## 6. 범위

### 포함

- `src/parser/mod.rs`
  - legacy HWPML 감지 로직 추가
  - `FileFormat` 또는 보조 감지 결과 확장
  - `parse_document()`의 미지원 포맷 분기 정정
  - 단위 테스트 추가
- `src/error.rs`
  - `UnsupportedFormat`의 사용자 표시 문자열과 Debug 누출 방지 테스트 보강
- 필요 시 `src/document_core/commands/document.rs`
  - `ParseError` → `HwpError` 변환 경로가 오류코드/문구를 보존하는지 확인
- 필요 시 `rhwp-studio` E2E
  - 미지원 파일 로드 시 기존 사용자 알림 UI 표시 검증
  - 이후 정상 문서 로드가 가능한지 검증

### 제외

- HWPML 2.1 파서 구현
- legacy HWPML을 HWPX/HWP로 변환하는 기능
- HWP 3.0 파서 회귀 정정
- 저장/렌더링 contract 변경

## 7. 오류코드 후보

구현 단계에서 확정할 안정 코드 후보:

| 코드 | 의미 |
|------|------|
| `UNSUPPORTED_HWPML` | raw HWPML 또는 legacy HWPML 계열 |
| `UNSUPPORTED_FILE_FORMAT` | magic/signature로 지원 포맷을 식별할 수 없는 파일 |

사용자 표시 문구는 예를 들어 다음 형태를 목표로 한다.

```text
지원하지 않는 포맷입니다: HWPML 2.1. 오류코드: UNSUPPORTED_HWPML. 현재 rhwp는 HWP 5.0, HWPX, 일부 HWP 3.0 문서만 지원합니다.
```

## 8. 검증 기준

1. 합성 HWPML 2.1 XML 바이트가 legacy HWPML로 감지된다.
2. `parse_document()`가 해당 입력에 대해 `UnsupportedFormat` 계열 오류를 반환한다.
3. 오류 문자열에 포맷명, 오류코드, 지원 포맷 안내가 포함된다.
4. 임의 unknown 바이트는 CFB 오류가 아니라 generic 미지원 포맷 오류로 반환된다.
5. 기존 HWP/HWPX/HWP3 감지 테스트와 실제 샘플 테스트가 유지된다.
6. 기본 Rust 테스트와 필요한 rhwp-studio 빌드/E2E가 통과한다.
7. rhwp-studio에서 미지원 파일 로드 실패 뒤 기존 토스트/상태 표시줄 알림이 표시되고,
   이어서 정상 HWP/HWPX 문서 로드가 성공한다.

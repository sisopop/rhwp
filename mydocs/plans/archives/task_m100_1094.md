# Task M100-1094 수행 계획서

## 1. 작업 개요

GitHub Issue #1094:

```text
[hwpx2hwp] aift 저장 HWP의 한컴 표 높이/페이지 배치 차이
```

#1092에서 `samples/hwpx/aift.hwpx`의 메모 컨트롤 HWP 저장 문제는 해결되었다. 하지만 생성 HWP를
한컴 에디터에서 열면 2페이지 표의 행/셀 높이가 정답지보다 크게 산정되어, 한 페이지에 들어가야 할
표가 다음 페이지로 넘어간다.

이번 작업은 #1092의 메모 저장 성공을 유지하면서, 한컴 에디터 기준 표 높이/페이지 배치 차이를
HWP5 table record contract 관점에서 해결한다.

## 3. 선행 관찰

Stage 4 trace에서 TABLE payload 비교 결과, 표 높이/페이지 배치에 관련된 후보 차이는
`table_attr` 상위 비트로 좁혀졌다.

Section 0:

| table | oracle | generated |
|---|---:|---:|
| 1x1 안내 표 | `0x04000006` | `0x00000006` |
| 35x27 본문 표 | `0x0600000e` | `0x0000000e` |

Section 1:

| table | oracle | generated |
|---|---:|---:|
| 14x17 표 | `0x0400000e` | `0x0000000e` |
| 10x6 표 | `0x06000004` | `0x00000004` |

공통점:

```text
1. record size 동일
2. rows/cols 동일
3. cell spacing 동일
4. 안쪽 여백 동일
5. row_count_hint/col_count_hint 동일
6. tail_after_0x16 동일
7. 차이는 table_attr 상위 비트뿐
```

현재 HWPX 저장기는 `pageBreak`, `repeatHeader` 같은 하위 비트만 재구성하고, 한컴 정답지에서
관찰되는 상위 비트 `0x04000000`, `0x06000000`을 생성하지 않는다.

## 4. 핵심 가설

이번 작업의 우선 가설은 다음이다.

```text
HWPX -> HWP 저장 시 TABLE attr 상위 비트 contract가 누락되어
한컴 에디터의 행/셀 높이 및 페이지 배치 산정이 정답지와 달라진다.
```

따라서 먼저 표 높이 자체를 사후 보정하지 않는다. 정답지와 생성본에서 유일하게 다르게 나타난
TABLE attr 상위 비트를 작은 후보로 검증한다.

## 7. Stage 3 계획 — 최소 구현

Stage 2에서 성공 후보가 확인되면, 해당 규칙을 실제 HWPX -> HWP adapter 직렬화 경로에 반영한다.

예상 수정 파일:

```text
src/model/table.rs
src/parser/hwpx/section.rs
src/serializer/control.rs
tests/hwpx_to_hwp_adapter.rs 또는 신규 회귀 테스트
```

구현 원칙:

```text
1. raw HWP roundtrip 경로의 raw_table_record_attr는 그대로 보존한다.
2. HWPX 출처 표에만 필요한 상위 비트 materialization을 적용한다.
3. Stage 2에서 판정된 최소 규칙만 반영한다.
4. #1092 메모 control contract를 건드리지 않는다.
```

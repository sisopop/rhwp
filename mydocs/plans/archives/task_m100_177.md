## 1. 배경 — 본 타스크의 재정립

### 이슈 원본 가정 (부정확함 판명)

이슈 #177 본문은 다음과 같이 기술됨:

> **증상**: HWPX로 저장 후 한컴에서 열면 줄 넘김이 되지 않고 같은 줄에 텍스트가 겹쳐 조판됨. 한컴이 line_seg를 재계산하지 않고 저장된 값을 그대로 사용.
> **원인**: section.rs에서 line_seg를 정적 `VERT_STEP=1600` 누적으로 생성.
> **수정 방향**: 문단 텍스트 폭을 계산하여 줄 바꿈 위치를 결정. 또는 IR의 기존 `line_segs` 데이터를 직렬화에 활용.

### 실측 관찰 (2026-04-18)

작업지시자의 웹에디터 검증 과정에서 다음이 확인됨:

```
한컴 HWPX ─▶ rhwp로 열기 ─▶ (편집 없음) ─▶ rhwp로 저장 ─▶ 한컴으로 열기: ✅ 줄넘김 정상
                                                        └▶ rhwp로 재로드: ❌ 겹침
```

**결론**: 한컴은 자기가 받아들인 lineseg 값을 **조용히 reflow 보정**하여 정상 렌더. rhwp 는 저장된 값을 명세대로 신뢰하여 렌더하기 때문에 겹침이 보임.

### 기업 윤리·엔지니어링 원칙에 따른 방향 재정립 (작업지시자 확정)

한컴의 "조용한 방어"는 기술부채 은폐에 해당. rhwp는 이를 받아들이지 않음:

1. **표준 준수 입력은 정확히 렌더** — 명세를 정답으로 취급
2. **비표준 입력은 감지하고 사용자에게 고지** — 경고·로그·validation 리포트
3. **자동 보정은 사용자 명시적 선택 후에만** — "이 파일은 비표준입니다. 보정하시겠습니까?" 같은 명시적 UI
4. **rhwp 자신도 비표준을 새로 생산하지 않음** — 저장 시 정확한 값 또는 원본 보존

이 원칙은 Discussion #188 로 커뮤니티에 공개됨.

## 2. 목표

HWPX `<hp:lineseg>` 의 **비표준 값을 rhwp가 감지하고 사용자에게 고지**하며, 사용자 동의 시에만 reflow 보정을 적용한다. rhwp serializer는 **비표준을 새로 생산하지 않는다**.

### 성공 기준

1. `Paragraph` IR에 `lineseg_validation` 경고가 기록됨
2. Document 로드 직후 `validation_warnings` 를 조회할 수 있는 API 제공
3. rhwp-studio 에서 비표준 감지 시 **UI 알림** 노출
4. `reflow_lineseg()` API 호출 시에만 reflow 적용 (기본값은 비활성)
5. rhwp serializer 는 `Paragraph.line_segs` 에 담긴 원본 값을 **그대로 보존**하여 출력 (새 부정확 값 생성 금지)
6. 기존 850개 단위 테스트 유지 + 신규 경고 검증 테스트 추가
7. 대형 실문서 4건에서 false positive 가 없거나 납득 가능한 수준

## 3. 전략 — 3층 구조

### 층 1: 감지 (Validation)

**위치**: `src/parser/hwpx/` 파싱 완료 직후, 또는 `document_core/commands/document.rs::from_bytes` 에서 호출하는 validation pass

**검증 규칙** (초기 안, 실험으로 확정):
- lineseg 의 `vertsize=0` AND `textheight=0` → 재계산 필요 (기존 조건과 동일)
- lineseg 배열이 비어있지 않은데 **첫 lineseg 의 `horzsize` 가 기본값(42520) 고정** → 의심
- lineseg 개수가 문단 내 `\n` 개수+1 과 다른 경우 → 불일치
- lineseg 간 `textpos` 증가량이 UTF-16 단위와 어긋남 → 의심

**결과**:
- `Paragraph::lineseg_warnings: Vec<LinesegWarning>` 에 기록
- `Document::collect_validation_warnings()` 로 전체 조회 API 제공

### 층 2: 고지 (User Notification)

**위치**: wasm_api + rhwp-studio UI

- `wasm_api`: `get_validation_warnings() -> JSON` 추가
- studio: 문서 로드 직후 경고 개수가 있으면 상태바/알림으로 표시
- 개발자용: 콘솔 로그에 상세 내용 출력

### 층 3: 보정 (Reflow, 사용자 동의 후)

**위치**: `document_core/commands/document.rs::reflow_lineseg_on_demand`

- 기본값: **자동 reflow 안함** (기존 `reflow_zero_height_paragraphs` 의 조건을 "사용자 요청 시" 로 축소)
- 사용자가 UI에서 "자동 보정" 버튼 클릭 시 → WASM API 호출 → reflow 수행 → dirty flag 설정
- 저장 시 reflow 결과가 반영됨

### Serializer 동작 규칙

- **원칙**: `Paragraph.line_segs` 가 비어있지 않으면 **그대로 직렬화**. 현재 `section.rs` 의 `push_lineseg` 가 `VERT_STEP=1600` 정적 누적으로 덮어쓰는 로직을 **제거**
- **예외 (불가피한 경우)**: IR에 line_segs 가 아예 없는 경우에만 최소 뼈대 출력 (`<hp:linesegarray/>` 빈 요소). 새 값 생성 금지
- **결과**: rhwp 저장본은 **원본 한컴 파일의 lineseg 를 보존** → 원본이 비표준이면 rhwp도 비표준이 보존되지만 **그것이 rhwp의 탓이 아님**. rhwp는 감지·고지로 사용자에게 알림

## 4. 범위

### 포함

- `src/model/paragraph.rs` 에 `lineseg_warnings` 필드 추가 (또는 별도 Validation 구조체)
- `src/parser/hwpx/section.rs` 파싱 시 검증 + 경고 기록
- `src/document_core/commands/document.rs` 의 `reflow_zero_height_paragraphs` 동작 조정 (자동 → on-demand)
- `src/serializer/hwpx/section.rs` 의 `push_lineseg` 로직 교체 (IR 원본 보존)
- `src/wasm_api.rs` 에 validation warnings 조회 API + reflow_on_demand API
- 통합 테스트: rhwp 저장 → rhwp 재로드 시 lineseg 보존 확인
- 경고 감지 단위 테스트

### 범위 외 (별도 이슈)

- UI 알림의 시각적 디자인·문구 확정 → rhwp-studio UX 이슈로 분리
- `rhwp validate` CLI 에 lineseg 검증 통합 → #185 범위
- 정확한 lineseg 계산 (textRun 폭 측정) → 별도 이슈 (현재는 원본 보존으로 우회)

## 5. 구현 단계 (초안, 3~4단계)

### Stage 1 — 감지 인프라 (Validation struct + Paragraph 필드)

- `src/model/paragraph.rs` 에 `LinesegWarning` enum 및 `lineseg_warnings: Vec<LinesegWarning>` 필드 추가
- `src/parser/hwpx/section.rs` 파싱 후 검증 규칙 적용
- 단위 테스트: 알려진 비표준 샘플에서 경고 검출

### Stage 2 — Serializer 원본 보존

- `src/serializer/hwpx/section.rs` 의 `push_lineseg` 교체
- IR의 `Paragraph.line_segs` 가 있으면 그대로 출력, 없으면 빈 array
- 기존 단위 테스트(`linesegs_emitted_per_linebreak` 등) 재설계 또는 이월 처리
- 통합 테스트: rhwp 라운드트립 시 lineseg 보존 확인

### Stage 3 — Reflow on-demand + WASM API

- `document_core` 에 `reflow_linesegs_on_demand(mode)` 메서드 추가
- `reflow_zero_height_paragraphs` 의 자동 호출 조건 축소 또는 제거 (기존 호환 경로 유지는 검토)
- WASM API: `getValidationWarnings()`, `reflowLinesegs()` 추가

### Stage 4 — 통합 검증 + 문서화

- `Paragraph.line_segs` 원본 보존을 확인하는 라운드트립 테스트
- `hwpx-02.hwpx` (작업지시자 제공) 기반 회귀 테스트
- 대형 실문서 4건에서 false positive 측정
- 최종 보고서 + 이슈 close

## 6. 위험 요소

| 위험 | 완화 |
|---|---|
| 기존 `reflow_zero_height_paragraphs` 자동 호출에 의존하는 테스트·기능이 깨짐 | Stage 3 에서 기존 경로를 유지하되 "설명 가능한" 보정으로 변경. 경고는 항상 기록 |
| 비표준 감지 규칙의 false positive | 초기 규칙은 보수적으로. 실문서에서 검증 후 점진적 강화 |
| Serializer 원본 보존으로 편집 시나리오에서 lineseg 가 stale 해짐 | 편집 시 IR 의 `Paragraph.line_segs` 를 무효화하는 기존 경로가 있으면 활용. 없으면 별도 이슈로 |
| UI 알림 디자인 확정 지연 | Stage 4 에 UI 작업은 최소 (콘솔 로그·API만), 본격 UI 는 후속 이슈 |

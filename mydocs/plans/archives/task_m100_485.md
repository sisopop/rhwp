## 2. 본질 추정

### 2.1 typeset 의 한도 산정 (`engine.rs:1697~1771`)

분할 행의 `split_end_limit = avail_content` 산정은 4곳에서 동일 패턴:

```rust
let padding = mt.max_padding_for_row(r);
let avail_content = (avail_for_rows - padding).max(0.0);
// (또는 (remaining_avail - row_cs - padding).max(0.0))
split_end_limit = avail_content;
```

이 값이 PartialTable.split_end_content_limit 으로 전달되어 layout 의 `compute_cell_line_ranges` 에서 `line_end_pos > abs_limit` 조건으로 줄 단위 break 를 결정한다.

### 2.2 layout 의 누적과 어긋남

`compute_cell_line_ranges` (table_layout.rs:2282~2316) 의 `cum` 누적은 line_height + line_spacing 을 픽셀 단위로 더하며, 마지막 줄 (`is_cell_last_line`) 만 line_spacing 제외 — typeset 의 `avail_content` 추정 (height_measurer 기반) 과 별도로 계산된다. 두 측정의 미세 차이로 마지막 줄이 typeset 추정 영역을 살짝 초과하면서도 layout 은 그 줄을 그린다 (`>` 조건이 `>=` 가 아니므로 경계와 정확히 일치하면 통과).

### 2.3 `>` vs `>=` 경계 처리

```rust
if has_limit && line_end_pos > abs_limit { break; }
```

`line_end_pos == abs_limit` 인 경우 통과되어 마지막 줄이 본문 경계까지 닿게 그려진다. descender (글자 하단) 가 line_h 안에서 baseline 아래로 약 0.2~0.25 line_h 차지하므로, line_end_pos 가 본문 경계와 정확히 일치하면 글자가 본문 외부로 살짝 빠진다.

### 2.4 후보 본질

| 후보 | 위치 | 설명 |
|------|------|------|
| A | engine.rs:1697~1771 (4곳) | typeset 의 `split_end_limit` 산정에 epsilon 마진(예: 1px) 차감 |
| B | table_layout.rs:2304 | layout 의 `>` 를 `>=` 또는 `>= abs_limit - eps` 로 변경 |
| C | engine.rs:1796 | `split_row_h = split_end_limit + padding` 산정 시 line_h 정합 검증 |
| D | layout.rs (vpos correction) | 마지막 줄 본문 침범 시 drop |

(Task #431 본문은 후보 1·2·3 을 제안. Task #362 회귀 영역은 후보 1 의 `avail_content` 추정.)

## 4. 위험 요소

### 4.1 layout drift 본질의 깊이 (`typeset_layout_drift_analysis.md`)

이슈 본문이 가리키는 두 트러블슈팅 문서:

- `mydocs/troubleshootings/typeset_layout_drift_analysis.md` — Task #331 (layout drift 본질) revert 회고
- `mydocs/troubleshootings/typeset_fit_accumulation_drift.md` — fit 누적 drift

선행 분석에 따르면 typeset/layout 의 height 측정 모델 차이 자체가 본질이며, 단순 epsilon 마진은 임시방편이고 근본은 `height_for_fit` 모델 통일이다. **본 이슈는 그 통일 작업의 전체 범위가 아니라, RowBreak 분할 표의 마지막 줄 케이스에 한정한 정합 정정**이다.

### 4.2 epsilon 값의 임의성

- 너무 작으면 (예: 0.5px) 부동소수점 오차로 회귀
- 너무 크면 (예: 5px) 페이지당 1줄 더 손실 → PDF 적재량과 어긋남
- line_h 의 함수로 정의 (예: `0.05 * line_h`) 가 안전

### 4.3 4곳 중복

`split_end_limit = avail_content` 패턴이 4곳에 중복 — 헬퍼로 통일하는 것이 회귀 방지에 유리하나, 본 이슈 범위에서 리팩터링까지 진행할지는 구현 계획 단계에서 결정.

## 5. 정정 방향 (제안 — 구현 계획에서 확정)

이슈 본문의 3가지 제안을 구현 가능 순서로 재배열:

1. **(우선)** `compute_cell_line_ranges` (table_layout.rs:2304) 의 break 조건에 epsilon 마진 추가 — 가장 좁은 영역 정정. layout 단계의 `>` 를 `>= abs_limit - eps` 로 변경하여 경계 줄을 다음 페이지로 밀어냄. epsilon = `line_h * 0.05` 또는 고정 0.5~1.0px 후보.
2. **(차순위)** `engine.rs` 의 `split_end_limit = avail_content` 4곳에 epsilon 차감 — typeset 단계에서 한도를 살짝 좁힘.
3. **(보완)** layout 의 vpos correction 단계에서 본문 영역 초과 시 드롭 (방어 코드).

후보 1 만으로 본 결함 해소 + 회귀 없음 이면 그것으로 종결. 회귀 발생 시 후보 2·3 추가.

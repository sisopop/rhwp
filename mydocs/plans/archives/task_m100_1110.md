# Task M100-1110 수행 계획서

## 1. 작업 개요

GitHub Issue #1110:

```text
[hwpx2hwp] exam_social.hwpx HWP 저장 파일손상 수정
```

`samples/hwpx/exam_social.hwpx`는 #1100, #1108에서 HWPX 렌더링 문제가 해결되었다. 하지만
rhwp-studio에서 HWP로 저장한 `saved/111exam_social.hwp`는 한컴 에디터에서 파일손상 판정을 받는다.

이번 이슈는 렌더링 문제가 아니라 HWPX to HWP 저장 경로의 HWP5 record contract 문제로 본다.

## 2. 대상 파일

```text
source HWPX:
  samples/hwpx/exam_social.hwpx

oracle HWP:
  samples/exam_social.hwp

current generated HWP:
  saved/111exam_social.hwp
```

## 3. 목표

```text
1. 한컴 에디터 파일손상 판정이 발생하는 record/control 구간을 특정한다.
2. 정답 HWP와 현재 저장 HWP의 HWP5 record inventory를 비교한다.
3. HWPX source control과 generated HWP record의 매핑 누락 또는 payload 차이를 찾는다.
4. 누락된 HWP5 저장 contract를 구현한다.
5. 한컴 에디터에서 파일손상 없이 열리는 HWP 저장 결과를 만든다.
```

## 4. 범위

이번 이슈의 범위:

```text
HWPX -> Document IR -> HWP5 serialization
```

이번 이슈의 비범위:

```text
HWPX SVG 렌더링
웹 캔바스 렌더링
HWP 원본 렌더링
문단번호 렌더링 문제 (#1108에서 해결)
```

## 5. 접근 원칙

이 이슈에서는 blind probe를 먼저 만들지 않는다. 또한 #1108 이전에 저장된 `saved/111exam_social.hwp`를
곧바로 재판정 대상으로 삼지 않는다.

#1108에서 이미 HWPX 렌더링 경로의 미구현 항목을 확인했다.

```text
1. hh:paraHead 본문 텍스트 번호 형식
2. paraHead start / numFormat / charPrIDRef / widthAdjust / textOffset
3. paraPr@condense
4. paraPr@fontLineHeight
```

따라서 먼저 위 항목들이 HWP 저장기에서 HWP5 record로 올바르게 serialize되는지 확인하고,
누락되어 있으면 저장쪽 구현을 보강한다. 그 다음 새 HWP를 생성해 한컴 파일손상 판정을 확인한다.

정답지와 현재 저장본의 구조 차이 추적은, 저장쪽 반영 후에도 파일손상이 남을 때 진행한다.

우선순위:

```text
1. #1108에서 구현된 HWPX 문단/번호 속성이 HWP 저장기까지 전달되는지 확인
2. HWP 저장기에서 해당 속성의 HWP5 serialization 누락 보강
3. 현재 코드로 fresh HWP 저장 파일 생성
4. 한컴 에디터 파일손상 재판정
5. 파일손상이 남으면 record inventory / boundary trace / contract diff 진행
```

## 6. Stage 1 계획 — #1108 저장쪽 반영 확인

출력 위치:

```text
output/poc/hwpx2hwp/task1110/stage1_save_side_gap_apply/
```

작업:

```text
1. #1108에서 보강한 HWPX IR 값이 저장기 입력 Document에 존재하는지 확인한다.
2. HWP 저장기에서 다음 값들이 HWP5 record에 반영되는지 확인한다.
   - Numbering level_formats
   - Numbering level_start_numbers / start_number
   - NumberingHead numFormat / charShape / widthAdjust / textOffset
   - ParaShape attr1 condense bits 9..15
   - ParaShape attr1 fontLineHeight bit 22
3. 누락된 저장쪽 serialization을 구현한다.
4. fresh HWP를 생성한다.
5. 생성 파일의 rhwp reload와 HWP5 inventory를 기록한다.
6. 작업지시자에게 한컴 에디터 판정을 요청한다.
```

산출물:

```text
save_side_gap_findings.md
fresh_generation.md
fresh_inventory.md
fresh_vs_oracle_inventory_diff.md
판정 대상 HWP 파일
```

## 7. Stage 2 계획 — 파일손상 boundary trace

Stage 1에서 저장쪽 미구현을 반영한 fresh HWP가 여전히 파일손상일 때만 진행한다.

작업:

```text
1. oracle/fresh HWP의 inventory를 생성한다.
2. inventory diff를 report별로 생성한다.
   - diff
   - hints
   - bundles
   - missing/docinfo/control focus
3. 한컴이 중단하는 지점이 있다면 작업지시자 판정과 inventory 위치를 연결한다.
4. source HWPX XML에서 해당 구간의 control tree를 추적한다.
5. 특정된 record/control 축만 대상으로 최소 수정 후보를 만든다.
```

후보 원칙:

```text
1. HWP 정답지의 payload를 그대로 복사하는 방식은 마지막 검증용으로만 사용한다.
2. 실제 구현은 HWPX source 의미 정보 또는 IR에 존재하는 값에서 HWP5 record를 재구성한다.
3. #1100, #1108에서 통과한 HWPX 렌더링은 변경하지 않는다.
4. samples/exam_social.hwp와 saved/111exam_social.hwp 비교 결과를 문서화한다.
```

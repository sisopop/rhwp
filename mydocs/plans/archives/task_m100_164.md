### IR 차이점 (`mydocs/tech/hwp_hwpx_ir_differences.md`)

HWPX 직렬화에서 **반드시 고려할 핵심 차이**:

1. `raw_ctrl_data` — HWPX는 빈 Vec. 직렬화 시에도 이 필드는 사용하지 않고 `table.common` 기반으로 쓴다.
2. `table.attr` 비트 연산 금지 → `table.common.text_wrap` 등 필드 직접 사용.
3. `lineSegArray.vpos` — HWPX에서는 **0**으로 출력 (파서도 0 가정). pagination 사전 계산값을 쓰면 안 된다.
4. `<tc hasMargin>` 기본값 `false` (미출력 시 표 기본 패딩 적용).
5. 표 CommonObjAttr는 `<pos>`, `<sz>`, `<outMargin>` 자식 요소로 분해 출력.

### 범위 판단 근거

HWPX 파서 자체가 80% 완성이고 다음 항목은 **파서 미구현**:
- 각주/미주, 머리말/꼬리말, 도형, 필드/하이퍼링크

→ 직렬화도 파서 지원 범위에 맞춰 **표·그림·본문** 중심 MVP로 구성. 파서가 나중에 확장되면 직렬화도 후속 이슈로 확장.

## 구현 계획 (단계별)

### 1단계: 모듈 스켈레톤 + 빈 HWPX 생성

**목표**: `src/serializer/hwpx/` 모듈 신설, 빈 Document를 HWPX로 저장 → 한컴에서 오픈

**산출물:**
- `src/serializer/hwpx/mod.rs` — 엔트리 `pub fn serialize_hwpx(doc: &Document) -> Result<Vec<u8>, SerializeError>`
- `src/serializer/hwpx/writer.rs` — ZIP 컨테이너 (`zip` crate, `BufWriter` 기반)
- `src/serializer/hwpx/content.rs` — `Contents/content.hpf` (OPF manifest + spine)
- `src/serializer/hwpx/header.rs` — 최소 `header.xml` (기본 fonts/charShapes/paraShapes)
- `src/serializer/hwpx/section.rs` — 최소 `section0.xml` (SECTIONDEF + COLUMNDEF + 빈 문단 하나)
- `src/serializer/hwpx/utils.rs` — XML escape, quick-xml 헬퍼
- `src/serializer/mod.rs` — `HwpxSerializer` 구현체 + `DocumentSerializer` 등록
- 단위 테스트: 빈 HWPX 생성 → 재파싱 성공

**검증:** `template/empty.hwp` 파싱 → HwpxSerializer 직렬화 → 결과 파일을 한컴 오피스에서 오픈 확인.

### 2단계: 본문 문단·텍스트 직렬화 (라운드트립)

**목표**: `<p>` 문단, `<run>` 런, charShape/paraShape 참조, `<lineSegArray>`까지 포함한 섹션 XML 완성

**산출물:**
- `section.rs` 확장: 문단 헤더 속성(`paraPrIDRef`, `styleIDRef`, `pageBreak`, `columnBreak`), `<run>` + charPrIDRef, `<t>` 텍스트 + 제어문자(Tab, LineBreak), `<lineSegArray>` (`vpos=0`, `horzpos=0`, `textwidth=segment_width`, `linespace/textheight/baseline`)
- `header.rs` 확장: fonts/borderFills/charShapes/paraShapes/styles/numberings/bullets/tabDefs 역매핑
- 제어문자 인코딩: 인라인 1 WCHAR / 확장 8 WCHAR 구분
- 테스트: 텍스트만 있는 `template/empty.hwp` 기반 샘플 라운드트립

**검증:**
- parse_hwpx → HwpxSerializer → parse_hwpx → IR diff 0 (텍스트·CharShape·ParaShape·LineSeg)
- `rhwp ir-diff` 도구로 자동 비교

### 3단계: 표(Table) 직렬화

**목표**: `<tbl>` + `<pos>`/`<sz>`/`<outMargin>`/`<tc hasMargin>` 완전 재구성

**산출물:**
- `section.rs` 확장: `<tbl>` attrs (`rowCnt`, `colCnt`, `textWrap`, `pageBreak`, `cellSpacing`, `borderFillIDRef`), 자식 `<pos>`/`<sz>`/`<outMargin>`/`<rowSz>`, 셀 `<tr><tc>` + `<cellAddr>`/`<cellSpan>`/`<cellMargin hasMargin>`
- IR → XML 역매핑: `table.common.{text_wrap, vert_rel_to, horz_rel_to, vert_offset, horz_offset, width, height, outer_margin}` → XML attrs/elements
- `table.attr` 비트는 **쓰지 않음** (HWPX 판별 규칙 준수)
- 셀 내부 문단은 2단계 로직 재사용

**검증:**
- HWPX 표 샘플 라운드트립 (예: `samples/*.hwpx` 중 표 포함 파일)
- `table.common` 필드 전체가 IR diff 0

### 4단계: 그림(Picture) + BinData 배치

**목표**: `<pic>` 내보내기 + `BinData/image{N}.{ext}` 파일 ZIP에 추가 + content.hpf manifest에 등록

**산출물:**
- `section.rs` 확장: `<pic>` + `<sz>`/`<pos>`/`<outMargin>`/`<img binaryItemIDRef>` + `<effects>` (MVP에서는 파서가 읽는 범위만)
- `writer.rs` 확장: Document.bin_data_content 순회 → ZIP에 `BinData/image{id}.{ext}` 추가
- `content.rs` 확장: BinData 항목을 manifest에 `<opf:item id="image{id}" href="BinData/image{id}.{ext}" media-type="image/{ext}"/>`로 등록
- MIME 매핑: png/jpg/jpeg/gif/bmp/wmf/emf

**검증:**
- 이미지 포함 HWPX 라운드트립, ZIP 엔트리 수·바이트 동일성 확인

### 5단계: 라운드트립 테스트 + CLI 통합 + 최종 보고서

**목표**: 전체 검증 자동화 + CLI `export-hwpx` 또는 `save --format=hwpx` 서브커맨드 제공

**산출물:**
- `tests/hwpx_roundtrip.rs` — `samples/`의 HWPX 파일 전부 순회, 파싱 → 직렬화 → 재파싱 → IR diff 0 검증
- `src/main.rs` CLI 엔트리 확장: `rhwp export-hwpx input.hwpx -o output.hwpx` (또는 HWP→HWPX 는 범위 외로 명확히)
- `mydocs/report/task_m100_164_report.md` — 최종 결과 보고서
- `CHANGELOG.md` 엔트리 추가

**검증:**
- `cargo test` 전체 통과
- 주요 샘플 3~5개 한컴 오피스에서 정상 오픈

## 작업 제약

- HWP↔HWPX 교차 변환은 이번 이슈 범위 외 (별도 이슈).
- HWPX 파서가 지원하지 않는 요소(각주/미주, 머리말/꼬리말, 도형, 필드)는 이번 이슈에서 직렬화하지 않는다. 해당 요소가 IR에 존재하는 경우에도 건너뛰거나 기본값으로 대체하고, 로그로 경고만 남긴다.
- `raw_ctrl_data`/`table.attr` 비트 연산에 의존하지 않는다 (IR 차이 문서 준수).
- 기존 `HwpSerializer` 동작은 변경하지 않는다.
- WASM 빌드 호환 유지 (zip/quick-xml 모두 WASM 대상에서 사용 가능한지 확인).

## 위험 및 완화

| 위험 | 완화 |
|------|------|
| HWPX 네임스페이스/스키마 버전 차이 (KS X 6101:2024 등) | 파서가 참조하는 네임스페이스를 그대로 미러링. 파서 미지원 요소는 직렬화도 생략. |
| `lineSegArray` 일부 HWPX 누락 허용 vs. 한컴 호환 | 최소 하나의 lineSeg 항상 출력 (vpos=0 기본). |
| ZIP 결정적 출력 (mtime 0, 압축 레벨 고정) 필요성 | `zip` crate 옵션으로 고정, 테스트 안정성 확보. |
| WASM 대상에서 `zip` crate deflate 의존성 | `Cargo.toml`에 `default-features = false, features = ["deflate"]` 이미 설정됨 → 문제 없음 (검증 필요). |

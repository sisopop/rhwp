GitHub README는 프로젝트의 현관문이다. 방문자가 3초 안에 "이것이 나에게 필요한가"를 판단한다.

현재 README는 기능 목록 중심이다. 브랜드 정체성이 드러나지 않는다.

| 항목 | 현재 | 개선 방향 |
|------|------|----------|
| 로고 | 없음 | `assets/logo/logo-256.png` 삽입 |
| 슬로건 | "모두의 한글" (단순 표기) | "알(R), 모두의 한글 — 알에서 시작하다" + 영문 태그라인 |
| 스크린샷 | 없음 | 한컴 vs rhwp 비교 렌더링 결과 (3~4장) |

## 2. 신뢰의 기반 — 품질 가시화

오픈소스 프로젝트에서 신뢰는 코드 품질의 투명성에서 온다.

### 2.1 CI/CD 파이프라인

| 단계 | 도구 | 트리거 |
|------|------|--------|
| 빌드 | GitHub Actions | push to devel/main |
| 테스트 | cargo test (783+) | PR / push |
| WASM 빌드 | Docker + wasm-pack | release tag |
| 린트 | clippy + rustfmt | PR |
| 배포 | VS Code Marketplace + Open VSX | release tag |

- 커버리지 뱃지 (codecov 또는 tarpaulin)
- 시각적 회귀 테스트 (SVG 비교)

### 2.3 성능 벤치마크

## 4. 이야기의 힘 — 브랜딩 일관성

브랜드는 로고가 아니다. 사용자가 프로젝트를 떠올릴 때 느끼는 감정이다.

### 4.1 시각적 일관성

| 적용 대상 | 현재 | 개선 |
|----------|------|------|
| GitHub README | 로고 없음 | `assets/logo/` 로고 삽입 |
| VS Code 확장 | icon.png 있음 | `assets/logo/`에서 참조 (완료) |
| rhwp-studio | favicon 추가됨 | 로고 + 브랜드 색상 적용 |
| 데모 페이지 | 미구현 | 브랜드 가이드라인 준수 |
| npm 패키지 | 미배포 | 로고 + README 포함 |

### 4.2 색상 팔레트 적용

브랜딩 전략에서 정의한 색상을 실제 UI에 반영:

- 남색(#1B3A6B): 주 UI 색상, 헤더
- 주황(#E8731A): 강조, CTA 버튼
- 백색(#FFFFFF): 배경
- 금색(#C8A951): 포인트, 아이콘

### 4.3 톤 앤 보이스

모든 사용자 대면 텍스트에 일관된 톤:

- **전문적이되 친근한**: 기술 용어를 피하지 않되, 설명을 곁들인다
- **겸손하되 자신감 있는**: "시도하고 있습니다" 대신 "해결합니다"
- **포용적인**: "개발자를 위한" 대신 "모두를 위한"

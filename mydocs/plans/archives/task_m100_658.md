# Task M100 #658 수행계획서

## 타이틀

rhwp-studio 마우스 드래그 텍스트 선택 하이라이트가 페이지 밖으로 튀고 버벅임

## 배경

Issue #658은 `rhwp-studio` web 편집 화면에서 텍스트를 마우스로 드래그 선택할 때 선택 하이라이트가 실제 텍스트 영역을 벗어나고, 드래그 중 화면이 버벅이는 문제이다.

첨부 영상 기준 관찰된 본질은 다음과 같다.

- 오른쪽 자료 박스 내부 텍스트를 드래그하면 파란 선택 하이라이트가 생성된다.
- 일부 줄의 하이라이트 폭이 박스 경계와 페이지 흰 영역을 넘어 오른쪽 회색 영역까지 확장된다.
- 드래그 중 하이라이트가 안정적으로 유지되지 않고 넓게 튀었다가 사라지는 동작이 반복된다.
- 선택 rect 오류와 DOM 하이라이트 재생성이 결합되어 드래그 UX가 무너지는 것으로 보인다.

## 목표

마우스 드래그 텍스트 선택의 rect 정확도와 드래그 중 렌더링 성능을 개선한다.

완료 기준은 다음과 같다.

- 선택 하이라이트가 페이지 바깥 회색 영역으로 확장되지 않는다.
- 오른쪽 자료 박스 내부 텍스트 선택이 실제 텍스트 줄/컨테이너 범위 안에서 안정적으로 표시된다.
- 긴 다중 줄 드래그 선택 중 눈에 띄는 버벅임이 감소한다.
- 기존 본문/다단/표 셀 선택 동작에 회귀가 없다.

## 의심 지점

### native selection rect 계산

`get_selection_rects_native()`의 폭 계산에서 `lh`와 `rh`가 같은 줄/같은 컨테이너에 속한다는 전제가 깨질 가능성이 있다.

특히 다음 영역을 점검한다.

- `range_end`가 줄 끝 또는 비렌더링 문자일 때 `right_hit`가 다음 줄/다른 컨테이너로 잡히는지
- `selection_continues` 분기에서 `area_right`가 실제 박스/셀/글상자 경계가 아닌 column/page 경계로 사용되는지
- 셀/글상자 내부 선택에서 컨테이너 bbox 기반 클램프가 누락되어 있는지
- 최종 rect가 페이지 영역을 벗어나는 경우 방어 클램프가 없는지

### frontend 선택 렌더링

드래그 중 `input-handler-mouse.ts`가 rAF마다 `updateCaret()`를 호출하고, `updateCaret()`가 매번 `updateSelection()`을 실행한다.

`SelectionRenderer.render()`는 매 호출마다 기존 div를 전부 제거하고 새 div를 생성한다. 긴 다중 줄 선택에서는 잘못된 rect가 들어오는 순간 DOM 생성/삭제 비용까지 커져 영상과 같은 버벅임이 발생할 수 있다.

## 검증 계획

### native/Rust

- `cargo test --lib --release`
- 선택 rect 경계 검증 테스트 추가 후보
  - 같은 문단 다중 줄 선택
  - 표/셀 내부 다중 줄 선택
  - 글상자 또는 박스 내부 다중 줄 선택
  - 다단 문서 선택

검증 조건:

- `x >= 0`
- `width > 0`
- `x + width <= page_width + epsilon`
- 컨테이너 bbox가 확인 가능한 경우 `x..x+width`가 컨테이너 bbox를 초과하지 않음

### rhwp-studio

- `cd rhwp-studio && npm run build`
- 가능하면 e2e 또는 Playwright/Puppeteer 계측 추가
  - 샘플 문서 로드
  - 오른쪽 자료 박스 내부 텍스트 drag
  - `.selection-layer` 하이라이트 div bbox 수집
  - 하이라이트가 페이지/박스 영역 밖으로 나가지 않는지 확인
  - 드래그 종료 후 선택 하이라이트 유지 확인

### 시각 판정

- 작업지시자 첨부 영상과 같은 화면에서 오른쪽 자료 박스 내부 텍스트를 드래그한다.
- 파란 하이라이트가 오른쪽 회색 영역으로 튀지 않는지 확인한다.
- 드래그 중 버벅임이 실사용 수준으로 완화됐는지 확인한다.

## 리스크

- selection rect는 본문/다단/표/글상자/각주 등 여러 편집 컨텍스트와 연결되어 있어 무리한 공통 클램프는 정상 선택 폭을 줄일 수 있다.
- DOM 재사용 최적화만 먼저 적용하면 rect 오버런 본질이 숨겨질 수 있다. native rect 정확도 진단을 우선한다.
- `#595`의 hit-test 정정과 좌표 변환 영역이 인접하므로, 본 이슈 구현 중 머리말/꼬리말 hit-test와 혼동하지 않도록 증상 범위를 분리한다.

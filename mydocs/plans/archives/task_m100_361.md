# Task #361 수행계획서 — TypesetEngine page_num + vpos reset 결함 정정

## 이슈

[#361](https://github.com/edwardkim/rhwp/issues/361) — TypesetEngine 의 페이지 경계 처리 결함:
1. `page_num` 이 section 내 모든 페이지에서 1 (정상은 1, 2, 3, ...)
2. dump-pages 의 vpos / hwp_used 가 페이지마다 reset 안되고 누적

v0.7.3 (Paginator) 에서는 정상. `edddebd Task #313` 에서 TypesetEngine 을 default 로 전환한 시점에 도입.

## 배경

v0.7.6 사이클에서 TypesetEngine 이 main pagination 으로 default 전환됨. fit drift (Task #359 수정) 외에도 페이지 경계 처리에서 회귀 발생. 작업지시자 결정 — **옵션 B**: TypesetEngine 자체의 결함을 직접 수정 (Paginator 환원하지 않음).

### 옵션 B 의 이유
- Task #359 의 fit drift 수정 (Option A) 효과 유지
- TypesetEngine 의 단일 패스 구조 장점 유지
- 향후 Task #345 등 페이지 분할 회귀 정정의 기반

## 증상 정량

### page_num 회귀

`samples/kps-ai.hwp` (section=0, 88 페이지):
- v0.7.3: p1 page_num=1, p2=2, ... p11=8
- main: 모든 페이지 page_num=1

`samples/k-water-rfp.hwp` (section=1, page_num=1 부터):
- v0.7.3: p3=1, p4=2, p5=3, ...
- main: 모두 p3 부터 끝까지 page_num=1

### vpos / hwp_used 누적

`samples/k-water-rfp.hwp` p4:
- 정상: vpos 페이지 시작부터 (≈0~)
- main: vpos=67200~80000 (이전 페이지 누적)
- main: hwp_used=1783.5 (정상 ~897)

## 목표

1. TypesetEngine 의 `page_num` 이 section 내 1, 2, 3, ... 으로 갱신
2. dump-pages 의 vpos / hwp_used 가 페이지마다 reset (페이지 시작 vpos 가 0 으로 표시)
3. v0.7.3 와 동등 또는 그 이상의 페이지네이션 정확도
4. Task #359 의 fit drift 수정 효과 유지 (k-water-rfp LAYOUT_OVERFLOW 0 유지)

## 범위

### 수정 대상 (예상)

- `src/renderer/typeset.rs`:
  - `TypesetState::push_new_page` 또는 `new_page_content` — page_num 증가 로직
  - 페이지 경계 시 vpos 추적 reset 로직 (필요 시)
- `src/main.rs::dump_pages` 또는 hwp_used 계산 함수 — vpos 누적 표시 결함의 origin 확인 후 위치 수정

### 제외

- 외부 API 변경 없음
- Paginator (옛 엔진) 코드 변경 없음
- TypesetEngine 의 fit / 표 / wrap 처리 로직은 유지
- Task #359 의 fit drift 수정 + 단독 항목 페이지 차단 가드 유지

## 접근 원칙

> Paginator 의 page_num 증가 로직 + vpos reset 로직을 참조하여 TypesetEngine 에 동등 동작 구현. 코드 복사가 아닌 같은 시멘틱.

## 단계 (4단계)

### Stage 1 — 결함 origin 정량 분석

1. TypesetEngine 의 page_num 갱신 로직 위치 + 결함 정확히 식별
   - `push_new_page` / `new_page_content` 추적
   - section 단위 page counter 가 어디에 저장되는지
2. Paginator 의 page_num 갱신 로직 비교 (`pagination.rs`)
3. dump-pages 의 vpos / hwp_used 가 누적되는 origin
   - typeset 결과의 PageContent 안에 저장된 vpos 를 누적된 절대값으로 보존하는 위치
   - 페이지 경계에서 reset 해야 하는 책임이 typeset 단계인지 dump 단계인지
4. 결과: `mydocs/working/task_m100_361_stage1.md`

### Stage 2 — 수정 방향 + 영향 분석

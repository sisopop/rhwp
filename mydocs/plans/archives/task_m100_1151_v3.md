# Task #1151 v3 수행계획서 — Paragraph 안의 TopAndBottom 표 + sibling inline picture 시각 layout 정합

v1 수행계획서: [task_m100_1151.md](task_m100_1151.md) · v2 수행계획서: [task_m100_1151_v2.md](task_m100_1151_v2.md) · v3 root cause 분석: [topandbottom_table_inline_picture_layout.md](../tech/topandbottom_table_inline_picture_layout.md)

## 1. 배경

Task #1151 v2 의 model 정합 완료 후 WASM 빌드 + rhwp-studio dev server 에서 사용자 직접 시각 확인 → **rhwp 의 layout 알고리즘이 한컴과 다른 시각 결함** 발견 (2026-05-29):

> 같은 paragraph 의 sibling control 로 `wrap=TopAndBottom` 표 + `tac=true` inline picture 가 있을 때
> - 한컴: 표가 paragraph 위쪽 자리차지 → picture 가 표 **아래** 영역에 inline 으로 그려짐.
> - rhwp: 표와 picture 가 paragraph 의 같은 시작 y (`132.27 px`) 에서 그려져 **오버랩**.

`samples/tac-verify/scenario-a-after.hwp` (한컴 산출물) 을 rhwp 가 직접 열기만 해도 동일 오버랩 → 본 결함은 v2 mutation 잘못이 아니라 **renderer 알고리즘 영역**.

### Root cause (Phase 1 분석)

[mydocs/tech/topandbottom_table_inline_picture_layout.md](../tech/topandbottom_table_inline_picture_layout.md) 의 추적 결과:

- `shape_layout.rs:2733-2738` 의 `calculate_shape_reserved_heights` 가 `vert_rel_to=Para` 인 control 을 명시 제외 → vert_rel_to=Para 인 inline picture 가 shape_reserved 에 반영 안 됨.
- `paragraph_layout.rs:994-1003, 1540-1553` 의 `tac_offsets_px` 는 text flow 기반으로만 picture 자리 결정 → **sibling 표의 vertical 영역을 picture y 에 반영하지 않음**.
- 결과: paragraph 의 inline picture 가 sibling TopAndBottom 표가 차지한 영역을 인식하지 못하고 paragraph 시작점에서 그려짐.

[mydocs/tech/table_layout_rules.md](../tech/table_layout_rules.md) 의 "TopAndBottom + treat_as_char=false" 규칙은 "본문 텍스트를 위/아래로 밀어냄" 이라 명시하지만 paragraph 내 sibling inline picture 의 자리 결정 alg 가 이를 따르지 않음.

## 2. 목표

paragraph 의 sibling control 로 `wrap=TopAndBottom` 표 + `tac=true` inline picture 가 있을 때, rhwp 의 layout 이 한컴 시각과 정합되도록 inline picture 의 y 위치를 sibling 표가 차지한 vertical 영역 만큼 가산한다.

### 성공 기준

- `samples/tac-verify/scenario-a-after.hwp` 를 rhwp 가 직접 렌더링한 SVG 좌표:
  - picture y `≥ table y + table height + outer_margin` (한컴 산출물 비교 기준)
- 사용자가 rhwp-studio dev server 에서 셀 영역 floating picture 를 토글한 시각 결과가 한컴 편집기 출력과 정합:
  - 표가 paragraph 위쪽 자리차지, picture 가 표 아래에 inline 으로 그려짐, 오버랩 없음.
- 4 시나리오 (A/B/C/D) 모두 한컴 산출물과 시각 정합.
- 회귀 0:
  - 본문 inline picture (sibling 표 없음)
  - 본문 TopAndBottom 표 (inline picture 없음)
  - 같은 paragraph 에 wrap=TAC 표 (tac=true) + inline picture
  - 본문 floating picture (wrap=Square 등)
- `cargo test --lib` 전수 통과 + `cargo clippy --lib -- -D warnings` + `cargo fmt --all -- --check`.

## 6. 위험 요소

- **Risk (중)**: tac picture 의 y 위치 가산이 paragraph 의 line_seg vpos 와 이중 계산될 가능성. 이미 paragraph 0.0 의 ls[0].vpos=13064 인 한컴 model 에서 표가 그 vpos 부터 시작하고, picture 는 같은 vpos 의 line 안에 있음. 가산 정책의 정확한 형식 (vpos 보정 vs picture y 추가) 결정 필요.
- **Risk (중)**: sibling 표의 정확한 차지 영역 — `common.height` 만? `outer_margin` 포함? Stage 3 의 첫 작업으로 한컴 산출물 SVG 분석으로 확정.
- **Risk (낮음)**: 본문 inline picture (sibling 표 없음) 케이스에서 helper 호출 → reserved=0 반환 → 기존 동작 보존. 회귀 0 보장.
- **Risk (낮음)**: 셀 내부 paragraph 의 같은 패턴이 동일하게 동작하는지 — Stage 4 시각 검증으로 확인.

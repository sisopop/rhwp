## 배경

aift.hwp의 우리 렌더러 page 18(=PDF 인쇄 page 12) 하단의 `pi=222` 단락
(`- 기능별 에이전트 구현 : 두아즈가 개발하는 도면 파싱·분석 VLM ...`)에서
PDF는 첫 2줄을 page 18에 담지만, 우리 렌더러는 1줄만 담고 line 1~3을 page 19로 밀어버린다.

## 현상

```
=== 페이지 18 (rhwp dump-pages) ===
  body_area: x=75.6 y=75.6 w=642.5 h=971.4
  단 0 (items=15, used=937.8px, hwp_used≈922.4px, diff=+15.4px)
   ...
   PartialParagraph  pi=222  lines=0..1  vpos=67980..69900   ← 우리: 1줄
```

PDF (정답) page 12 하단:
```
⋅ 워크플로 실행 이력 로깅 및 모니터링

- 기능별 에이전트 구현 : 두아즈가 개발하는 도면 파싱·분석 VLM 및 기능 모듈을      ← line 0
  기반으로 주관기관의 에이전트 빌더를 이용한 BOM 자동생성 에이전트와 변경점 탐지 에 ← line 1
```

`diff=+15.4px` — 우리가 page 18 본문에 HWP보다 +15.4px 더 누적해서, pi=222의 line 1
(line slot ≈25.6px)이 본문 하단(971.4px)을 넘어 page 19로 밀린다. 1줄 슬롯이 25.6px이므로
+15.4px만 줄여도 line 1이 안전하게 들어간다.

## 페이지 18 누적 항목

| pi | 종류 | 비고 |
|----|------|------|
| 209~213 | 5줄 텍스트 | 정상 |
| 214 | 표 1×1, 168.0×56.4mm, wrap=TopAndBottom, outer=0.5mm 사방 |
| 215 | 5줄 텍스트 (들여쓰기 단락) |
| **216** | **그림 단락 (tac=true)** | lh=22180 (~295.7px), 이미지 71mm(~268.4px), `cur 169.6×71.0mm` |
| 217~221 | 5줄 텍스트 |
| 222 | 다중줄 텍스트 (1~2줄째가 page 18 끝) |

## 원인 후보 (가설)

### 후보 A: pi=216 그림 라인 높이 (가장 유력)

`pi=216` LINE_SEG는 `lh=22180, th=22180, bl=18853` (HWP unit). 이미지 자체는 71.0mm = 20126 HU.
`lh - 이미지높이 = 22180 - 20126 = 2054 HU ≈ 7.3px`. 이 값이 line 자체의 위/아래 여백에서
계산되는데, 우리 렌더러가 라인 슬롯을 산출할 때 line spacing(160%)나 char_shape height,
혹은 `bl/lh` 비율에서 PDF/한컴 대비 약 +7~15px을 더 잡고 있을 가능성.

### 후보 B: pi=214 표 outer_margin / 표 후 단락 spacing

표 outer = 0.5mm 사방 (141 HU = 1.88px). pi=215가 표 직후 단락이라 표/문단 경계의
라인 슬롯 결정에서 1줄 분량(약 25.6px) 중 일부가 추가 누적되는 경우.

### 후보 C: 페이지 누적 산식 (line spacing 적용 위치)

각 단락 슬롯 = `lh × line_spacing_ratio`인데, pi=216처럼 lh가 큰(>20000) 라인에
160% 적용 시 약 ~+7~10px 차이가 발생할 가능성.

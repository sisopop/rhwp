# Task #83: Safari 확장 프로그램 HWP 뷰어 — 수행계획서 (v2 갱신)

## 목표

Chrome 확장(`rhwp-chrome/`)을 기반으로 Safari Web Extension을 개발하여 macOS/iOS Safari에서 HWP 파일을 열 수 있도록 한다.

## 구현 단계 (실제 진행 반영)

### 1단계: Xcode 빌드 + 기본 동작 확인 ✅

- `safari-web-extension-converter`로 Chrome dist → Xcode 프로젝트 변환
- Safari 전용 dist 분리 (`rhwp-safari/dist/`)
- ES module → 단일 파일 번들링, `service_worker` → `scripts` 변환
- macOS/iOS 빌드 성공, Safari 확장 등록 확인

### 2단계: Safari 전용 코드 재작성 + 보안 + UX ✅

Chrome 코드의 단순 변환이 아닌, Safari Web Extension 표준에 맞춰 전면 재작성.

**2-1. Safari 전용 소스 작성** (`rhwp-safari/src/`)
- `background.js`: `browser.*` API, 단일 파일, `storage.local` 사용
- `content-script.js`: DOM API 전용 (innerHTML 제거), 호버 카드, 토스트 알림
- `manifest.json`: Safari 호환 CSP, `scripts` 형식 background

**2-2. 보안 수정** (Task #84 연동)
- 공통 보안 모듈 (`rhwp-shared/security/`)
- fetch-file: URL 검증, 내부 IP 차단, 매직 넘버 검증, sender 검증
- open-hwp: 3단계 URL 판별, filename 새니타이즈
- XSS: innerHTML 완전 제거 → DOM API
- fingerprinting 완화: 허용 도메인에서만 확장 노출

**2-3. 옵션 페이지** (`options.html` + `options.js`)
- Apple HIG 디자인 (세그먼트 컨트롤 탭, Apple 토글, 다크 모드)
- 사이트 허용 목록, 기능 토글, 보안 설정, 개발자 도구, 보안 로그
- `storage.local` 사용 (Safari `storage.sync` 불안정)

**2-4. UX 개선**
- 차단 시 토스트 알림 + 설정 가이드 안내
- 명시적 행위(배지 클릭) vs 자동 동작(링크 가로채기) 구분
- 한글 인코딩 문제 해결 (reason 코드 방식)

**2-5. 빌드 파이프라인**
- `build.sh`: Chrome 빌드 → Safari dist 생성 → JS 문법 검사 → Xcode 빌드
- `node --check`로 문법 오류 시 빌드 중단

### 3단계: macOS + iOS 빌드 테스트 (예정)

- macOS Safari에서 전체 기능 통합 테스트
- iOS Simulator에서 동작 확인
- 보안 테스트 페이지(`test/06-security.html`) 검증

### 4단계: App Store 배포 준비 (예정)

- Apple Developer Program 계정 확인
- 앱 아이콘, 스크린샷, 설명 준비
- 개인정보 처리방침 URL
- App Store Connect 등록 + 심사 제출

## 교훈 (계획 vs 실전)

| 계획 | 실전 |
|------|------|
| Chrome 코드 변환으로 빠르게 완성 | Safari API 차이로 전면 재작성 필요 |
| 보안은 별도 단계 | 개발 중 보안 감사 → 즉시 반영 |
| 옵션 페이지는 간단히 | Apple HIG + 탭 UI + 다크 모드 |
| 인코딩 문제 없을 것 | background → content-script 한글 깨짐 |
| CSP는 Chrome과 동일 | 인라인 스크립트 차단, connect-src 제한 |
| storage.sync 사용 | Safari에서 불안정 → storage.local 전환 |

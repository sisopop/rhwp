## 1. 배경

Firefox AMO 제출 과정에서 두 종류의 워닝이 보고되었다.

### 1.1 manifest strict_min_version 모순

`rhwp-firefox/manifest.json`의 Gecko 설정은 현재 다음 조합을 사용한다.

```json
"strict_min_version": "112.0",
"data_collection_permissions": {
  "required": ["none"]
}
```

AMO는 `browser_specific_settings.gecko.data_collection_permissions`가 Firefox 140 이상, Android 142 이상에서 지원된다고 판단한다. 따라서 `strict_min_version`을 112로 유지하면 manifest 내부 요구 버전과 실제 지원 필드가 충돌한다.

### 1.2 빌드 결과물 보안 워닝

AMO 정적 분석에서 다음 유형이 보고되었다.

- `wasm/rhwp.js`의 `Function` 생성자: wasm-bindgen 표준 로딩 패턴일 가능성이 높음
- `assets/viewer-*.js`의 `Function`, `innerHTML`, `document.write`: Vite 번들 또는 애플리케이션 코드 기원 분류 필요

이슈의 핵심은 모든 워닝을 무조건 제거하는 것이 아니라, 우리 코드에서 제거 가능한 위험 사용을 수정하고 의존성/툴체인 코드에 대해서는 AMO 검토자에게 명확한 사유를 제공하는 것이다.

## 2. 목표

1. `data_collection_permissions` 유지 조건에서 AMO manifest 워닝을 해소한다.
2. 빌드 결과물 보안 워닝을 우리 코드, 번들러 코드, wasm-bindgen 코드로 분류한다.
3. 우리 코드에서 발생한 `innerHTML`/`document.write`/동적 `Function` 사용이 있으면 DOM API 또는 안전한 대체 구현으로 수정한다.
4. 제거 불가능하거나 표준 툴체인에서 발생하는 워닝은 reviewer note 초안을 작성한다.
5. Firefox 확장 빌드 결과물 기준으로 재검증 가능한 절차를 남긴다.

- AMO reviewer note 문서 작성

### 범위 외

- 폰트 번들링 전략 변경

## 4. 접근 원칙

1. **AMO 요구 우선**: `data_collection_permissions`는 제출 필수 항목으로 보고 제거하지 않는다.
2. **최소 변경**: manifest 버전 상향과 실제 워닝 원인 제거에 집중한다.
3. **소스 기준 분류**: dist만 보고 판단하지 않고 sourcemap/번들 출력/원본 검색으로 기원을 확인한다.
4. **보안 문서 준수**: `innerHTML`은 사용자 입력 또는 외부 데이터 경로에서 금지하고, 필요한 초기화 용도도 가능한 DOM API나 `textContent`로 제한한다.
5. **검토자 설명 가능성**: wasm-bindgen/Vite 표준 코드로 남는 워닝은 위치, 이유, 사용자 데이터와의 관계를 reviewer note에 명시한다.

## 5. 예상 구현 단계

본 수행계획서는 범위와 접근 방향만 확정한다. 세부 파일별 변경은 구현계획서(`task_m100_338_impl.md`)에서 확정한다.

### Stage 1 — AMO 워닝 재현 및 분류

- 현재 `rhwp-firefox/dist`와 새 빌드 결과물에서 워닝 후보 위치 확인
- `manifest.json`의 `strict_min_version`과 `data_collection_permissions` 충돌 정리
- `Function`/`innerHTML`/`document.write` 사용 위치를 소스, 번들러, wasm-bindgen으로 분류

완료 기준: 수정 대상과 reviewer note 대상이 분리된 조사 결과 확보.

### Stage 2 — manifest 및 소스 보안 워닝 수정

- `strict_min_version`을 AMO 요구에 맞춰 상향
- 우리 코드 기원 워닝을 DOM API, `textContent`, 안전한 URL/HTML 처리로 교체
- 필요 시 테스트/샘플 HTML의 배포 제외 여부 확인

완료 기준: 소스 레벨에서 제거 가능한 위험 사용이 사라지고 manifest 충돌이 해소됨.

### Stage 3 — Firefox 확장 빌드 및 dist 갱신

- `rhwp-firefox` 빌드 실행
- `rhwp-firefox/dist` 산출물 갱신
- manifest, 서비스워커, viewer 리소스가 dist에 정상 반영되는지 확인

완료 기준: Firefox 확장 dist 빌드 성공 및 산출물 반영.

### Stage 4 — AMO reviewer note 및 완료 보고

- 남는 워닝별 reviewer note 초안 작성
- 단계 완료 보고서 작성
- 최종 검증 결과와 제출 시 주의사항 정리

완료 기준: 작업지시자가 AMO 재제출에 사용할 수 있는 설명 문서 확보.

# 수행 계획서: Task #288 (#287 Phase 2)

## 타스크

[#288](https://github.com/edwardkim/rhwp/issues/288) — 수식 SVG 레이아웃: 빈 runs 줄의 TAC 수식 x 위치에 탭(\t) 반영

- 마일스톤: v1.0.0 (M100)
- 브랜치: `local/task288` (from `local/task287`)
- 선행: #287 (이미 merge 된 상태에서 시작하거나 stacked PR)

## 배경

#287 에서 빈 runs comp_line 의 TAC 수식 인라인 처리를 추가해 "박스 좌상단 겹침" 문제는 해결됨. 그러나 x 좌표는 여전히 PDF 대비 약 30-50 px 좌측에 치우쳐 있다.

- `samples/exam_math_8.hwp` 큰 수식: 현재 x=**133.27**, PDF x≈**162-182** (탭 pos=13600 HU=181 px 에 해당)
- HWP ParaShape 에 `tab_def_id=6, tab[0] pos=13600 HU` 가 있으나 빈 runs 줄에서는 탭 처리가 돌지 않음

## 목적

1. `samples/exam_math_8.hwp` 큰 수식의 x 가 PDF 와 ±2 px 이내로 일치
2. 같은 패턴을 가진 다른 문서 (`exam_math_008` 등) 도 함께 개선
3. 기존 수식 샘플(`exam_math.hwp`, `equation-lim.hwp`) 회귀 없음

## 조사 필요 사항 (단계 1 에서 확정)

#287 단계 1 은 paragraph_layout 의 값을 측정했으나, #288 은 **한 단계 위 계층(composer)** 의 조사가 필요:

1. **HWP composed text 에 `\t` 가 실제로 존재하는가?**
   - `dump` 에 나타나는 text_len=21 은 visible text 만 셈
   - paragraph.text 원본 및 controls 배열의 "tab" 엔트리를 직접 조사해야 함
2. **`compose_lines` 가 \t 를 어떻게 처리하는가?**
   - `split_by_char_shapes` 에서 탈락? `ComposedLine` 의 line break 처리 (`has_line_break`)와 연관?
3. **ls[1].text_start=41 의 40자 차이는 무엇으로 채워지는가?**
   - HWP 포맷에서 inline control 은 composed text 에서 여러 바이트를 차지 (대체로 8자)
   - "(가) 모든 자연수 n[eq-8byte] 에 대하여\n" 이 총 41 positions

## 접근 방침 (후보, 단계 2 에서 결정)

### (A) 휴리스틱 — paragraph_layout 국소 수정
- 빈 runs TAC 처리 블록에서 `tab_stops` 가 있으면 첫 탭 pos 로 `inline_x` 이동
- 수정 규모: 5-10 줄
- 장점: 범위 좁음
- 단점: 탭 2개 이상, tab 없는 문서, 우측탭/가운데탭 등 복잡 케이스 미커버. `ParaShape.alignment` 중앙 정렬 등도 놓침

### (B) compose 계층 수정 — 근본 수정
- `compose_lines` 에서 ls 앞의 `\t` 제어문자(또는 composed text 의 \t)를 빈 comp_line 의 선행 run(또는 탭 세그먼트)으로 보존
- 이후 기존 run 루프의 탭 처리 로직(`char_x_map`, `find_next_tab_stop`)이 자동으로 돌아감
- 장점: 일반성 확보, 장기적으로 옳은 설계
- 단점: composer 수정 → 기존 수식·표·이미지 등 모든 TAC 경로에 영향 가능, 테스트 스냅샷 회귀 검증 범위 큼

### (C) 하이브리드 — paragraph_layout 의 빈 runs 블록에서 HWP 레벨 tab_stops + ParaShape.align 을 직접 조회
- `tab_stops` 가 있으면 탭 위치, 없으면 `alignment` (Center/Right) 기반 계산
- 장점: composer 수정 없이 대부분 케이스 커버
- 단점: 인라인 커서 방식의 기존 탭 로직과 이원화. 향후 유지보수 시 이중 경로 부담

단계 2 에서 데이터를 보고 결정. 선호 순서: **(B) > (C) > (A)**, 단 (B) 회귀 범위가 크면 (C) 로 물러남.

# Task #1253 작업 계획

## 문제

`3-09월_교육_통합_2022.hwp`에서 미주 모양 설정값이 한컴오피스 2024와 RHWP-studio에서 동일하게 보이는데, 실제 렌더링은 다르게 나온다. 또한 RHWP-studio의 `주석 모양 > 미주 모양` UI가 한컴오피스의 대화상자와 다르게 표시된다.

작업지시자가 제공한 한컴 기준 설정은 다음과 같다.

- 번호 모양: `1,2,3`
- 앞 장식 문자: `문`
- 뒤 장식 문자: `)`
- 구분선 넣기: 켬
- 구분선 종류: 이중 실선 계열
- 구분선 길이: 사용자 `50.0mm`
- 굵기: `0.7mm`
- 색: 초록
- 구분선 위: `0.0mm`
- 미주 사이: `7.0mm`
- 구분선 아래: `2.0mm`
- 번호 매기기: 앞 구역에 이어서
- 미주 위치: 문서의 끝

## 목표

1. RHWP-studio의 미주 모양 UI를 한컴오피스 2024 기준에 가깝게 맞춘다.
2. `미주 사이`와 `구분선 아래`가 내부 저장 슬롯, HWPX 속성, Studio JSON API, 렌더러에서 같은 의미로 전달되는지 확인한다.
3. PR #1232에서 도입한 미주 간격 공통 로직이 동일 설정을 한컴과 다르게 해석하는지 재검토한다.
4. 수정 후 `issue_1139_inline_picture_duplicate` 회귀 테스트와 시각 검증으로 방어한다.

## 선행 사실

- HWP5 `FOOTNOTE_SHAPE` 실제 레코드는 스펙 표의 26바이트보다 2바이트 길다.
- 기존 검증 기준:
  - `note_spacing`: 한컴 UI `구분선 아래`
  - `raw_unknown`: 한컴 UI `미주 사이`
- 기존 테스트 `issue_1139_exam_2022_endnote_shape_matches_hancom_reference`가 이 매핑을 확인한다.
- 기존 API 테스트 `issue_1139_stage31_endnote_shape_api_updates_section_shape`도 Studio JSON의 `separatorMarginBottom`이 내부 `note_spacing`, `noteSpacing`이 내부 `raw_unknown`으로 저장되는 것을 확인한다.

## 위험

- `미주 사이 7mm`는 원본 `LINE_SEG`/VPOS 흐름에 이미 일부 반영되어 있어 무조건 더하면 페이지 수가 증가한다.
- `구분선 아래`는 구분선과 첫 미주 사이의 실제 시각 높이이며, `미주 사이`와 같은 레이어에서 단순 합산하면 중복 계산될 수 있다.
- UI 이름을 내부 필드명과 1:1로 맞추면 오히려 한컴 UI 의미가 깨질 수 있으므로, 내부 저장명과 사용자 표시명을 분리해야 한다.

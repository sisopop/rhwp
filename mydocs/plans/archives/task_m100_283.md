# Task #283 수행계획서 — 수식 괄호 path 폭 조정 (#280 Phase 2)

## 이슈

[#283](https://github.com/edwardkim/rhwp/issues/283) — 수식 SVG 렌더링: 괄호 path 폭이 폰트 글리프 대비 과대 (#280 Phase 2)

## 목표

`left(` `right)` 로 생성되는 확장 괄호의 폭·형상이 본문 글자 폰트 글리프(Times New Roman) 대비 과대·부자연스럽게 보이는 문제 해소. Phase 1(#280) 에서 폰트를 얇은 세리프로 바꾼 직후 드러난 잔여 불균형을 해결한다.

## 배경

### 현재 구현

`EqNode::Paren { left, right, body }` → `LayoutKind::Paren` → `draw_stretch_bracket` 경로:

- 폭 할당: `paren_w = fs * 0.3` (`layout.rs:832`, `svg_render.rs:230,237`, `canvas_render.rs:184,188`)
- 형상: `draw_stretch_bracket` 이 SVG `<path d="M ... Q ... Z">` 또는 Canvas `quadraticCurveTo` 로 직접 그림
- 높이: `lb.height` (내용 layout box 전체 높이, full stretch)

`(` 예시 (`svg_render.rs:277-285`):
```rust
"(" => {
    svg.push_str(&format!(
        "<path d=\"M{:.2},{:.2} Q{:.2},{:.2} {:.2},{:.2}\" ...",
        mid_x + w * 0.2, y,        // 시작
        x, y + h / 2.0,             // 제어점 (박스 좌측 끝)
        mid_x + w * 0.2, y + h,     // 끝
        ...
    ));
}
```

### 관찰된 문제 (Task #280 단계 4)

`samples/equation-lim.hwp` 의 `f left(2+h right)` 렌더링:
- 파렌이 본문 글자 `f`, `h` 대비 **다소 큼** — 파렌만 두드러져 보임
- 한컴 PDF 는 HyhwpEQ 폰트 글리프를 사용해 파렌이 본문과 자연스럽게 어우러짐
- 원인 추정: `fs * 0.3` 폭 할당 + 제어점 `x`(박스 좌측 끝) 위치 → 과장된 곡선

### 단순 `(` vs `left(` 의 차이

- 단순 `(` / `)` 는 `TokenType::LParen` → `EqNode::Symbol("(")` → `LayoutKind::Symbol` → **`<text>(</text>`** (폰트 글리프 사용)
- `left(` / `right)` 는 `EqNode::Paren` → `LayoutKind::Paren` → **`draw_stretch_bracket`** (SVG path 사용)

`samples/exam_math.hwp` p009 의 `P(A|B)` 같은 단순 파렌은 폰트 글리프라 자연스러운데, p013 의 `∫_0^10 (x+2)/(x+1) dx` 에서 단순 파렌이 fraction 내부에 있으면 괜찮고, 스트레치 파렌은 p005 의 `left({최저입찰가격} over ...)` 류에서 눈에 띔.

## 수정 방침 후보

### A안: 상수 튜닝 (최소 변경)

`fs * 0.3` → `fs * 0.25` 정도로 축소. 내부 곡선 제어점은 유지.

- **장점**: 1~2줄 수정. 회귀 리스크 최소.
- **단점**: 스트레치가 필요한 경우(분수 등 큰 파렌) 도 일률적으로 축소되어 부자연스러울 수 있음.

### B안: 높이-조건부 렌더링 (권장)

- 내용 높이 ≤ 임계치(예: `fs * 1.2`) → **폰트 글리프** 렌더링 (`<text>(</text>`)
- 초과 → 기존 path 기반 스트레치 유지

- **장점**: 텍스트 높이 파렌은 폰트와 자연스럽게 어우러짐. 스트레치 품질은 기존 유지.
- **단점**: 렌더링 경로 분기 추가. 회귀 테스트 범위 확대.

### C안: 곡선 제어점 재튜닝

`draw_stretch_bracket` 의 제어점을 `x`(박스 좌측 끝) → `x + w*0.1` (약간 안쪽) 등으로 이동해 곡선 과장 완화.

- **장점**: 스트레치 품질 개선, 폭은 그대로 유지.
- **단점**: 숫자 튜닝 많아져 매직 넘버 증가.

### 복합안

- 1단계: A안 (상수 축소) + C안 (제어점 재튜닝) 일부 적용
- 2단계: B안 적용 (텍스트 높이 파렌을 글리프로 전환)

## 제안 접근

**A + C 최소 튜닝을 먼저 시도** 하고, 시각 검증 결과가 충분치 않으면 B안 추가.

이유:
- 사용자 제보는 `left(2+h right)` 같은 **텍스트 높이** 파렌이 어색하다는 것
- 이 케이스가 가장 많고, 스트레치 파렌(분수·행렬 괄호) 은 변화 후에도 허용 가능한 품질이어야 함
- 단순 상수 1~2개 조정으로 시각 임팩트가 커질 수 있음 — 최소 변경으로 최대 효과 우선

## 조사 항목 (단계 1~2 에서 수행)

1. Times New Roman 의 `(` 글리프 실측 (Chrome headless 로 `<canvas>` measureText 또는 `<text>` getBBox 호출):
   - advance width / em
   - stroke thickness
   - bowl depth
2. 현재 rhwp 렌더 파렌 실측 (크롭 이미지 픽셀 측정):
   - 폭, 높이, 곡률
3. 목표 비율 도출 → 새 `paren_w`, 제어점 위치 결정

## 리스크 및 주의점

- **행렬 괄호 회귀** — `draw_stretch_bracket` 은 `LayoutKind::Matrix` 에서도 호출됨. 샘플 확보 어렵지만 단위 테스트(`test_paren_svg`) 는 통과해야 함.
- **과도 축소** — `paren_w` 를 너무 줄이면 곡선이 너무 납작해져 파렌처럼 안 보일 수 있음. 실측 기반 튜닝 필요.
- **스트레치 파렌 품질 유지** — `left(` `right)` 가 분수·sum 등 큰 구조를 감쌀 때 높이는 보존되어야 함. 높이는 건드리지 않음.
- **Phase 1(#280) 과 같은 브랜치/커밋** — `local/task283` 은 `local/task280` 위에 분기. task280 가 devel 머지 전이지만 task283 도 같은 파일들을 건드리므로 merge 시 conflict 없도록 순서대로 머지.

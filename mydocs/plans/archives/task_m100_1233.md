# 수행계획서 — Task #1233: 수식 큰 연산자(Σ/∏/∫) 피연산자 간격 누락

- **이슈**: #1233 (M100 / v1.0.0)
- **브랜치**: `feature/issue-1233-eq-bigop-spacing` (base: `stream/devel` f83c43b5)
- **성격**: 수식 렌더 레이아웃 버그 수정 (간격), 레이아웃 코어 무관
- **작성일**: 2026-06-02

## 1. 배경과 증상

`samples/3-09월_교육_통합_2023.hwp` 6쪽 문26 "…∑bₙ의 값은?" 등에서 합산기호 **∑의
피연산자(bₙ)가 Σ에 거의 붙어** 렌더된다. 한컴 PDF 정답지는 Σ와 피연산자 사이에 적정
간격(thin/med space)이 있다. (#1224 폰트 조사 중 발견, 폰트 변경과 무관 — 배치는 변경 전후 동일.)

## 2. 원인 (조사 완료)

`src/renderer/equation/layout.rs`:

- AST: "∑ bₙ" = `Row[BigOp{Σ, sub, sup}, Subscript(b,n)]` — **Σ와 bₙ은 형제 노드**.
- `layout_row`(L234-238): 형제를 `x += b.width` 로 **간격 0(커닝 없음)** 배치 → 다음 노드는
  BigOp 박스 우측 끝에 즉시 붙음.
- `layout_big_op`(∑/∏ limits, **L668-671**): `width: max_w`(= max(op_w, sub.w, sup.w)) —
  **연산자 trailing 간격 없음**. `layout_integral`(L728) `width: total_w` 동일.
- 대비: `layout_symbol`(L281-292)은 `+ - =` 등에 `pad`(fs×0.15)를 양쪽 추가하나 **큰
  연산자만 누락**. 파서도 자동 간격 미삽입.

## 3. 수정의 안전성 (조사 확증)

인라인 수식은 `svg.rs`(L443-453)에서 `scale_x = bbox.width / layout_box.width`(= 컨트롤
advance `tac_w` ÷ 자연폭)로 **가로 스케일되어 tac_w 에 맞춰진다.** 따라서 BigOp width 를
키워도:

- 수식 전체가 tac_w 에 맞게 자동 압축 → **다음 글자("의") 침범(오버플로) 없음**.
- 늘어난 trailing 간격이 tac_w 안에서 재분배되어 **Σ-피연산자 간격이 비례적으로 생김**.

즉 수정은 `layout.rs` 내부로 **자기완결적**이며, 문단 advance·메트릭 DB·페이지네이션 무변경.

## 4. 목표

큰 연산자(Σ/∏ limits 및 ∫ 적분)의 레이아웃 폭에 **trailing 수학 간격**을 추가해, 피연산자가
연산자에 붙지 않고 한컴 PDF 와 시각 정합하도록 한다.

## 5. 적용 범위

| 영역 | 변경 |
|------|------|
| `equation/layout.rs::layout_big_op` | limits 스타일 BigOp width 에 trailing pad 추가 |
| `equation/layout.rs::layout_integral` | 적분(nolimits) width 에 trailing pad 추가 (동일 증상 예방) |
| 상수 | `BIG_OP_TRAIL_PAD`(fs 비율) 신규 — `layout_symbol` pad 관례와 정합 |

### 범위 밖

- `layout_row` 의 일반 커닝 도입(광범위 영향) — 본 건은 큰 연산자 trailing 한정.
- 연산자 leading 간격(앞쪽) — 앞 항과의 간격은 현재 문제 없음(별도 판단).
- 메트릭 DB·문단 레이아웃 — 무변경(수식은 tac_w 로 스케일).

## 6. 검증 방법

1. **시각 정합**: `export-svg 3-09월_교육_통합_2023 -p5` → 6쪽 문26 ∑bₙ 간격을 PDF 와 대조.
2. **다(多)수식 표본**: ∑/∏/∫ 포함 샘플 2~3종에서 피연산자 간격·오버플로 없음 확인.

# Task #314 수행계획서: HWPX → 어댑터 → HWP 변환 후 TypesetEngine 페이지 수 차이 보정

상위: 잔존 사안 (Epic #309 외부)
선행: #313 (TypesetEngine main 전환)
브랜치: `task314` (task313에서 분기)

## 목적

HWPX → 어댑터 → HWP 변환 후 TypesetEngine 으로 페이지네이션 시 hwpx-h-02 가 +1쪽 발생. 어댑터가 보존하지 못하는 paragraph/line_seg 정보를 식별하고 보강하여 페이지 수 보존.

## 배경

#313 에서 진입점을 TypesetEngine 으로 전환하면서 발생한 회귀:

```
[#178 Stage 4] hwpx-h-02: orig=9, after_adapter=10
```

- HWPX 직접 로드 + TypesetEngine: 9쪽
- HWPX → 어댑터 → HWP → 재로드 + TypesetEngine: 10쪽 (+1)
- 비교: HWPX → HWP (어댑터 미적용) → 재로드 + TypesetEngine: **316쪽** (폭주)

어댑터는 폭주 316 → 10쪽으로 96% 해소. 잔존 +1쪽이 본 작업 대상.

격리된 테스트 (`#[ignore]`):
- `tests/hwpx_to_hwp_adapter.rs::stage4_page_count_recovered_hwpx_h_02`
- `tests/hwpx_to_hwp_adapter.rs::stage5_all_three_samples_recover_via_unified_entry_point`
- `tests/hwpx_to_hwp_adapter.rs::stage6_verify_recovered_for_all_three_samples`

## 원인 추정

어댑터(`src/document_core/converters/hwpx_to_hwp.rs`)는:
1. SectionDef 컨트롤 삽입 (PAGE_DEF 출력 보장)
2. 표 raw_ctrl_data + attr 동기화

소스 코멘트에 따르면 `reflow_zero_height_paragraphs` 가 IR 의 line_seg.vertical_pos 를 갱신하여 보존된다고 함. 이것이 정말 보존되는지 검증 필요. TypesetEngine 은 LINE_SEG vpos / line_height / line_spacing 등을 직접 사용하므로 미세한 차이에도 민감.

## 작업 범위

### 1. 정밀 측정: 변환 전후 LINE_SEG 비교

hwpx-h-02 에 대해:
- HWPX 직접 로드 후 LINE_SEG (vpos, line_height, line_spacing) 덤프
- 어댑터 적용 → HWP 직렬화 → 재로드 후 LINE_SEG 덤프
- 두 덤프 비교 → 차이 식별

도구: 기존 `dump` 명령 활용 또는 신규 `diag-hwpx-roundtrip` 명령

### 2. 차이 origin 식별

후보:
- a) HWP 직렬화기가 LINE_SEG 일부 필드를 손실
- b) HWP 재로드 시 LINE_SEG 재계산되어 원본 vpos 손실
- c) 어댑터가 HWPX 의 paragraph 일부를 변환하지 않음 (예: spacing, control_mask)
- d) HWPX 와 HWP의 paragraph property 차이 (sb/sa, page_break 등)

### 3. 어댑터 보강

식별된 차이를 어댑터에서 보강 (또는 직렬화기/파서 수정).

### 4. 격리 테스트 재활성화

- `#[ignore]` 속성 제거
- 3개 테스트 모두 통과 확인
- `cargo test` 회귀 0

### 5. hwpx-h-01, h-03 동시 검증

본 작업이 hwpx-h-02 만 대상이지만 hwpx-h-01, h-03 도 함께 검증 (회귀 가능성).

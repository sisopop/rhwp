# 수행계획서 — Task #1239: 미주 인라인 수식 줄 병합 (문20 S= 블록)

- **이슈**: #1239 (M100 / v1.0.0)
- **브랜치**: `feature/issue-1239-equation-multiline-merge` (base: `stream/devel`)
- **성격**: 레이아웃 버그 — 미주 문단 내 인라인 수식 줄 배정
- **작성일**: 2026-06-02

## 1. 증상

`samples/3-11월_실전_통합_2022.hwpx` 13쪽 문20 풀이의 "S = ∫… = … = … = 17" 정렬 블록에서,
PDF(한컴)는 각 "=…" 줄을 별도 줄로 두는데(5줄) rhwp 는 **두 줄을 한 줄로 병합**(=∫₀²(-2t³+8t)dt
… = [-½t⁴+4t²]…)하고 **위에 빈 줄(공백)** 이 생긴다(사용자 "4줄→3줄").

## 2. 조사 결과 (확인)

- S= 블록은 **단일 미주 문단 pi=602** ("(빈)" 텍스트, vpos 361142..372239).
- **5개 LINE_SEG**(각 "=…" 줄), vpos = 361142/363892/366642/369392/372239 — **균등 간격(~2750), 정상**.
- composed.lines = 5 = line_segs (개수 일치). 모든 composed 줄은 **runs=0(빈 텍스트)** — 즉 S= 수식들은
  **인라인 객체(U+FFFC placeholder)** 로 별도 렌더된다(텍스트 아님).
- → 줄 개수·vpos 는 정상이나, **인라인 수식이 잘못된 줄에 배정**됨: 3번째 줄이 비고 4번째 줄에
  두 수식이 겹침(수평 병합).

**→ 원인 영역**: 미주 문단의 **인라인 수식(U+FFFC) → LINE_SEG(줄) 배정** 로직.
수식 placeholder(8 UTF-16 단위)의 char 위치 ↔ 줄 경계(`line_break_char_indices`,
line_segs[i].text_start) 매핑이 어긋나 두 수식이 같은 줄로 배정되는 것으로 추정.

## 3. 목표

미주 문단 내 각 인라인 수식이 한컴 LINE_SEG 와 동일하게 **올바른 줄에 배정**되어, 문20 S= 블록이
PDF 와 동일하게 줄별 분리 렌더되도록 한다. 일반 문단/타 미주 회귀 없음 보장.

## 4. 적용 범위

| 영역 | 내용 |
|------|------|
| 조사 | 인라인 수식 char 위치 ↔ 줄 경계 매핑 정확 지점 특정 (U+FFFC 8유닛 offset) |
| 수정 | 수식 placeholder 의 줄 배정 보정 (`line_break_char_indices` 또는 인라인 객체 dispatch) |
| 검증 | 문20 S= PDF 정합 + 일반 문단/타 미주 수식 회귀 없음 |

### 범위 밖
- 미주 between-notes margin(#1238), 줄간격(#1236) — 별개.
- 수식 내부 레이아웃(#1233) — 수식 자체는 정상 렌더(개별 OK), 줄 배정만 문제.

## 5. 검증 방법

1. **시각**: `export-svg … -p 12` 문20 S= 블록 ↔ PDF 줄별 분리 정합.
2. **수치**: 인라인 수식별 배정 줄(vpos)이 line_segs 와 1:1.
3. **회귀**: 골든 스냅샷 + 전체 `cargo test`. 인라인 수식 포함 문서(exam 등) 시각 점검.

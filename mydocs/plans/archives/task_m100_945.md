## 1. 목표

`rhwp-studio`의 커서 이동 단축키를 한컴 공식 단축키 기준에 맞춰 플랫폼별로 분리한다.

| 플랫폼 | 단어 이동 | 줄 처음/끝 | 문단 이동 |
|--------|-----------|------------|-----------|
| macOS | `Option+←/→` | `Command+←/→`, `Home/End` | `Command+↑/↓` |
| Windows/Linux | `Ctrl+←/→` | `Home/End` | `Ctrl+↑/↓` |

선택 확장도 동일한 primitive를 재사용한다.

- macOS: `Option+Shift+←/→`
- Windows/Linux: `Ctrl+Shift+←/→`
- 공통: `Shift+Home/End`

## 2. 현재 코드 진단

대상 파일:

- `rhwp-studio/src/engine/input-handler-keyboard.ts`
- `rhwp-studio/src/engine/input-handler-text.ts`
- `rhwp-studio/src/engine/cursor.ts`

현재 구조:

1. `onKeyDown()`에서 `e.ctrlKey || e.metaKey`를 먼저 `handleCtrlKey()`로 보낸다.
2. `handleCtrlKey()`의 `ArrowLeft/ArrowRight`는 항상 `moveToLineStart()/moveToLineEnd()`를 호출한다.
3. 따라서 macOS `Command+←/→`는 기대 동작과 맞지만, Windows/Linux `Ctrl+←/→`도 줄 처음/끝으로 해석된다.
4. `cursor.moveToWordBoundary()`와 `Alt/Option+Arrow` 경로는 이미 존재하므로, 단어 이동 primitive 자체는 새로 크게 만들 필요가 없다.
5. IME 조합 종료 후 처리하는 `processPendingNav()`는 현재 방향키/Home/End의 기본 이동만 재생하므로, navigation keymap 정리 시 수정자 전달 범위도 함께 확인한다.

## 3. 수행 범위

### 포함

- navigation shortcut을 파일/편집 command shortcut과 분리한다.
- macOS 여부 판별 기준을 한 곳으로 모은다.
- `Ctrl+←/→`는 Windows/Linux에서 단어 이동으로 처리한다.
- `Command+←/→`는 macOS에서 줄 처음/끝으로 처리한다.
- `Home/End`와 `Shift+Home/End` 기존 동작을 보존한다.
- `Ctrl+↑/↓`는 Windows/Linux 문단 이동, `Command+↑/↓`는 macOS 문단/문서 이동 현행 동작을 한컴 기준과 대조해 정리한다.
- 단어 이동 선택 확장(`Shift`)을 기존 selection anchor 모델로 처리한다.
- 본문, 표 셀, 글상자/캡션 내부는 기존 `CursorState` primitive 재사용을 우선한다.

### 제외

- 단어 경계 알고리즘 자체의 대규모 재설계
- 한컴 전체 단축키 체계 재정의
- 브라우저/OS가 차단하는 reserved shortcut 우회
- 파일 저장, 복사, 붙여넣기 등 일반 command shortcut 변경

## 4. 구현 접근

1. `input-handler-keyboard.ts`에서 Ctrl/Meta 조합을 `handleCtrlKey()`로 보내기 전에 navigation key를 먼저 판별한다.
2. 플랫폼별 navigation keymap helper를 추가한다.
   - macOS 판별: `navigator.platform`/`navigator.userAgentData` 계열의 기존 사용 여부 확인 후 최소 범위로 도입
   - 단어 이동: `cursor.moveToWordBoundary(-1 | 1)`
   - 줄 이동: `cursor.moveToLineStart()/moveToLineEnd()`
   - 문단/문서 이동: 기존 `moveToParagraphBoundary()`/`moveToDocumentStart()/moveToDocumentEnd()` 의미를 재검토
3. `Shift`가 있으면 이동 전 `cursor.setAnchor()`, 없으면 `cursor.clearSelection()`을 호출하는 기존 패턴을 유지한다.
4. `processPendingNav()`도 실제 keydown과 다른 결과를 만들지 않도록 수정자 정보 전달 여부를 점검한다.
5. `Alt/Option+Arrow` 기존 경로는 macOS 단어 이동으로 유지하되, Windows/Linux에서 불필요한 동작으로 남길지 여부는 구현 계획서에서 최종 결정한다.

## 5. 검증 계획

### 정적/단위 검증

- `npm run build` (`rhwp-studio`)
- `npm test` (`rhwp-studio`)

### E2E 검증

신규 또는 기존 E2E에 다음 케이스를 추가한다.

- macOS keymap 시뮬레이션
  - `Alt/Option+ArrowLeft/Right` → 단어 이동
  - `Meta+ArrowLeft/Right` → 줄 처음/끝
  - `Alt/Option+Shift+ArrowLeft/Right` → 단어 선택
- Windows/Linux keymap 시뮬레이션
  - `Ctrl+ArrowLeft/Right` → 단어 이동
  - `Home/End` → 줄 처음/끝
  - `Ctrl+Shift+ArrowLeft/Right` → 단어 선택
- 기존 회귀
  - `Ctrl/Cmd+C/X/V/S/Z/Y/A` 등 command shortcut 유지
  - `Shift+End` 기존 선택 테스트 유지
  - 표 셀 내부 단어 이동이 기존 `moveToWordBoundaryInCell()` 경로를 사용

### 수동 확인

- 로컬 dev server에서 빈 문서 또는 샘플 문서로 macOS 실제 입력 확인
- 가능하면 Playwright/브라우저 이벤트 합성으로 Windows/Linux platform override 확인

## 6. 위험 평가

| 위험 | 평가 | 대응 |
|------|------|------|
| Ctrl/Meta 공통 처리 분리로 command shortcut 회귀 | 중 | navigation key만 선처리하고 나머지는 기존 `handleCtrlKey()` 유지 |
| macOS 판별 오류 | 중 | helper를 작게 만들고 E2E에서 platform override 검증 |
| 단어 경계가 한글/영문/숫자/문장부호 기대와 다름 | 중 | 기존 primitive를 우선 재사용하고 부족하면 별도 후속 이슈로 분리 |
| IME 조합 중 pending navigation과 실제 keydown 불일치 | 중 | `processPendingNav()` 수정 여부를 Stage 1에서 확인 |
| 표 셀/글상자/각주/머리말 내부 경로 누락 | 중 | 기존 CursorState 모드별 primitive 지원 범위 확인 후 구현 |

# Task #296 수행계획서 — WASM Canvas 경로 inline_tabs 무시 버그

## 이슈

[#296](https://github.com/edwardkim/rhwp/issues/296) — WASM Canvas 렌더 경로가 inline_tabs 무시 (#290 SVG 수정의 Canvas 버전 누락)

## 배경

PR #292 (#290 merge commit `085beb0`) 로 **네이티브 SVG 경로**의 cross-run 탭 감지가 `composed.tab_extended` (inline_tabs) 를 존중하도록 수정됨. `resolve_last_tab_pending` 헬퍼 + `paragraph_layout.rs` cross-run 블록 2곳.

그러나 브라우저 검증에서 `samples/exam_math.hwp` p.7 #18 "수열" 문항이 여전히 우측으로 밀림 확인:
- **SVG (`rhwp export-svg` CLI)**: ✅ 정상 (`translate(109.80, 162.69)`)
- **rhwp-studio Canvas (브라우저)**: ❌ 여전히 `x≈290.91`

## 근본 원인 (사전 조사 완료)

`src/renderer/layout/text_measurement.rs` 의 두 측정기 비대칭:

### `EmbeddedTextMeasurer` (네이티브, L167~330)

**이미 inline_tabs 분기 존재** (`tab_char_idx` + `style.inline_tabs[tab_char_idx]`):
```rust
if tab_char_idx < style.inline_tabs.len() {
    let ext = &style.inline_tabs[tab_char_idx];
    let tab_width_px = ext[0] as f64 * 96.0 / 7200.0;
    let tab_type = ext[2];                             // ⚠️ 전체 u16 사용 — 잘못됨
    match tab_type {
        1 => { /* RIGHT */ }
        2 => { /* CENTER */ }
        _ => { /* LEFT */ }
    }
}
```

**버그**: `tab_type = ext[2]` 전체 u16. 실제 HWP 값이 최소 256 이상 (high byte 에 종류 enum+1 이 인코딩됨) 이라 `1`, `2` 와 매칭 실패 → 전부 `_ =>` LEFT 처리. 이번 exam_math #18 케이스에서 inline LEFT 탭이라 우연히 결과가 맞아서 SVG 경로에서는 증상 노출 안 됨.

### `WasmTextMeasurer` (WASM, L520~680)

**inline_tabs 분기 자체가 없음**:
```rust
if c == '\t' {
    if has_custom_tabs {
        let (tab_pos, tab_type, _) = find_next_tab_stop(
            abs_x, &style.tab_stops, tab_w,
            style.auto_tab_right, style.available_width,
        );
        // ...
    }
}
```

→ `style.inline_tabs` 완전 무시. TabDef 만 사용 → `auto_tab_right` 폴스루 → RIGHT 오판 → 다음 run 우측 밀림.

## 목표

1. `WasmTextMeasurer::estimate_text_width` 에 `EmbeddedTextMeasurer` 와 동등한 inline_tabs 분기 추가
2. `WasmTextMeasurer::compute_char_positions` 에도 동일 처리 추가 (동기화)
3. **양쪽 모두 `tab_type = (ext[2] >> 8) & 0xFF` 로 high byte 만 사용** (PR #292 실증 포맷)
4. 네이티브 `EmbeddedTextMeasurer` 의 같은 버그도 함께 수정 (2곳)

## 접근 방식

### 옵션 A (선택) — 헬퍼 중앙화 + 두 측정기 모두 수정

PR #292 의 `resolve_last_tab_pending` 헬퍼와 유사하게, `text_measurement.rs` 내부에서 **inline 탭 종류를 해석하는 헬퍼** 를 분리:

```rust
/// inline_tabs[idx] 의 ext[2] 고바이트에서 탭 종류를 추출.
/// 반환: 0/1 = LEFT, 2 = RIGHT, 3 = CENTER, 4 = DECIMAL, others = LEFT
fn inline_tab_type(ext: &[u16; 7]) -> u8 {
    ((ext[2] >> 8) & 0xFF) as u8
}
```

두 측정기 네 함수(estimate × 2, compute × 2) 가 이 헬퍼를 호출.

### 옵션 B — WasmTextMeasurer 에만 inline_tabs 분기 복사

빠르게 고치는 방식. 단점: `EmbeddedTextMeasurer` 의 기존 버그(`tab_type = ext[2]`) 는 방치되어 이번 수정에서만 비대칭이 남음.

**→ 옵션 A 채택**. 네이티브 SVG 경로의 숨어있던 버그까지 함께 해결.

## 리스크

### R1 — 네이티브 `EmbeddedTextMeasurer` 수정으로 SVG 회귀 가능성
- 현재 `tab_type = ext[2]` 가 `_ =>` LEFT 로만 떨어져 우연히 LEFT 탭 케이스에서 동작.
- 수정 후 RIGHT/CENTER inline 탭이 실제로 RIGHT/CENTER 로 처리되면서 기존 샘플의 x 좌표가 바뀔 수 있음.
- **완화**: `git worktree` baseline 으로 184 페이지 SVG byte-diff (PR #292 와 동일 기법). 1건 이상 변경되면 이유 검증.

### R2 — `WasmTextMeasurer` 의 `tab_char_idx` 동기화
- `EmbeddedTextMeasurer` 는 `tab_char_idx += 1` 을 모든 탭 분기에서 증가시킴 (inline/custom/default 모두).
- `WasmTextMeasurer` 복사 시 동일 관리 필요.
- **완화**: 네이티브 코드를 그대로 복사해 구조 일치.

### R3 — `ext[2]` 포맷 재확인
- PR #292 의 실증 (high byte = enum+1, low byte = fill) 을 신뢰하되, 이번 수정에서 한 번 더 덤프로 검증.
- **완화**: Stage 1 에서 임시 `eprintln!` 로 ext[2] 16진 확인.

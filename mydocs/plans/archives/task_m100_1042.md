### 2.3 한컴 viewer 의 layout 동작 추론

한컴 viewer 는 PARA_LINE_SEG 없어도 layout 가능 (paragraph header + text + font metric → line break + layout). PARA_LINE_SEG 는 layout cache.

**rhwp baseline (composer fallback path) 가 viewer 동작 모방**으로 sample16 64 정합 (baseline). 다만:
- 2022 의 +1 baseline 회귀 — paragraph data 동일이지만 section_def/header 차이 추정
- k-water-rfp 의 +2 over-split — line_segs 완전 정합인데 paragraph_layout 의 height 측정이 한컴 viewer 보다 미세하게 큼 (paragraph 별 +0.5~+1 px 누적)

### 2.4 합성 공식 도출 (2018/2024 reverse engineering)

```
lh = th = max base_size
ls = th × (line_spacing - 100) / 100   (Percent type)
bl = th × 0.85
cs = ParaShape.margin_left
sw = body_width - margin_left - margin_right
tag: 첫 line=0x00060000, 이어지는 lines=0x00160000
```

`lh+ls` 999/999 매칭 입증 (variant 와 2018/2024 둘 다 채워진 paragraph 의 lh+ls 100% 매칭).

### 2.5 paragraph_layout 의 multi-path 차이 (본 task 의 모든 fix 시도 회귀 원인)

paragraph_layout 이 line_segs.empty vs filled 에서 paragraph height 측정 알고리즘 다름:
- **empty path**: composer fallback → ComposedLine line_height=400 sentinel → corrected_line_height(5.33, 17.33, Percent, 160) → 27.7 px/line ✓
- **filled path**: ls.line_height + ls.line_spacing 합산 → 정합

본 task 의 모든 fix 시도 회귀 (sentinel 변경 / corrected_line_height 분기 / line_segs 합성 + word wrap / font shaper) — paragraph_layout multi-path 통합이 진정한 본질.

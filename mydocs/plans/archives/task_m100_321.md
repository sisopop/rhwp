# Task #321: TypesetEngine 문단 높이 산정을 LINE_SEG 기반으로 정렬 — 수행계획서

## 배경

`samples/21_언어_기출_편집가능본.hwp` 1페이지 우측단 하단에서 문단 pi=28/29/30이 본문 영역 아래쪽 경계(y=1436.2)를 초과하여 렌더되며, pi=29와 pi=30이 같은 y 좌표(1421.5)에 겹쳐 찍힌다. 사용자 보고: "1페이지 우측하단 겹쳐보임."

### 재현

```
$ ./target/release/rhwp export-svg samples/21_언어_기출_편집가능본.hwp -p 0 --debug-overlay
LAYOUT_OVERFLOW: page=0, col=1, para=28, y=1445.7, bottom=1436.2, overflow=9.5px
LAYOUT_OVERFLOW: page=0, col=1, para=29, y=1445.7, bottom=1436.2, overflow=9.5px
LAYOUT_OVERFLOW: page=0, col=1, para=30, y=1445.7, bottom=1436.2, overflow=9.5px
```

### 환경 정리

- 현재 `task318` 기준: 페이지 수 15 (PDF 일치). Task #313에서 `TypesetEngine`이 default paginator로 전환된 상태.
- 본 task321은 `task318`을 기반으로 분기한다 (devel 아님). Task #313~#318의 후속이며 devel에는 아직 merge되지 않은 브랜치 체인 위에서 작업.

### 원인 분석

1. **Pagination 단계 (TypesetEngine)**: col 1의 items 누적 높이 `current_height = 1223.1px` 가 `available = 1226.4px`에 들어간다고 판단하여 pi=30까지 모두 배치. `height_for_fit = total_height - trailing_ls` 기반 누적으로, 포맷터가 산출한 줄 높이/여백 합만 본다.
2. **Layout 단계 (LayoutEngine)**: 실제 배치 시 `y_offset`을 HWP의 LINE_SEG `vertical_pos`로 보정한다(`vpos_page_base`/`vpos_lazy_base`). 보정 결과 pi=28의 y=1407.8, pi=29의 y=1421.5, pi=30의 y=1421.5. pi=29/30은 col_bottom 초과로 보정 조건(`end_y <= col_bottom`)을 만족하지 못해 스킵되고, 순차 y도 col_bottom을 넘어 동일 y로 클램프되어 **두 문단이 같은 위치에 중첩 렌더됨**.

**핵심 불일치**: TypesetEngine은 포맷터가 독립적으로 계산한 줄 높이/여백으로 "fits"를 판단하나, LayoutEngine은 HWP LINE_SEG vpos를 y_offset 기준으로 사용한다. 두 계산이 약 **~15px 드리프트**를 만들어 TypesetEngine이 "들어간다"고 본 문단이 실제 배치에서 col 하단을 넘어선다.

## 목표

**TypesetEngine의 fit 판단을 LayoutEngine의 실제 y_offset 진행(=LINE_SEG vpos 기반)과 일치시켜**, HWP가 원본에서 의도한 단/페이지 경계를 그대로 재현한다. 결과:

- 21_언어 pi=29, pi=30은 col 1 하단이 아닌 다음 단/페이지로 자연 흐름.
- pi=28은 경계에 걸치지 않게 됨 (또는 타이트하게 배치되어 오버플로우 0).
- 총 페이지 수 15 유지 (또는 PDF와 동일 선에서 변동).

## 범위

### 수정 대상

- `src/renderer/typeset.rs` — `format_paragraph()`, `typeset_paragraph()`의 height 산정 로직에 LINE_SEG vpos 기반 실측 경로 추가.
- 단위 테스트: LINE_SEG가 주어진 문단의 fit 누적이 LayoutEngine과 일치하는지 확인.
- 회귀 테스트: 기존 `samples/` 전체 `export-svg` 페이지 수/레이아웃 변동 확인.

### 제외

- LayoutEngine 수정 (y_offset 산정 로직은 그대로 유지, TypesetEngine이 이를 따라간다).
- 파서 변경 (LINE_SEG 데이터는 이미 Paragraph에 있음).
- HWPX 전용 처리: LINE_SEG가 결여된 경로는 기존 포맷터 기반 계산 유지 (fallback).

## 접근 원칙

> HWP 파일에 LINE_SEG가 있으면 **그 값이 진실(source of truth)**이고 포맷터 계산은 근사치다. TypesetEngine은 LayoutEngine과 같은 데이터 기반으로 fit을 판단해야 한다.

## 단계 (4단계)

### Stage 1 — 드리프트 origin 정량화

- 21_언어 col 1 page 1의 각 문단(pi=9 partial, pi=10..30)에 대해:
  - TypesetEngine의 `fmt.total_height` 누적값
  - LayoutEngine의 실제 y_offset 진행값 (vpos 보정 포함)
  - 두 값의 차이 로깅
- 진단 도구: `--dump-typeset-vs-layout` (임시 디버그 플래그) 또는 테스트 단위 로거.
- 목표: 드리프트가 어떤 문단/어떤 구성요소(spacing_before, line_spacing, 개체 높이 등)에서 발생하는지 파악.
- 결과물: `mydocs/working/task_m100_321_stage1.md` 에 정량 표 + origin 요약.

### Stage 2 — TypesetEngine height 산정 LINE_SEG 정렬

- `FormattedParagraph.total_height` 산정을 LINE_SEG 기반 실측 경로로 교체:
  - Paragraph가 LINE_SEG를 가지고 있으면 `(last_seg.vpos + last_seg.lh) - first_seg.vpos` 를 사용.
  - LINE_SEG가 없거나 불완전하면 기존 포맷터 계산을 사용 (fallback).
- `current_height` 누적도 동일 기준으로 전환.
- **안전장치**: LINE_SEG 실측값이 포맷터 계산값과 크게 다를 때(예: ±20% 초과)는 경고 로깅 후 더 큰 값 사용 (보수적 fit).
- 단위 테스트: 합성 Paragraph로 양 경로가 같은 fit 결과를 내는지 확인.
- 결과물: `mydocs/working/task_m100_321_stage2.md`.

### Stage 3 — 21_언어 샘플 적중 확인

- 본 샘플 1페이지 col 1 오버플로우 3건 해소 확인.
- `dump-pages -p 0` 에서 col 1 items 개수 변화 관찰 (pi=29, pi=30이 다음 단/페이지로 이동).
- 기타 페이지(2~15)도 동일 원리로 점검.
- 결과물: `mydocs/working/task_m100_321_stage3.md` (Before/After 비교).

### Stage 4 — 회귀 + 최종 보고서

- `cargo test` (전체 992개 이상) 통과 확인.
- 주요 샘플 export-svg 회귀:
  - 21_언어: 15쪽 유지 + 오버플로우 0 목표.
  - exam_math: 20쪽 유지.
  - exam_kor: 페이지 수 변화 기록.
  - exam_eng: 페이지 수 변화 기록.
  - 기타 `samples/` 파일들: 경고 증가 없음 확인.

## 리스크

- **타 샘플 회귀**: LINE_SEG 기반 height가 포맷터 계산과 괴리가 큰 샘플에서 페이지 수 변동 가능. → Stage 2의 안전장치와 Stage 4 회귀 표로 추적.
- **LINE_SEG 신뢰성**: 일부 문단은 LINE_SEG가 HWP 저장 시점의 layout 결과라 후편집/재파싱 경로에서 불일치할 수 있음. → fallback 경로 + 경고 로깅으로 보호.
- **TypesetEngine-LayoutEngine 2중 소스**: 본 task는 pagination 단계를 layout 단계와 일치시키는 작업이지만, 향후 재편집/재레이아웃 시 LINE_SEG 재계산 경로와의 상호작용은 별도 이슈.

## 1. 배경

#142 1차 작업에서 수식 파서/레이아웃 버그 다수를 수정했으나, 시각 검증 중 탭+TAC 위치 계산 불일치 문제가 발견되었다. 탭 position과 수식 TAC 너비는 같은 조판 흐름에서 상호 의존하므로 함께 처리한다.

### 1차에서 해결 완료된 항목

| 항목 | 수정 내용 |
|------|----------|
| OVER 파서 | 중위 연산자 방식으로 재설계 |
| sqrt {n} of {x} | 중괄호 인덱스 패턴 지원 |
| bar{rm AD it} | parse_single_or_group에서 RBrace 처리 |
| 적분 nolimits | INT 계열을 MathSymbol+SubSup으로 변경 |
| 함수 첨자 | try_parse_scripts에서 Thin 공백 건너뛰기 |
| Fraction baseline | axis_height(0.25em) 기반 계산 |
| font_size_from_box | 복합 요소 height를 font-size로 오용하는 버그 수정 |
| 분수선 y좌표 | baseline - axis_height로 보정 |
| Unicode 폭 | 수학 기호별 세분화 (prime 0.3 등) |

### 미해결 문제

1. **탭 position `/2.0`**: `style_resolver.rs:618`에서 탭 스톱 위치를 `/2.0`으로 나누고 있음. 한컴 격자 비교 결과 `/2.0`이 올바른 변환임이 확인됨 (HWP 탭 position이 실제 좌표의 2배로 저장)
2. **원문자 전각 폭**: `is_fullwidth_symbol`에 ①~⑤ 범위 미포함 → 반각(0.5×fs)으로 계산되어 탭 점프 시작점이 어긋남
3. **텍스트 폭 추정 오차**: 원문자/공백 등의 폭 추정이 한컴과 다르면 탭 스톱 이전의 x 위치가 어긋남

### 한컴 격자 비교로 확인된 사실

- **탭 스톱 위치 자체는 정확**: 숫자+탭, 원문자+탭 비교 시 2/3줄의 탭 위치가 일치
- **탭은 수식 TAC 너비에 영향받지 않음**: 탭은 현재 x 위치에서 다음 탭 스톱으로 점프
- **문제의 본질**: 탭 스톱 이전의 텍스트(원문자+공백) 폭 추정 오차가 탭 점프 시작점을 잘못 계산

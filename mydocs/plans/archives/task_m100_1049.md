## 1. 배경

#1046에서 본문 하단 `LAYOUT_OVERFLOW` 를 18→5 로 정합했으나, 잔여 1건(약 4.6px,
page 112 보안 서약서 폼 pi=781)은 **측정 통일(B)과 다른 축** — 렌더러 줄높이 모델의
과대 계산이 원인으로 확인되어 별도 이슈(#1049)로 분리했다.

부모 진단: `mydocs/working/task_m100_1046_v2_stage3_781diag.md`

## 2. 문제 정의 (진단 확정 사항)

문제 문단 **pi=760** "(용역업체 대표자용)", align=Center, **line spacing=110%(Percent)**,
hanging indent=-13100, 모든 char shape = 15.0pt(id=122 "(", id=450 끝마커):

| 항목 | 값 |
|------|-----|
| 올바른 줄높이 (HWP vpos 피치) | pi=761.vpos − pi=760.vpos = 1652 HU = **22.0px** |
| 기대 계산 | 15pt × 110% = 16.5pt = **22.0px** (vpos 피치 일치) |
| 렌더러 실측 | **34.8px** (+12.8px 과대, ~23.7pt 상당) |

- 헤더(pi=757~760)에서 **21.7px 누적 드리프트**, 이후 폼 줄(769~781)에서 상수 유지.
- 페이지네이터(`cur_h`)는 vpos 기반으로 정확(들어감 판정)하나, **렌더러 줄높이 모델**이
  pi=760 류 문단에서 어긋난다.

## 3. 의심 원인 (Stage 1에서 격리)

진단 문서가 제시한 3가지 가설 + 코드 조사로 추가한 후보:

1. **끝마커 빈 run charshape**가 max_fs 에 포함 → 의도치 않은 큰 폰트가 max 에 반영
   - 단, 진단상 id=450 끝마커도 15.0pt 로 기록 → 이 경우만으로는 12.8px 설명 부족.
2. **음수 indent(hanging indent -13100)** 와 결합한 줄높이/줄 분할 계산 오류.
3. **작은 line-spacing %(110%)** 의 줄높이 모델 — `corrected_line_height`(mod.rs:545-561)
   재계산 분기, line_spacing 이중 가산 가능성(Task #969 double-count 방지 로직 점검).
4. **3개 경로의 max_fs/줄높이 계산 불일치** — layout(`paragraph_layout.rs:1145-1166`),
   `height_measurer.rs:260-276`, `typeset.rs:1461-1493` 가 동일 규칙인지 확인.
   특히 fallback 12.0pt 처리, `recompute_lh` 시 `line_spacing_px=0` 분기.

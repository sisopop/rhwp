## 2. 현황 분석 (2026-06-13 — 코드 확인 완료)

### 결함 구조

| 축 | 위치 | 현황 |
|----|------|------|
| 분포 | aift 전 섹션 단어 경계 정밀 분류 | **15건 전량 `hp:pic`(그림) 하위** — 이슈 표현 "도형 설명"과 달리 실제 결손 경로는 그림 단일 경로 |
| parser | `parse_picture`(`:2182`)·`parse_shape_object`(`:3464`) | `common.description`으로 정상 적재 — **수정 불요** |
| serializer | `write_picture`(picture.rs) | shapeComment 방출 없음 ← **결함 본체**. 도형 경로(shape.rs:695)는 기존 방출 코드 존재 (rect 계열) |
| 게이트 | `diff_documents` | `description` 비교 없음 — 사각 |

실물(aift) 자식 순서: `… pos → outMargin → shapeComment → …` — OWPML 순서 문서
(table.rs doc: "(caption, shapeComment, …)")와 함께 1단계에서 caption 공존 사례 포함 확정.

### 트러블슈팅·열린 PR 점검

- 트러블슈팅: shapeComment 선행 기록 없음.

  (단, #1405는 표 pageBreak 파서 역변환을 포함 — **#1393 착수 시 선점검 필요**로 기록)

  방출 위치(caption과의 순서)는 실물 공존 사례로 확정한다.

### 분류 방법론 주의 (자기 정정)

종전 역방향 분류 스크립트의 `<hp:line` 매칭이 `<hp:lineseg`에 오매치되는 결함을
확인·수정했다 (#1406에서 oksure가 정정한 "pic 8+line 3 → pic 9+container 2"와 동일
원인). 본 계획의 분포는 단어 경계 정밀 분류 기준이다.

## 3. 목표와 완료 조건

1. HWPX 저장 시 그림의 `hp:shapeComment`(IR `common.description`)가 원본대로 방출된다.
2. `diff_documents`가 Picture(및 도형 공통) description을 비교하여 사각이 해소된다.
3. aift 15건 전량 복원, 전수 배치 신규 실패 0.

## 4. 단계 개요 (구현계획서에서 상세화 — 3단계)

1. **측정/순서 확정**: aift 전수 shapeComment-caption 공존·순서 실측 + 도형 경로
   (shape.rs 기존 방출) 커버리지 측정 + 게이트 동승 영향 사전 판정
2. **serializer + 게이트**: `write_picture`에 shapeComment 방출 (실물 순서) +
   게이트 description 비교 동승 (Picture arm + `shape_caption` 류 접근 — #1403 패턴)
   + 단위 테스트
3. **전수 검증 + 문서**: 배치, aift 복원 수치, 매뉴얼 갱신, 최종 보고서

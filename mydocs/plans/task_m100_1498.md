# Task M100 #1498 수행계획서 — 확장 다운로드 관찰자 신선도 가드

- 이슈: #1498 "[Bug] 확장 0.2.6: 과거 다운로드 기록 접근으로 미사용 문서 뷰어 창 다발 열림"
- 마일스톤: M100 (v1.0.0)
- 브랜치: `local/task1498`
- 작성일: 2026-06-24

## 1. 배경

확장 0.2.6 업데이트 후, 과거에 다운로드한 HWP/HWPX 문서들의 뷰어 창이 다수 자동으로
열린다. #1471/#1480 에서 다운로드 가로채기를 `onDeterminingFilename` → `onCreated`/
`onChanged` 관찰자로 전환하면서 발생한 회귀.

## 2. 원인 (소스 조사 완료)

`rhwp-chrome/sw/download-interceptor.js` · `rhwp-firefox/sw/download-interceptor.js`:

- `onChanged` 리스너가 `!handled.has(id) && filename 확정` 이면 `downloads.search({id})`
  로 항목을 재조회하여 처리한다.
- service worker 재기동 시 in-memory `handled` Set 이 초기화된다.
- 브라우저에 쌓인 **과거 다운로드 항목**에 `onChanged` 가 발화하면, `onCreated` 를 거치지
  않았음에도 `handled` 에 없고 확장자/MIME 이 매치되어 `openViewer` 가 호출된다.
- `shouldInterceptDownload`(공통)는 확장자/MIME 만 검사하고 **다운로드 시점(신선도)을 보지
  않는다.** 과거/현재 다운로드를 구분하지 못한다.

이전 `onDeterminingFilename` 은 "지금 결정 중인 다운로드"에만 발화하여 이 문제가 없었다.

## 3. 목표

다운로드 관찰자가 **service worker 기동 이후 새로 시작된 다운로드만** 처리하도록 하여,
과거 기록 항목이 뷰어로 열리지 않게 한다.

## 4. 수정 방향

- **핵심**: `onCreated` 로 관측한 id 집합(`seen`)을 도입한다. `onCreated` 는 새 다운로드에만
  발화하므로 SW 기동 이후 시작된 다운로드만 `seen` 에 들어간다.
- `onChanged` 의 재판정 경로를 **`seen` 에 있는 id 에 한해서만** 수행한다. 과거 항목은
  `onCreated` 를 거치지 않아 `seen` 에 없으므로 자동 제외된다.
- (방어선) 필요 시 `startTime` 기반 최근성 검사를 보조로 둘 수 있으나, `seen` 기반이
  단순·확실하므로 우선 적용.
- chrome/firefox 양쪽 동일 적용. 공통 판정(`shouldInterceptDownload`)은 변경하지 않는다
  (확장자/MIME 판정은 유효하며, 신선도는 관찰자 레이어에서 가드).
- 정상 동작 보존: 새 HWP 다운로드(onCreated 즉시 판정 가능 / filename 확정 후 onChanged
  재판정) 경로는 그대로 동작해야 한다.

## 5. 범위

- **포함**: `rhwp-chrome/sw/download-interceptor.js`, `rhwp-firefox/sw/download-interceptor.js`
  의 `seen` 기반 신선도 가드.
- **제외**: 공통 판정 로직, WASM/Rust, viewer-launcher.

## 6. 검증 기준

1. SW 재기동 후 과거 다운로드 항목에 onChanged 발화 → 뷰어 미오픈.
2. 새 HWP/HWPX 다운로드 → 뷰어 정상 오픈(onCreated 또는 onChanged 재판정 경로).
3. SW 테스트(mock): 과거 항목(onCreated 미관측 id) onChanged → 무시 / 새 항목 → 오픈.
4. 기존 SW 테스트 회귀 없음. 확장 빌드 통과.

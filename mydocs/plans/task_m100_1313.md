# 수행계획서 — Task M100 #1313

## 제목
적분기호(∫) 상·하한이 위아래로 벌어지고 기호 높이가 정답보다 작음

- 이슈: edwardkim/rhwp#1313
- 마일스톤: M100 (v1.0.0)
- 브랜치: `local/task1313` (base: devel == stream/devel @ 345e32de)

## 1. 배경 / 증상

`samples/3-10월_교육_통합_2022.hwp` 9페이지의 적분식 3개에서 적분 상·하한(범위)이
적분기호로부터 위아래로 벌어져 보인다. 다른 페이지 적분식에서도 동일하게 관찰된다.

작업지시자 신고: "적분기호 오른쪽 인자(= 범위, 상·하한)의 위치가 아래위로 약간 벌어짐."

## 2. 현상 분석 (코드 무수정 조사 완료)

정답 자료: `pdf/3-10월_교육_통합_2022.pdf` (한글 2022) p.9 — macOS/Linux 환경 1차 정답지.

| 항목 | 우리 렌더 | 한글 2022 (정답) |
|------|----------|------------------|
| 적분기호 ∫ | 짧고 곧음 | 길고 기울어짐 |
| 상한(4·2) | 기호 위로 떠서 간격 발생 | 기호 윗끝에 밀착 |
| 하한(0) | 기호 아래로 떨어짐 | 기호 아랫끝에 밀착 |

원인 위치: `src/renderer/equation/layout.rs` `layout_integral()` (703행~)

```rust
let op_fs = fs * BIG_OP_SCALE;   // BIG_OP_SCALE = 1.5
let op_h  = op_fs;
let op_baseline = op_h * 0.6;
let sup_shift = op_h * 0.1;   // 상한: 기호 상단 위
let sub_shift = op_h * 0.55;  // 하한: 기호 중앙 아래
```

- 실제 렌더되는 ∫ 글리프 높이가 정답보다 작다.
- 상·하한은 `op_h` 기준으로 위·아래로 밀려나는데, 글리프가 작아 그 빈 공간에 상·하한이
  떠 보인다 → "위아래로 벌어짐".

즉 **(a) 적분기호 글리프 크기**와 **(b) 상·하한 수직 오프셋이 글리프 실제 높이와 어긋남**이
겹친 문제이다.

## 3. 수정 방향 (작업지시자 승인 사항)

작업지시자 결정: **"기호 높이도 함께"** — 적분기호 글리프를 정답에 맞게 키우고,
상·하한을 그 끝에 밀착하도록 오프셋을 재조정한다.

## 4. 영향 범위

- `src/renderer/equation/layout.rs` — `layout_integral()` (필요 시 `BIG_OP_SCALE`/관련 상수)
- `src/renderer/equation/svg_render.rs` — BigOp integral 렌더 분기 (op_y, 첨자 배치)
- `src/renderer/equation/canvas_render.rs` — 동일 분기 (SVG와 동작 일치 유지)
- `src/renderer/equation/skia` (해당 시) — Skia 경로 일치 확인

> HWP3 전용 분기 아님. 공통 수식 레이아웃/렌더 모듈만 수정.

## 5. 리스크

- 적분기호 크기 변경은 ∑/∏ 등 다른 큰 연산자(`BIG_OP_SCALE` 공유)에 영향을 줄 수 있다
  → 적분 전용으로 분리하거나 영향 검증 필요.
- 상·하한 오프셋 변경이 적분 뒤 피연산자(f(x)dx) trailing 간격(`BIG_OP_TRAIL_PAD`)·
  전체 식 baseline 정렬과 상호작용할 수 있다.
- 기존 회귀 테스트(`issue_1139`, `issue_1219`, `issue_1061` 등) 영향 확인 필요.

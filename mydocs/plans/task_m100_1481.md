## 1. 배경

rhwp-studio에서 일반 표를 만든 뒤 셀 컨텍스트 메뉴의 `왼쪽에 칸 추가하기`를 실행하면 표 높이가
얇은 선처럼 줄어드는 증상이 제보되었다.

추가 확인 결과, 한컴 도움말의 `줄/칸 추가하기 <Alt+Insert>`와 `줄/칸 지우기 <Alt+Delete>`는
방향/개수를 고르는 대표 대화상자 흐름인데, rhwp-studio는 단건 빠른 실행 명령만 있고 대표
대화상자 명령과 단축키 매핑이 누락되어 있었다.

기존 조사 결과, UI 명령은 다음 경로로 이어진다.

```text
table:insert-col-left
-> wasm.insertTableColumn()
-> DocumentCore::insert_table_column_native()
-> Table::insert_column()
-> Table::update_ctrl_dimensions()
```

일반 `createTable()` 경로는 셀 자체 height를 `282HU`로 만들고, 표 전체 `common.height`는 렌더용 높이인
`1282HU * 행수`로 둔다. 그런데 열 추가 후 `update_ctrl_dimensions()`가 `get_row_heights()` 합으로
`common.height`를 다시 써서 표 전체 높이가 셀 height 합 수준으로 축소될 수 있다.

## 2. 확인한 사실

- `rhwp-studio/src/command/commands/table.ts`
  - `table:insert-col-left` / `table:insert-col-right`가 `wasm.insertTableColumn()`을 호출한다.
- `src/document_core/commands/table_ops.rs`
  - `insert_table_column_native()`가 `table.insert_column()` 후 `recompose_section()`과 `paginate_if_needed()`를 호출한다.
- `src/model/table.rs`
  - `insert_column()`은 새 셀을 만들고 마지막에 `update_ctrl_dimensions()`를 호출한다.
  - `delete_column()`도 열 삭제 후 마지막에 `update_ctrl_dimensions()`를 호출한다.
  - `update_ctrl_dimensions()`는 `get_row_heights().iter().sum()`으로 `common.height`와 `raw_ctrl_data[HEIGHT]`를 갱신한다.
- `src/document_core/commands/object_ops.rs::create_table_native`
  - 일반 표 생성 시 셀 height는 `padding_top + padding_bottom = 282HU`이다.
  - 표 common height는 `padding_top + line_height(1000) + padding_bottom = 1282HU` 기준이다.
- `mydocs/working/task_m100_1443_stage14.md`
  - PR #1446에서는 `Table.common.height`를 표 외곽 높이 기준으로 쓰는 렌더 보정이 들어갔다.
- `mydocs/working/task_m100_1443_stage19.md`
  - 보상 resize에서는 표 외곽 크기를 보존하기 위해 `common.width/height`와 raw bytes를 복원하는 패턴이 이미 있다.

## 3. 회귀 후보

잠복 원인은 초기 구현부터 존재한 `cell.height`와 `common.height`의 의미 차이로 보인다.

화면 즉시 회귀 후보는 다음이다.

- `784e0063 Task #1151 v6 Stage 10: Table::update_ctrl_dimensions self.common 동기화 fix`
  - 이전에는 raw bytes만 갱신하던 경로가 `self.common.height`도 갱신하도록 바뀌었다.
  - 이로 인해 열 추가 직후 레이아웃이 축소된 `common.height`를 더 직접적으로 사용할 수 있다.
- PR #1446 Stage 14 이후
  - 렌더러가 `Table.common.height`를 표 외곽 높이 기준으로 더 적극적으로 맞추므로, 잘못 축소된 common height가 더 잘 드러난다.

## 4. 작업 범위

### 4.1 모델/명령 보정

- 열 추가/삭제 시 기존 표의 시각적 높이(`common.height`)를 보존한다.
- 행 추가 시 기준 행의 시각적 높이만큼 `common.height`를 증가시키는 방안을 검토한다.
- `raw_ctrl_data[HEIGHT]`와 `table.common.height`를 함께 맞춘다.
- 기존 PR #1446의 보상 resize common width/height 복원 패턴을 구조 편집 경로에 맞게 재사용한다.

### 4.2 회귀 테스트

- 일반 `createTable()`로 만든 표에서 열 추가/삭제 후 `common.height`가 납작해지지 않는지 고정한다.
- 가능하면 행 추가도 함께 검증해, 열 추가 보정과 행 추가 보정의 의미를 분리한다.
- 기존 `createTableEx()` 명시 높이 경로와 TAC 표 경로는 영향 여부를 확인한다.

### 4.3 Studio 시각 검증

- IAB에서 일반 표 생성 후 `왼쪽에 칸 추가하기`, `오른쪽에 칸 추가하기`, `칸 지우기`를 실행한다.
- 표 높이가 기존 행 높이를 유지하는지 확인한다.
- `Alt+Insert`로 `줄/칸 추가하기` 대화상자가 열리고, 방향과 줄/칸 수를 선택해 적용되는지 확인한다.
- `Alt+Delete`로 `줄/칸 지우기` 대화상자가 열리고, 줄/칸 선택 삭제가 적용되는지 확인한다.
- 필요 시 일반 표와 상세 대화상자 표를 분리해 확인한다.

# Task M100 #1413 수행계획서 — 3인자 초과 WASM API에 options object(*Ex) 병행

- 이슈: #1413 "[API] positional 인자 3개 이상 API에 options object 도입 제안" (sacru2red)
- 마일스톤: M100 (v1.0.0)
- 브랜치: `local/task1413` (`local/devel` 분기)
- 작성일: 2026-06-20

## 1. 제안 (외부 기여자 → 메인테이너 직접 처리)

`@rhwp/core` 소비자가 `insertPicture` 의 중간 삽입형 변경(`cell_path_json` 4번째 인자
추가)으로 타입 오류(`Expected 11-13 arguments, but got 10`)를 겪음. positional 인자가
많은 API 는 중간 삽입/순서 변경 시 downstream 수정 비용이 큼 → **options object(JSON)
형태 병행** 제안.

## 2. 전수 조사 (완료)

- wasm_bindgen public API 293개 중 **3인자 초과 126개**. 전부 *Ex 병행은 과도(유지보수·
  문서·바인딩 부담, 안정 API 다수는 이득 없음).
- 작업지시자 결정: **고인자(7+) 핵심 API 우선 + 설계 관행 문서화**.
- **7인자 이상 26개** 대상 (인자수 desc): insertPicture(13), insertClickHereFieldInCell(11),
  splitTableCellsInRange(10), splitTableCellInto(9), moveVertical(9), 그 외 8/7인자 21개
  (대부분 `*InCell`/`*ByPath` 계열).

## 3. 설계 방침 (구현계획서에서 확정)

- **기존 positional API 유지**(하위 호환) + **`*Ex(options_json: &str)` 병행 추가**
  (`createTableEx` 확립 패턴 동형). JSON 필드 추출은 기존 `json_u32`/`json_bool` 등 헬퍼
  + 필요 시 보강.
- **바이너리/특수 인자는 하이브리드**: `insertPictureEx(options_json, image_data: &[u8])`
  — image_data 는 별도 Uint8Array 인자, 좌표·cellPath·크기·확장자 등은 JSON. (base64
  오버헤드 회피.)
- **설계 관행 문서화**: "신규/확장 가능성 높은 고인자 API 는 *Ex(options) 병행 권장 +
  positional 중간 삽입 금지(끝에 추가)" + breaking change 대응 가이드(CHANGELOG 표기·
  migration snippet).

## 4. 목표와 완료 조건

1. 우선 대상 *Ex API 추가 (insertPicture 우선, 이후 고인자 순). 기존 positional 불변.
2. 각 *Ex 가 positional 과 동일 동작 (round-trip 동치 테스트).
3. 설계 관행 가이드 문서(`mydocs/manual/` 또는 `docs/`) + CHANGELOG 표기 규약.
4. `cargo test`/WASM 빌드 그린, 기존 호출 회귀 0.

## 5. 단계 개요 (구현계획서에서 확정 — 3~5단계 예상)

1. **insertPictureEx + 패턴 확립**: 하이브리드 *Ex 1개 구현 + 동치 테스트 + 헬퍼 보강.
2. **고인자 일괄(8~13인자)**: insertClickHereFieldInCell/split* 등 + 동치 테스트.
3. **7인자 군**: `*InCell`/`*ByPath` 계열 + 동치 테스트.
4. **가이드 문서 + CHANGELOG 규약** + WASM 빌드 + 최종 보고서.

(단계 수·묶음은 구현계획서에서 26개를 위험·유사도 기준으로 분할.)

## 6. 위험과 대응

| 위험 | 대응 |
|------|------|
| 26개 일괄 → 대형 diff·회귀 | 단계 분할 + 각 *Ex 동치 테스트(positional==Ex) |
| JSON 파싱 헬퍼 한계(중첩 배열 등) | createTableEx 의 수동 파서 한계 → serde_json 활용 검토 |
| 바이너리 인자 설계 비일관 | 하이브리드 규약 문서화(image_data류는 별도 인자) |
| public API 표면 증가 | 고인자(7+)로 제한 + 가이드로 신규 API 방향 통일 |

## 7. 검증 방법

- `cargo test --profile release-test --tests` + fmt + clippy.
- WASM 빌드(Docker) + 타입 정의(rhwp.d.ts)에 *Ex 노출 확인.

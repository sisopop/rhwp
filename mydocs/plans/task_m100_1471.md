## 1. 배경

사용자 보고에 따르면 다른 Chrome 확장이 blob URL PDF를 다음과 같이 다운로드할 때, rhwp가 활성화되어 있으면 지정한
하위폴더/파일명이 무시되고 blob UUID 기반 파일명이 다운로드 루트에 저장된다.

```javascript
chrome.downloads.download({
  url: 'blob:chrome-extension://<id>/<uuid>',
  filename: 'Privacy Filter/Filtered/Session_2026-06-22_1434/filtered_xxx.pdf',
  conflictAction: 'uniquify',
});
```

rhwp를 비활성화하면 같은 다운로드가 지정 하위폴더와 파일명으로 정상 저장된다.

## 2. 재현 결과

Mac Chrome에서 최소 재현 확장을 만들어 동일 현상을 확인했다.

| 조건 | 요청 filename | 실제 저장 결과 |
|------|---------------|----------------|
| rhwp ON | `Privacy Filter/Filtered/Session_2026-06-22_1434/filtered_test.pdf` | `/Users/melee/Downloads/5ec9ae5a-5bcb-46d8-8d0a-4f4e0d652dbd.pdf` |
| rhwp OFF | 동일 | `/Users/melee/Downloads/Privacy Filter/Filtered/Session_2026-06-22_1434/filtered_test (1).pdf` |

따라서 이 문제는 Windows 환경에 한정된 파일시스템 문제가 아니라 Chrome downloads API 레벨의 상호작용으로 본다.

재현용 상대 확장은 `output/poc/issue1471-download-repro-extension/`에 작성했지만, `output/`은 `.gitignore` 대상이므로
PR에는 포함하지 않는다.

## 3. 현재 코드와 이력 확인

- rhwp Chrome 확장 0.2.5 소스에는 #198 가드가 포함되어 있다.
- #198 이후 `shouldInterceptDownload(item)`가 false이면 rhwp는 명시적으로 `suggest({ filename })`를 호출하지 않는다.
- `shouldInterceptDownload()` 기준으로 PDF/blob 다운로드는 HWP/HWPX가 아니므로 false가 맞다.
- 그럼에도 `chrome.downloads.onDeterminingFilename` 리스너가 등록되어 있는 것 자체가 다른 확장의 `download({ filename })`
  파일명 결정에 영향을 준다.

즉 본 이슈는 HWP 판정 함수 오탐이 아니라, Chrome `onDeterminingFilename` 리스너 등록 자체의 전역 부작용이다.

## 4. 근본 수정 방향

Chrome 확장에서 `downloads.onDeterminingFilename` 리스너를 상시 등록하지 않는다.

현재 구조는 filename 결정 단계에 rhwp가 참여한다. 이를 다운로드 상태 관찰 방식으로 바꾼다.

- 제거 대상: `chrome.downloads.onDeterminingFilename.addListener(...)`
- 대체 방향: `chrome.downloads.onCreated` / `chrome.downloads.onChanged` 기반 관찰자
- HWP/HWPX 판정: 기존 `rhwp-shared/sw/download-interceptor-common.js`의 `shouldInterceptDownload()` 재사용
- 뷰어 열기: HWP/HWPX로 판정된 다운로드에 한해 `openViewer()` 호출
- 로컬 `file://` HWP 중복 다운로드 억제(#1131): 관찰자 방식에서 가능한 범위로 유지하되, 다른 확장의 filename 결정을 깨지 않는 것을 우선한다.

## 5. 작업 범위

1. `rhwp-chrome/sw/download-interceptor.js`에서 `onDeterminingFilename` 의존을 제거한다.
2. Chrome 다운로드 이벤트 관찰자로 HWP/HWPX 다운로드를 감지하도록 재구성한다.
3. `autoOpen` 설정, 대용량 경고, `openViewer()` 호출 흐름을 유지한다.
4. `file://` HWP 다운로드 억제 동작을 관찰자 구조에 맞게 조정한다.
5. 단위 테스트 또는 모의 Chrome downloads API 테스트를 추가해 다음을 보장한다.
   - 비-HWP/blob/PDF 다운로드에서는 filename 결정 이벤트에 개입하지 않음
   - HWP/HWPX 다운로드는 뷰어 열기 대상이 됨
   - `autoOpen=false`에서는 뷰어를 열지 않음
   - DEXT5 블랙리스트는 유지됨
6. 최소 재현 확장으로 rhwp ON/OFF 경로 차이가 해소되는지 수동 검증한다.

## 7. 위험 요소

| 위험 | 대응 |
|------|------|
| `onCreated` 시점에 filename/mime이 아직 부족할 수 있음 | `onChanged`와 `downloads.search({ id })`로 보강하고, 기존 common 판정 함수를 단계별로 재사용 |
| 다운로드 완료 후 뷰어가 늦게 열릴 수 있음 | 기존 자동 열기 기능은 유지하되 filename 결정 무개입을 우선순위로 둔다 |
| `file://` 중복 다운로드 억제 타이밍이 달라질 수 있음 | best-effort로 유지하고, 실패해도 뷰어 동작과 타 확장 다운로드 안정성을 우선한다 |
| 다른 확장이 시작한 HWP 다운로드까지 rhwp가 열 수 있음 | `byExtensionId`/URL/filename/mime 필드를 확인해 과도한 개입이 없는지 구현계획서에서 확정한다 |

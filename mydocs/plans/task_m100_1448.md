## 1. 배경

사용자 리뷰에서 작성 중 컴퓨터가 꺼진 경우 저장하지 않은 문서를 복구할 수 있는지, 백업과 자동 저장 기능이
있는지 문의가 들어왔다.

현재 rhwp-studio에는 dirty 상태 추적과 `beforeunload` 기본 경고, 문서 교체 전 저장 확인 모달은 있지만,
전원 차단·브라우저 강제 종료·탭 종료 뒤 미저장 문서를 되살리는 자동 백업/복구 플로우는 없다.

Task #1341에서는 파일 핸들을 IndexedDB/localStorage/Service Worker cache에 저장하지 않고 자동 저장/자동
덮어쓰기를 추가하지 않는 것으로 정리했다. 이번 이슈도 원본 파일 자동 덮어쓰기가 아니라, 사용자가 명시적으로
복구 여부를 선택하는 로컬 복구본 저장 기능으로 접근한다.

## 2. 현재 구현 판정

| 항목 | 현재 상태 | 판단 |
|---|---|---|
| dirty 상태 추적 | `DocumentDirtyState`가 dirty/clean 전환과 `beforeunload` 경고 담당 | 있음 |
| 문서 교체 전 확인 | `showUnsavedChangesDialog`로 저장/저장 안 함/취소 제공 | 있음 |
| 수동 저장 | File System Access API 또는 다운로드 fallback | 있음 |
| 문서 이력 IndexedDB | 비교/이력 스냅샷용 `rhwpStudioDocHistory` | 복구용 아님 |
| 자동 백업 | 없음 | 구현 필요 |
| 앱 시작 시 복구 안내 | 없음 | 구현 필요 |
| 비정상 종료 후 복구 | 없음 | 구현 필요 |

## 3. 작업 범위

### Stage 1: 복구 저장소와 자동 백업 매니저

- 문서 비교 이력과 분리된 자동 백업용 IndexedDB 저장소를 추가한다.
- dirty 문서가 변경된 뒤 debounce/interval 기반으로 복구 스냅샷을 저장한다.
- 스냅샷에는 파일명 후보, 원본 포맷, 저장 시각, 바이트 길이, 복구 표시용 메타데이터를 포함한다.
- 자동 백업은 원본 파일 핸들을 저장하지 않고, 원본 파일을 자동 덮어쓰지 않는다.
- 정상 저장 또는 사용자가 변경을 버린 경우 현재 문서의 복구 스냅샷을 정리한다.

### Stage 2: 앱 시작/문서 초기화 복구 안내

- 앱 시작 시 복구 가능한 스냅샷을 조회한다.
- 복구 후보가 있으면 사용자에게 복구/삭제/나중에 결정 선택지를 제공한다.
- 복구를 선택하면 스냅샷 바이트로 문서를 로드하고 dirty 상태를 복구본 기준으로 설정한다.
- 복구본 파일명은 원본을 직접 덮어쓰지 않도록 `복구본` 표시를 붙인다.

### Stage 3: HWP/HWPX/새 문서 정책 정리

- 기존 HWP 문서는 복구본을 HWP 스냅샷으로 저장한다.
- 새 문서는 `새 문서 복구본.hwp` 같은 안전한 이름을 사용한다.
- HWPX 출처 문서는 원본 HWPX를 자동 덮어쓰지 않고, 복구 스냅샷 정책을 명시한다.
- HWPX 직접 저장 비활성 안내와 충돌하지 않게 한다.

### Stage 4: 검증과 UX 보강

- dirty 문서 변경 후 자동 백업 생성, 저장 후 정리, 문서 교체 후 정리/보존 조건을 테스트한다.
- 재시작 상황은 IndexedDB에 복구 스냅샷을 남겨둔 뒤 앱 초기화 루틴으로 검증한다.
- 복구 모달 문구는 사용자가 원본 자동 덮어쓰기로 오해하지 않도록 작성한다.

## 4. 제외 범위

- OS 파일 핸들 저장 또는 자동 권한 재사용
- 원본 파일 자동 덮어쓰기
- Service Worker cache 기반 백업
- 여러 기기 간 동기화
- 한컴 자동 저장 파일 형식 호환

## 5. 회귀 방지 기준

- 기존 수동 저장과 다른 이름으로 저장 동작을 깨지 않는다.
- 새 문서/파일 열기/드롭 시 저장 확인 모달 동작을 유지한다.
- HWPX 직접 저장 비활성 안내를 우회하지 않는다.
- 문서 비교 이력 IndexedDB와 자동 백업 IndexedDB를 섞지 않는다.
- 대용량 문서에서 타이핑마다 즉시 export하지 않도록 debounce/interval을 둔다.

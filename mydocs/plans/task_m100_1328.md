## 1. 문제 요약

Chrome 웹 스토어 리뷰에서 더 많은 폰트 지원과 사용자의 보유 폰트 반영 요청이 확인되었다.
현재 rhwp-studio는 로컬 글꼴 감지를 지원하지만, 사용자가 `환경 설정 > 로컬 글꼴 감지하기`를
직접 실행해야만 동작한다.

현재 흐름에는 다음 한계가 있다.

1. 문서의 원본 글꼴명이 표시 단계에서 대체 글꼴명으로 너무 이르게 치환될 수 있다.
2. `font-loader.ts`의 번들 웹폰트 등록과 alias 등록이 로컬 설치 글꼴 우선 사용과 충돌할 수 있다.
3. 로컬 글꼴 감지 결과가 메모리 캐시에만 있어 새로고침 또는 재실행 후 재사용되지 않는다.
4. 감지 직후 툴바, 글자 모양 대화상자, 대표 글꼴 등록 UI, 렌더링 결과가 전역 갱신되지 않는다.
5. 사용자는 현재 문서의 글꼴이 원본 글꼴, 웹 대체 글꼴, 누락 상태 중 어디에 해당하는지 알기 어렵다.

## 2. 목표

1. 문서 모델과 편집 UI에서 원본 글꼴명을 보존한다.
2. 표시 단계에서만 로컬 글꼴, 웹 대체 글꼴, generic fallback 순서의 resolved chain을 적용한다.
3. 문서 로드 후 현재 문서에 필요한 글꼴 상태를 계산하고 사용자에게 안내한다.
4. 사용자 승인 전에는 `queryLocalFonts()`로 로컬 글꼴 목록을 읽지 않는다.
5. 승인 후 감지 결과를 저장하고 다음 실행부터 재사용한다.
6. 감지 직후 툴바, 글자 모양 대화상자, 대표 글꼴 등록 UI, 렌더링 결과를 즉시 갱신한다.
7. Local Font Access API 미지원 브라우저에서도 기존 웹폰트와 fallback 렌더링을 유지한다.
8. Chrome 확장 컨텍스트에서는 `localStorage`에 의존하지 않고 `chrome.storage.local`을 사용한다.

## 3. UX 및 보안 원칙

### 3.1 동의 기반 로컬 글꼴 감지

로컬 폰트 목록은 기기 fingerprinting에 가까운 민감한 환경 정보로 볼 수 있다. 따라서 문서 로드 시
전체 로컬 글꼴 목록을 조용히 자동 조회하지 않는다.

권장 흐름은 다음이다.

1. 문서의 원본 글꼴명과 rhwp 기본 지원/웹 대체 가능 여부를 먼저 계산한다.
2. 기본 지원 밖 글꼴이 있으면 HWPX 자동보정 모달과 유사한 안내 모달을 띄운다.
3. 사용자가 `로컬 글꼴 감지`를 선택한 경우에만 `queryLocalFonts()`를 호출한다.
4. 감지 결과를 저장하고 현재 문서 글꼴 상태를 다시 계산한다.
5. 사용자가 거절하거나 API가 미지원이면 웹 대체 글꼴/fallback으로 계속 표시한다.

### 3.2 안내 문구 방향

모달의 핵심 메시지는 다음 관점으로 작성한다.

> 현재 문서에 rhwp 기본 지원 글꼴이 아닌 글꼴이 있습니다. 원본에 가깝게 표시하려면 로컬 글꼴 감지를 허용해 주세요.

모달은 감지 목적, 저장 위치, 서버 전송 없음, 거절 시 대체 글꼴로 계속 표시됨을 명확히 안내한다.

### 3.3 상태 분류

문서에 사용된 글꼴은 최소 다음 상태로 분류할 수 있어야 한다.

| 상태 | 의미 |
|---|---|
| 사용 가능 | 번들 웹폰트 또는 저장된 로컬 감지 결과로 바로 사용 가능 |
| 로컬 확인 필요 | 문서 원본 글꼴이 기본 지원 밖이며 아직 로컬 감지를 승인하지 않음 |
| 웹 대체 사용 | 원본 글꼴은 확인되지 않았지만 rhwp의 오픈소스 대체 글꼴로 표시 가능 |
| 누락 | 로컬/웹 대체 모두 명확하지 않아 generic fallback까지 내려갈 수 있음 |

   - Canvas 렌더링 재계산

   - 원본 글꼴명은 문서/편집 값으로 유지

## 7. 수행 단계 후보

구현계획서에서는 다음 후보를 4~6개 단계로 구체화한다.

### Stage 1: 저장소와 로컬 글꼴 상태 기반 정리

- `local-fonts.ts`를 저장소 기반으로 확장한다.
- Web/PWA와 Chrome 확장 저장소 adapter를 분리한다.
- 저장된 감지 결과 로드, 저장, 초기화, 재감지를 지원한다.
- API 미지원/권한 거절/저장 실패를 명시 상태로 다룬다.

### Stage 2: 문서 글꼴 상태 분석과 안내 모달

- `docInfo.fontsUsed` 기준으로 현재 문서의 글꼴 상태를 계산한다.
- 기본 지원 밖 글꼴이 있고 저장된 로컬 감지 결과가 없거나 오래된 경우 안내 모달을 표시한다.
- 모달은 감지 승인, 대체로 보기, 상세 보기를 제공한다.
- 사용자 승인 전에는 `queryLocalFonts()`를 호출하지 않는다.

### Stage 3: resolved font-family chain 적용

- 원본 글꼴명과 표시용 resolved 글꼴을 분리한다.
- `font-loader.ts`, `font-substitution.ts`, `wasm-bridge.ts`의 책임을 정리한다.
- 로컬 설치 후보가 있으면 웹 대체 글꼴보다 앞서 사용한다.
- 원본 글꼴명이 UI 편집 값에서 대체명으로 덮이지 않도록 한다.

### Stage 4: 감지 완료 후 UI와 렌더링 갱신

- 감지 완료 이벤트를 발행하고 관련 UI가 즉시 갱신되도록 연결한다.
- 툴바, 글자 모양 대화상자, 대표 글꼴 등록 UI가 로컬 글꼴 목록을 재구성한다.
- Canvas 문서를 다시 렌더링하고 상태 표시줄 또는 toast로 결과를 안내한다.

### Stage 5: 환경설정 재감지/초기화와 회귀 가드

- 기존 환경설정 버튼은 재감지 버튼으로 유지한다.
- 감지 결과 초기화 또는 다시 감지 진입점을 제공한다.
- 저장 결과 재사용, 권한 거절, API 미지원, 확장 저장소 경로를 검증한다.

## 8. 검증 계획

필수 검증 후보:

```bash
cd rhwp-studio && npm run build
cd rhwp-studio && npm run typecheck
```

`npm run typecheck`가 없는 경우 구현계획서에서 실제 package script를 확인하고 대체 명령을 확정한다.

가능하면 추가할 검증:

```bash
cd rhwp-studio && node e2e/<font-detection-flow>.test.mjs --mode=headless
```

수동 검증:

- Local Font Access API 지원 브라우저에서 문서 로드 후 안내 모달 표시 확인
- 승인 전 `queryLocalFonts()` 미호출 확인
- 승인 후 로컬 글꼴 목록 저장 및 새로고침 후 재사용 확인
- 권한 거절 시 웹 대체 글꼴 렌더링 유지 확인
- API 미지원 브라우저에서 기존 fallback 렌더링 유지 확인
- 감지 후 툴바, 글자 모양 대화상자, 대표 글꼴 등록 UI 갱신 확인
- Chrome 확장 컨텍스트에서 `chrome.storage.local` 저장 확인

## 9. 리스크 및 대응

| 리스크 | 대응 |
|---|---|
| 사용자 동의 전 로컬 폰트 목록을 읽어 개인정보/보안 우려가 생김 | 문서 글꼴 상태 분석과 로컬 목록 조회를 분리하고, 모달 승인 후에만 `queryLocalFonts()`를 호출한다. |
| 원본 글꼴명이 대체 글꼴명으로 저장 또는 편집값에 반영됨 | 원본 font family와 resolved display chain 타입/헬퍼를 분리한다. |
| 웹폰트 alias가 로컬 글꼴 우선순위를 덮음 | `font-loader.ts`는 실제 번들 웹폰트 등록에 집중시키고, local 우선 chain은 표시 단계에서 구성한다. |
| 감지 결과 저장 실패로 매번 모달이 반복됨 | 저장소 예외를 상태로 기록하고 현재 세션 캐시와 "이번에는 대체로 보기" 선택을 분리한다. |
| 확장 Service Worker에서 `localStorage` 의존이 발생함 | 확장 컨텍스트 저장소 adapter에서 `chrome.storage.local`만 사용한다. |
| UI 갱신 누락으로 감지 후에도 목록이 오래된 상태로 보임 | 감지 완료 이벤트와 각 UI의 목록 재구성 메서드를 명시적으로 연결한다. |
| Local Font Access API 미지원 환경에서 회귀 | API 미지원은 정상 상태로 처리하고 기존 웹폰트/fallback 경로를 그대로 유지한다. |

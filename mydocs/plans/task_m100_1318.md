# Task #1318 수행 계획서

## 이슈

- GitHub Issue: #1318
- 제목: 한컴식 Shift+Tab 커서 기준 내어쓰기 설정 구현
- 브랜치: `local/task1318`

## 배경

한컴에디터에서는 여러 줄 문단의 첫 줄에서 특정 문자 위치에 커서를 둔 뒤 `Shift+Tab`을 누르면,
그 커서 x 위치를 기준으로 문단 내어쓰기가 설정된다. 설정 직후 같은 문단의 두 번째 줄 이후가
해당 기준점에 맞춰 다시 조판된다.

rhwp-studio는 현재 `Tab` 입력을 탭 문자 삽입으로 처리한다. `Shift+Tab`도 같은 경로로 들어갈
가능성이 있으므로, 편집 UX 관점에서 한컴식 내어쓰기 단축키와 일반 탭 삽입 동작을 분리해야 한다.

## 목표

1. 여러 줄 문단 첫 줄의 커서 위치에서 `Shift+Tab`을 누르면 내어쓰기 값이 설정된다.
2. 내어쓰기 값은 문단의 현재 왼쪽 여백과 커서의 문단-local x 위치를 기준으로 계산한다.
3. 문단 조판이 즉시 갱신되어 두 번째 줄 이후의 시작 위치가 커서 기준점에 맞춰진다.
4. 문단 모양 대화상자와 ruler가 변경된 내어쓰기 값을 일관되게 표시한다.
5. 표 셀 내부 문단, 머리말/꼬리말, 각주/미주 등 특수 문맥에서 동작 가능 여부를 명시적으로 판정한다.

## 비범위

- 한컴의 모든 탭/내어쓰기 자동 보정 규칙을 완전 복제하지 않는다.
- `Shift+Tab`으로 outline/list level을 변경하는 워드프로세서 계열 UX는 이번 범위가 아니다.
- 문단 단위계 UI의 mm/cm/ch 등 선택 상태 보존은 이번 범위가 아니다.

## 핵심 쟁점

1. 현재 커서 x 좌표를 어떤 좌표계에서 얻을 것인가
   - page 좌표, body 좌표, column 좌표, paragraph-local 좌표 중 변환 기준을 확정해야 한다.

2. 한컴식 내어쓰기 산식
   - 첫 줄 위치는 기존 `marginLeft`를 유지하고, 두 번째 줄 이후 위치가 커서 x에 맞춰져야 한다.
   - 따라서 후보 산식은 `indent = -(cursor_x - paragraph_first_line_base_x)` 계열이다.
   - 정확한 기준점은 Stage 1에서 한컴 기준 샘플과 현행 cursor rect를 비교해 확정한다.

3. raw 스케일 변환
   - UI/WASM 경계에서는 px 기준 `ParaProperties`와 raw 2x HWPUNIT 값이 섞인다.
   - `ParaShapeDialog.collectMods()`의 `ptToRaw2x`, `pxToRaw2x` 변환과 같은 계약을 재사용할지 검토한다.

4. undo/redo
   - 직접 WASM 호출이 아니라 command/history 경로로 적용해야 하는지 확인한다.
   - 최소한 기존 `applyParaPropsToRange()` 경로와 `afterEdit()` 갱신 계약을 유지해야 한다.

5. 표 셀/특수 문맥
   - 셀 내부 문단은 `applyParaFormatInCell` 경로를 사용한다.
   - 머리말/꼬리말, 각주/미주는 별도 API가 있으므로 Stage 1에서 지원 범위를 결정한다.

## 하이퍼-워터폴 단계

### Stage 1. 진단 및 기준선 확정

- `Shift+Tab` 키 이벤트가 현재 어떤 경로로 처리되는지 확인한다.
- 커서 위치에서 얻을 수 있는 rect/line/paragraph 정보를 조사한다.
- 일반 본문 문단과 표 셀 문단의 좌표 기준을 비교한다.
- 한컴 동작 기준을 다음 관점으로 정리한다.
  - 첫 줄 커서 위치
  - 기존 marginLeft
  - 새 indent 값
  - 두 번째 줄 이후 x 위치
- 소스 수정은 하지 않는다.
- 산출물: `mydocs/working/task_m100_1318_stage1.md`

### Stage 2. 구현 계획서 작성

- Stage 1에서 확정한 좌표/스케일 계약에 따라 3~6단계 구현 계획을 작성한다.
- 키 처리, 코어 API, 테스트, WASM 검증 범위를 분리한다.
- 산출물: `mydocs/plans/task_m100_1318_impl.md`

### Stage 3+. 구현 및 검증

- 작업지시자 승인 후 소스 수정에 착수한다.
- 각 단계 완료 후 단계별 보고서를 작성하고 승인 후 다음 단계로 진행한다.

# Task #1310: 수식-only 흐름 자동 줄바꿈 조판

## 배경

Task #1308 CI 회귀를 수정하는 과정에서, `samples/3-09월_교육_통합_2022.hwp` 10쪽의
수식-only 미주 흐름이 한컴 편집기와 다르게 배치되는 별도 조판 이슈를 확인했다.

한컴 편집기는 수식 TAC가 연속되는 줄에서도 열 폭을 기준으로 자동 줄바꿈을 수행한다.
현재 rhwp는 강제 줄바꿈, TAC/텍스트/고정탭 순서, 커서 이동은 #1308에서 보정했지만,
수식-only 흐름 자체의 자동 wrap은 아직 구현되어 있지 않다.

## 재현 자료

- GitHub issue: #1310
- 샘플: `samples/3-09월_교육_통합_2022.hwp`
- 확인 페이지: 10쪽 (`export-svg -p 9`)
- 현행 확인 SVG: `output/poc/ci27052730125-fixed/3-09월_교육_통합_2022_010.svg`

## 목표

- 수식 TAC로만 구성되거나 수식 TAC 중심인 visual line이 열 폭을 넘을 때, 한컴처럼 수식 단위로 다음 줄로 넘긴다.
- `[수식]` 표식, 일반 텍스트, 고정탭, 강제 줄바꿈 순서 보존을 유지한다.
- 수식 앞/뒤 logical cursor stop을 잃지 않는다.
- #1308에서 보정한 강제 줄바꿈 뒤 내어쓰기와 미주 TAC-only 저장 x 보정을 회귀시키지 않는다.

## 구현 조사 포인트

1. `src/renderer/composer.rs`
   - TAC와 텍스트의 logical order를 보존하는 현재 `ComposedLine` 생성 경로 확인
   - 수식-only 줄을 composer 단계에서 분리할지, layout 단계에서 가상 분리할지 결정

2. `src/renderer/layout/paragraph_layout.rs`
   - `equation_only_tac_line_assignment()`, `tac_offsets_for_line()`, `line_tac_offsets_for_width` 경로 확인
   - empty-run TAC-only 렌더 경로에서 `available_width` 초과 판정 가능 여부 확인
   - 미주 가상 문단의 저장 LINE_SEG x/y 흐름과 새 wrap line y 증가 규칙의 충돌 여부 확인

3. cursor/selection
   - 수식 앞/뒤 stop이 wrap 후 line index를 올바르게 반영하는지 확인
   - 문단 경계 이동과 강제 줄바꿈 이동 테스트 유지

## 권장 접근

- 1차는 layout 단계의 TAC-only 줄 분할로 제한한다.
- 줄 분할 단위는 수식 TAC 1개와 그 직후 `[수식]` 표식/공백 텍스트를 하나의 묶음으로 취급한다.
- 저장 LINE_SEG가 있는 미주 문단은 기존 x/y를 우선하되, 실제 수식 bbox가 열 폭을 넘는 경우에만 후속 줄을 합성한다.
- 일반 텍스트가 섞인 줄은 기존 composer/wrap 경로를 유지하고 이번 범위에서 확장하지 않는다.

## 문제 인식

#1274, #1284에서 교육 통합 문제집의 미주 흐름을 여러 단계로 보정했지만, 현재 방식은
한컴오피스의 `미주 모양` 설정을 공식 의미대로 모델링하지 못하고 증상별 y/gap 보정을
누적하는 구조다. 이 방식은 새 샘플마다 overlap과 overflow를 만들 수 있다.

특히 `구분선 위`, `구분선 아래`, `미주 사이`는 서로 다른 경계에 적용되는 값인데,
현재 rhwp 내부에서는 포맷 슬롯 이름, UI JSON 이름, 렌더 계산식의 의미가 일치하지 않는다.

## 공식 기준

한컴오피스 도움말 기준:

- `구분선 위`: 미주와 본문이 만나는 경우, 본문과 미주 구분선 사이의 간격
- `구분선 아래`: 미주 구분선과 미주 내용 사이의 간격
- `미주 사이`: 앞 번호 미주 내용과 다음 번호 미주 내용 사이의 간격
- `미주 위치`: 문서의 끝 또는 구역의 끝
- `미주 모양`: 문서 전체 또는 구역 단위 적용

참조:

- https://help.hancom.com/hoffice/multi/ko_kr/hwp/insert/annotations/endnote_format.htm
- https://help.hancom.com/hoffice130_assistant/ko-KR/Hwp/insert/annotations/endnotes.htm

## 조사 범위

1. HWP5 `HWPTAG_FOOTNOTE_SHAPE` 실제 바이너리 슬롯과 한컴 UI 값의 대응 확인
2. HWPX `<hp:noteSpacing betweenNotes belowLine aboveLine>`의 의미와 내부 IR 대응 확인
3. `FootnoteShape` 모델 필드명/주석과 실제 사용 의미 정리
4. 타입셋, 렌더, height cursor, visual sweep이 쓰는 미주 간격 계산식 정리
5. `구분선위20`, `구분선아래20`, `미주사이20`, `구분선위20+구분선아래20` 샘플 기준 수립

## 목표

1. 공식 UI 의미를 드러내는 정규화 접근자 또는 구조를 만든다.
2. HWP5/HWPX 파서가 같은 의미의 정규화 값을 제공하도록 고정한다.
3. 타입셋과 렌더가 동일한 미주 모양 계산식을 사용하게 한다.
4. 증상별 수치 보정 대신 공식 설정값 기반으로 미주 흐름을 계산한다.
5. 기존 #1284/PR #1292 보정 커밋은 새 모델 기준으로 유지/폐기/분리 여부를 재판단한다.

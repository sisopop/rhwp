## 1. 문제 요약

이슈 #258은 한컴의 누름틀(필드 입력)과 양식 모드를 rhwp에서 지원해 달라는 요청이다.
요청의 핵심은 서식 문서에서 사용자가 정해진 입력 위치만 작성하고, 그 밖의 본문이나
누름틀 자체는 삭제하거나 변경하지 못하게 하는 것이다.

한컴 도움말 기준으로 누름틀은 `[입력-개체-필드 입력]`의 한 종류이며, 안내문/메모/필드
이름과 함께 `양식 모드에서 편집 가능` 속성을 가진다. 양식 모드는 문서 내용을 보호하고
편집 가능 속성을 가진 개체만 수정할 수 있게 하는 모드로 설명된다.

## 2. 현재 rhwp 기반

이미 구현된 기반:

- `FieldType::ClickHere`와 `Field` 모델
- ClickHere command의 `Direction`, `HelpState`, `Name` 파싱/재구축
- `properties bit 0` 기반의 `is_editable_in_form()`
- HWPX `<hp:fieldBegin editable="...">` 파싱/직렬화
- HWP5 ClickHere CTRL_HEADER, memo_index, CTRL_DATA name 직렬화 규칙
- WASM `getFieldList`, `setFieldValue`, `getClickHereProps`, `updateClickHereProps`
- rhwp-studio의 누름틀 고치기 대화상자와 `양식 모드에서 편집 가능` 체크박스
- 누름틀 클릭/활성화/필드 경계 보호 일부

부족한 부분:

- 에디터 전체의 `양식 모드` 상태가 없다.
- 양식 모드에서 일반 본문 입력, 삭제, 붙여넣기, 개체 삭제, 서식 변경을 막는 정책 계층이 없다.
- `insert:field` 명령은 아직 스텁이라 한컴식 `[필드 입력]` 대화상자가 없다.
- 사용자 정보, 문서 요약, 작성한 날짜, 파일 이름/경로 필드의 동적 값 갱신 정책이 정리되지 않았다.

## 3. 목표

1. rhwp-studio에 양식 모드 상태를 추가한다.
2. 양식 모드에서는 `editable=true`인 ClickHere 누름틀 안에서만 텍스트 입력/삭제를 허용한다.
3. 양식 모드에서는 누름틀 삭제, 본문 삭제, 문단 분할/병합, 붙여넣기, 일반 삽입/서식 변경 등
   문서 구조를 바꾸는 동작을 차단한다.
4. 기존 ClickHere 저장/라운드트립 규칙은 유지한다.
5. 한컴 도움말의 `양식 모드에서 편집 가능` 의미가 코드와 UI에서 같은 속성을 가리키게 한다.
6. 후속 스테이지에서 `[필드 입력]` 대화상자와 기타 필드 타입을 확장할 수 있도록 경계를 잡는다.

## 4. 비목표

- 이번 1차 구현으로 한컴 데스크톱의 일반 편집 모드에서 누름틀 삭제를 파일만으로 차단한다고
  보장하지 않는다.
- 배포용 문서 보호/암호/배포 문서 기능은 이번 범위에 넣지 않는다.
- 사용자 정보/문서 요약/날짜/파일 경로 필드의 전체 동적 갱신 엔진은 양식 모드 MVP 이후로 분리한다.
- 모든 양식 개체(Edit, CheckBox, RadioButton, ComboBox, PushButton)의 완전한 상호작용은
  기존 FormObject 작업과 이어서 후속으로 처리한다.

## 5. 진행 단계

### Stage 1 — 착수 조사와 정책 경계 확정

- 한컴 도움말 기준을 작업 문서에 요약한다.
- 기존 ClickHere 모델/직렬화/스튜디오 UI 기반을 확인한다.
- 양식 모드 MVP에서 허용/차단할 입력과 명령 목록을 확정한다.

산출물:

- `mydocs/working/task_m100_258_stage1.md`

### Stage 2 — 양식 모드 MVP

- rhwp-studio에 `normal`/`form` 편집 모드를 추가한다.
- 현재 커서의 ClickHere 필드가 양식 모드에서 편집 가능한지 확인하는 API/헬퍼를 추가한다.
- 키보드 입력, IME 입력, Backspace/Delete, Enter/Tab, 붙여넣기, 잘라내기, 누름틀 삭제를
  양식 모드 정책으로 가드한다.
- 양식 모드 토글 명령과 상태 갱신을 추가한다.

### Stage 3 — 누름틀 이동/선택 UX 정리

- 양식 모드에서 Tab 또는 F11 기반으로 다음 편집 가능 누름틀로 이동하는 흐름을 정리한다.
- 상태 표시줄에 양식 모드와 현재 누름틀 정보를 일관되게 표시한다.
- 표/글상자 내부 ClickHere가 기존 경로 기반 탐색과 충돌하지 않는지 확인한다.

### Stage 4 — 필드 입력 대화상자 1차

- `insert:field` 스텁을 실제 대화상자로 교체한다.
- 우선 누름틀 탭을 구현하고 안내문/메모/필드 이름/양식 모드 편집 가능 값을 저장한다.
- 기타 필드 탭은 UI 골격 또는 후속 계획으로 분리한다.

### Stage 5 — 검증과 보고

- ClickHere 샘플과 form 샘플에서 양식 모드 입력 제한을 검증한다.
- HWP/HWPX 저장 후 `editable` 속성과 ClickHere command/CTRL_DATA 보존을 확인한다.
- 필요한 단위 테스트와 스튜디오 빌드 검증을 수행한다.
- 최종 보고서를 작성한다.

## 6. 검증 기준

- 양식 모드 OFF: 기존 편집 동작 회귀 없음.
- 양식 모드 ON:
  - editable ClickHere 내부 텍스트 입력/삭제 가능
  - editable이 아닌 ClickHere는 수정 불가
  - 일반 본문/표 밖 본문/개체 삭제/누름틀 삭제 차단
  - 저장 후 HWPX `editable`과 HWP5 `properties bit 0` 보존
- 기존 누름틀 테스트와 필드 값 설정 테스트 통과
- 프론트엔드 타입 체크 또는 빌드 통과

## 7. 리스크 및 대응

| 리스크 | 대응 |
|---|---|
| 양식 모드 정책이 키보드 경로 일부만 막아 우회 가능 | `CommandDispatcher`, keyboard/input handler, 필드 삭제 명령을 함께 점검 |
| IME 입력이 가드 밖에서 들어감 | `onInput`, composition 처리, raw insert 경로를 같이 가드 |
| 표/글상자 내부 ClickHere 경로 누락 | 기존 `getFieldInfoAtByPath`, `setActiveFieldByPath` 경로를 기준으로 검증 |
| 일반 편집 모드 보호와 양식 모드 보호가 혼동됨 | 이슈 응답/보고서에 “외부 프로그램 제어용 양식 모드” 범위를 명시 |
| 필드 입력 전체 구현 범위가 커짐 | Stage 4는 누름틀 탭 우선, 기타 필드는 후속으로 분리 |

## 2. 결함 본체 (spot 조사로 확정)

borderFill ID의 **등록 축과 방출/참조 축 불일치 (off-by-one)**:

| 경로 | 축 | 위치 |
|------|-----|------|
| 방출 | borderFill id = **`idx + 1`** (1-based) | `serializer/hwpx/header.rs:300` (주석: "HWPX 관찰 1-based") |
| 참조 | borderFillIDRef = **원본 값 1-based** 그대로 | parser `parse_u16`(보존), serializer `to_string`(무변환) |
| **등록** | **`idx`** (0-based) ← **결함** | `serializer/hwpx/context.rs:117` |

→ doc_info.border_fills 등록만 0-based라, 마지막 id(예: exam_social 31)가 등록
범위(0~30) 밖 → 미등록 판정 → `assert_all_refs_resolved` SERIALIZE_FAIL.

### 왜 일부만 실패했나

- charShape는 원본 id가 **0-based**(exam_social charPr id 0~)라 `idx`(0-based) 등록과
  일치 → 통과. borderFill만 **1-based**(id 1~)라 어긋남 — 같은 header.xml 안에서
  리소스별 id 기준이 달랐던 것.
- 인라인 컨트롤(표/셀)의 borderFillIDRef는 별도로 IR 값 그대로 등록
  (`context.rs:135~140`)되므로, 그 경로로 커버된 참조는 통과(k-water-rfp 등).
  실패는 **doc_info charShape/paraShape의 borderFillIDRef가 인라인에 없는 id를
  참조**할 때만 표면화 → 4샘플.

## 3. 수정안 (1줄 — spot 가설 검증 완료)

`context.rs:117` 등록을 방출 축과 일치:
```rust
ctx.border_fill_ids.register((idx + 1) as u16);  // idx → idx+1
```
- **검증 완료**(임시 적용): 4샘플 전부 PASS 전환 + k-water-rfp 회귀 없음.
- borderFill **방출 id가 이미 1-based**이므로 등록만 맞추면 정합 — 다른 변경 불요.

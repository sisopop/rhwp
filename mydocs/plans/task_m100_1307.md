# Task m100 #1307 수행 계획

## 제목

확장 프로그램 service worker의 외부 URL fetch 경로 보안 강화

## 배경

2026-06-06 보안 제보에서 Chrome/Firefox 확장 프로그램의 content script가 감지한 HWP/HWPX 링크를 service worker가 대신 fetch하는 흐름이 내부망/localhost 요청 유발 및 미리보기 데이터 노출 경로가 될 수 있음이 보고되었다.

제보 원문에는 PoC 링크와 재현 세부가 포함되어 있어 공개 이슈에는 요약만 등록했다.

## 대상

- `rhwp-chrome/sw/message-router.js`
- `rhwp-chrome/sw/thumbnail-extractor.js`
- `rhwp-chrome/content-script.js`
- `rhwp-firefox/sw/message-router.js`
- `rhwp-firefox/sw/thumbnail-extractor.js`
- `rhwp-firefox/content-script.js`
- 필요 시 공통 URL 검증 모듈 및 확장 보안 테스트 문서

## 구현 방향

1. service worker 메시지 sender를 검증한다.
   - `fetch-file`은 extension viewer 내부 페이지에서만 허용한다.
   - `extract-thumbnail`은 content script에서 올 수 있으나 URL 정책을 엄격히 적용한다.

2. extension-side fetch URL 정책을 명문화한다.
   - 허용 scheme은 `https:`와 제한적 `http:`로 한정한다.
   - loopback, localhost, link-local, private IP, 내부 호스트명을 차단한다.
   - username/password 포함 URL을 차단한다.
   - 문서 확장자는 pathname 기준으로 판단한다.

3. redirect를 수동 처리한다.
   - `redirect: "manual"`로 fetch한다.
   - `Location` 대상 URL을 재검증한다.
   - redirect 반복 횟수 제한을 둔다.

4. credentials를 전송하지 않는다.
   - `credentials: "omit"`을 사용한다.

5. page DOM 노출을 줄인다.
   - private/internal URL의 thumbnail request 자체를 차단한다.
   - 차단 응답은 dataUri를 포함하지 않는다.
   - 향후 Shadow DOM/extension UI 분리 검토는 후속 과제로 남길 수 있다.

6. Chrome/Firefox 동일 정책을 유지한다.
   - 가능하면 동일 이름의 모듈을 양쪽에 추가한다.
   - 기존 build 구조를 크게 바꾸지 않는다.

## 검증

- `npm run build` in `rhwp-chrome`
- `npm run build` in `rhwp-firefox`
- 보안 정책 함수 단위 테스트 또는 로컬 스크립트 검증
- `cargo test` 범위 영향 없음 확인은 필요 시 제한적으로 수행

## 완료 기준

- content script가 삽입한 localhost/private URL 링크의 자동 썸네일 fetch가 차단된다.
- redirect로 내부 주소로 이동하는 URL이 차단된다.
- viewer 내부가 아닌 발신자의 `fetch-file` 요청이 거부된다.
- 정상적인 공개 HWP/HWPX URL 경로는 기존처럼 동작한다.
- Chrome/Firefox 확장 빌드가 통과한다.

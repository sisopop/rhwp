## 1. 배경

#1380 1단계 전수 lineseg 진단(aift)에서 검출. 원본 `hp:fieldBegin type="MEMO"` 2건의
`hp:parameters` + `hp:subList`(메모 본문 문단)가 roundtrip 후 소실된다.

PR #1405(physwkim, 직렬화 충실도 묶음 — 어제 머지) 머지 후 재실측: fieldBegin **태그
자체는 2건 보존**되나 parameters·subList는 **여전히 소실** (RT: parameters=0,
subList=0, 메모 본문 "기업 소개 및 본 과제…" 0건). #1405와 무겹침 확인.

## 2. 현황 분석 (2026-06-14 — 코드·실물 확인 완료)

### serializer 결손 본체

- `write_field_begin`(field.rs:36)이 **`empty_tag`로 자기닫힘 방출** → 자식
  (parameters/subList) 들어갈 자리 자체가 없음. MEMO 타입은 자식 보유 → start/end
  태그로 전환 필요.
- 호출부(section.rs:660)는 `<hp:ctrl>` 포장 + `writer_to_string(|w| write_field_begin(w, f))`
  — memo subList 문단 방출에 `ctx`(para id 발급) 필요 → 시그니처 변경.

### 원본 parameters 구조 (aift MEMO 실물)

  <hp:integerParam name="Prop">0</hp:integerParam>

### 범위 판정

- MEMO 외 fieldBegin(BOOKMARK 1·FORMULA 1)도 존재하나, 이슈는 **MEMO 본문 소실**이
  본질. subList는 MEMO만 보유(파서도 `field_type == Memo` 가드). parameters는 다른
  필드도 가질 수 있으나, 1단계에서 전수 분포 확인 후 범위 확정.

## 3. 설계 결정점 (1단계에서 확정, 권고안 제시)

parameters 보존 방식:
- **(A) 원본 verbatim 보존** — `Field`에 `raw_parameters_xml: Option<String>` 추가,
  파서가 parameters 요소 원문 저장, serializer가 그대로 방출. **장점**: 7필드 외
  필드 변형/누락 위험 0, 구현 단순. **단점**: IR 의미 비교 약함(문자열 비교).
- **(B) 구조화 보존** — 각 파라미터를 IR 필드로. **장점**: 의미 비교. **단점**:
  파라미터 종류·순서 가변(cnt 7 vs 다른 필드), MemoShape 참조 등 부수 처리 다수.

→ **권고: (A)**. #1405가 채택한 "원본 부수 항목 verbatim" 기조와 일치하고, 메모
parameters는 한컴 내부 메타라 의미 비교 실익이 낮다. 1단계 전수 분포로 확정.

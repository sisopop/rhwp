# Task M100 #1425 수행계획서 - PR 리뷰 워크플로 문서의 maintainer/collaborator 규칙 분리

- 이슈: #1425 "PR 리뷰 워크플로 문서의 maintainer/collaborator 규칙 분리"
- 마일스톤: M100 (v1.0.0)
- 브랜치: `task_m100_1425` (`upstream/devel` 분기)
- 작성일: 2026-06-21

## 1. 배경

PR #1420에서 `mydocs/manual/pr_review_workflow.md`에 collaborator self-merge 후보 운영 규칙이
추가되었다. 이 규칙은 PR 운영 문서를 PR head에 포함하고, 처음부터 `mydocs/pr/archives/` 경로에
작성하며, collaborator가 `upstream` 작업 브랜치에 직접 push하는 예외 흐름을 정리한 것이다.

다만 해당 변경이 기존 maintainer 중심 외부 PR 검토 절차를 대체하는 위치에 들어가면서, 현재 문서는
maintainer 일반 경로와 collaborator self-merge 후보 경로를 같은 기본 규칙처럼 읽히게 한다.

PR #1424 처리 과정에서는 review 문서에 `draft`, `mergeable`, `head SHA`, `CI 상태` 같은 값이
확정 사실처럼 남으면, ready 전환 또는 CI 재실행 이후 문서가 곧바로 낡아 보이는 문제도 확인되었다.

## 2. 현황 (조사 완료)

- 현재 `pr_review_workflow.md`의 "리뷰 문서 작성" 절은 모든 PR 리뷰 문서를 처음부터
  `mydocs/pr/archives/pr_{N}_review*.md`에 작성한다고 설명한다.
- 같은 절에 collaborator 권한, 문서 동반 push, `upstream` 직접 push 규칙이 일반 PR 처리 절차와
  분리되지 않은 채 포함되어 있다.
- PR #1420 패치 기준, 기존 `mydocs/pr/pr_{N}_review.md` 작성 후 archive 이동 흐름이
  `mydocs/pr/archives/` 직접 작성 흐름으로 바뀌었다.
- PR #1424 review 문서에는 `draft true`, `mergeable CLEAN`, 특정 head SHA, GitHub Actions
  1차 확인 상태가 작성 시점 값으로 남아 있다.
- 이슈 #1425의 요구는 collaborator 규칙 삭제가 아니라, maintainer 기본 경로를 복구하면서 collaborator
  self-merge 후보 경로를 별도 섹션으로 보존하는 것이다.

## 3. 수정 방향

1. **maintainer 일반 경로 복구**
   - 외부 기여자 PR 검토의 기본 문서 작성 위치를 `mydocs/pr/pr_{N}_review.md`와
     `mydocs/pr/pr_{N}_review_impl.md`로 되돌린다.
   - 처리 완료 후 `mydocs/pr/archives/`로 이동하는 후속 절차를 복구한다.
   - PR 브랜치 fetch, merge 시뮬레이션, devel sync, 후속 보고 흐름은 maintainer 운영 방식으로
     읽히게 정리한다.

2. **collaborator self-merge 후보 경로 분리**
   - collaborator가 본인 PR을 self-merge 후보로 준비할 때만 예외 경로로 둔다.
   - PR 번호 생성 후 review 문서를 PR head에 포함할 수 있음을 별도 섹션에 명시한다.
   - 처음부터 `mydocs/pr/archives/pr_{N}_review*.md`를 사용할 수 있는 조건을 제한한다.
   - `upstream` 작업 브랜치 직접 push는 collaborator self-merge 후보 경로의 운영 규칙으로만 둔다.

3. **volatile 상태값 기록 규칙 강화**
   - `draft`, `mergeable`, `head SHA`, `CI 상태`는 확정 사실처럼 쓰지 않도록 제한한다.
   - 필요한 경우 "문서 작성 시점 참고값" 또는 "merge 전 최신 상태 확인 조건"으로 표현한다.
   - 최종 merge 기준은 "PR head 최신 커밋, 최신 GitHub Actions 통과, 작업지시자 승인"으로 정리한다.

4. **문서 구조 정리**
   - 기존 maintainer 대상 매뉴얼의 주 독자를 유지한다.
   - collaborator self-merge 후보는 예외 섹션 또는 별도 하위 절로 분리한다.
   - PR #1420과 PR #1424의 경험은 사례 또는 주의 규칙으로 남기되, 기본 절차를 덮어쓰지 않게 한다.

## 4. 목표와 완료 조건

1. `pr_review_workflow.md`에서 maintainer 일반 경로와 collaborator self-merge 후보 경로가 명확히 분리된다.
2. #1420에서 추가된 collaborator 운영 규칙은 보존되지만 maintainer 기본 규칙을 대체하지 않는다.
3. review 문서의 volatile 상태값 기록 방식이 merge 후에도 모순되지 않도록 제한된다.
4. 향후 collaborator self-merge PR은 CI 통과 후 ready/merge되어도 review 문서가 낡거나 모순되지 않는다.
5. 문서 변경만 수행하며 코드, GitHub branch protection, 저장소 권한 설정은 변경하지 않는다.

## 5. 단계 개요

1. **구현 계획서 작성**
   - `pr_review_workflow.md`의 수정 대상 절과 새 섹션 배치를 확정한다.
   - maintainer 기본 경로, collaborator 예외 경로, volatile 상태값 규칙의 문구 수준 체크리스트를 만든다.

2. **문서 개정**
   - maintainer 기본 경로를 복구한다.
   - collaborator self-merge 후보 규칙을 별도 섹션으로 이동하고 조건을 제한한다.
   - volatile 상태값 기록 규칙과 최종 merge 조건을 추가한다.

3. **검증과 보고**
   - 변경 diff가 `mydocs/manual/pr_review_workflow.md`와 작업 문서 범위에 머무는지 확인한다.
   - `git diff --check`로 Markdown 공백 오류를 확인한다.
   - 최종 보고서에 변경 의도, 비목표, 검증 결과를 기록한다.

## 6. 위험과 대응

| 위험 | 대응 |
|------|------|
| #1420 collaborator 규칙을 삭제하거나 약화 | 별도 섹션으로 보존하고 적용 조건만 제한 |
| maintainer 기본 경로와 collaborator 예외 경로가 다시 섞임 | 문서 구조를 "기본 경로"와 "예외 경로"로 분리 |
